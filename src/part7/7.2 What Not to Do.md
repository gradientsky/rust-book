# What Not to Do

Every language has idioms that feel natural to newcomers but work against the
language's strengths. In Rust, the most common mistakes come from fighting the
compiler instead of listening to it. Developers arriving from C++, Java, or
Python bring habits that were reasonable in those languages but produce slow,
fragile, or unidiomatic code in Rust.

This chapter collects the anti-patterns that experienced Rust programmers have
learned to avoid. Each section opens with the _why_ — the instinct that leads
to the mistake — shows the problematic code, and then demonstrates the
idiomatic alternative. The goal is not to shame anyone for writing code that
works. It is to show that Rust offers a better way, and the better way is
usually simpler.

## Cloning to Silence the Borrow Checker

The borrow checker exists to prevent data races and use-after-free bugs at
compile time. When it rejects your code, it is telling you that two parts of
your program want conflicting access to the same data. The most common escape
hatch newcomers reach for is `.clone()` — make a copy, and both sides get their
own. The compiler stops complaining, the code runs, and the problem seems
solved.

It is not solved. It is hidden.

### The Problem with Gratuitous Clones

Cloning has two costs. The obvious cost is performance: cloning a `String`
allocates new heap memory and copies every byte. The subtle cost is
correctness — cloning creates an independent value, so changes to the clone
never affect the original:

```rust
fn main() {
    let mut original = String::from("hello");
    let cloned = original.clone();

    original.push_str(", world");

    println!("original: {original}");
    println!("cloned:   {cloned}");
}
```

```
original: hello, world
cloned:   hello
```

If you intended `cloned` to reflect changes to `original`, the clone broke that
expectation silently. The program compiles and runs, but it does not do what you
meant.

### What to Do Instead

When the borrow checker rejects your code, ask: _can I restructure the code so
the conflicting accesses do not overlap?_

#### Narrow the borrow scope

Often the simplest fix is to ensure the borrow ends before the mutation begins.
The borrow checker uses _non-lexical lifetimes_ — a borrow ends at its last use,
not at the end of the block:

```rust
fn main() {
    let mut scores = vec![10, 20, 30, 40, 50];

    // Read phase: borrow ends after this line
    let highest = *scores.iter().max().unwrap();

    // Mutate phase: no active borrow
    scores.push(highest + 10);

    println!("{scores:?}");
}
```

```
[10, 20, 30, 40, 50, 60]
```

If you had written `let highest = scores.iter().max().unwrap()` without
dereferencing, the borrow would stay alive through `scores.push(...)` and the
compiler would reject the code. Copying the `i32` out with `*` ends the borrow
immediately.

#### Use indices for in-place mutation

When you need to read from a collection and mutate it in the same loop, index
into the collection instead of holding a reference:

```rust
fn main() {
    let mut values = vec![1, 2, 3, 4, 5];

    for i in 0..values.len() {
        values[i] *= values[i];
    }

    println!("{values:?}");
}
```

```
[1, 4, 9, 16, 25]
```

This works because `values[i]` borrows and releases within a single expression.
An even better alternative when you do not need the index is `iter_mut`:

```rust
fn main() {
    let mut values = vec![1, 2, 3, 4, 5];

    for v in &mut values {
        *v *= *v;
    }

    println!("{values:?}");
}
```

```
[1, 4, 9, 16, 25]
```

#### Accept borrowed types in function signatures

A function that takes `&String` forces the caller to own a `String`. A function
that takes `&str` accepts anything string-like — a `String`, a `&str`, a string
literal — without requiring the caller to clone:

```rust
// Accepts &str, &String, string literals — no clone needed
fn greet(name: &str) {
    println!("Hello, {name}!");
}

fn main() {
    let owned = String::from("Alice");
    let borrowed = "Bob";

    greet(&owned);
    greet(borrowed);
}
```

```
Hello, Alice!
Hello, Bob!
```

The same principle applies to `&[T]` instead of `&Vec<T>`. Prefer the borrowed
slice; it accepts more types with less friction.

### When Cloning Is Correct

Cloning is not inherently wrong. It is wrong when it hides a design problem.
It is correct when you genuinely need an independent copy:

- Sending data to another thread that outlives the current scope.
- Storing a snapshot of a value that will be mutated later.
- When the value is small and `Copy` (`i32`, `f64`, `bool`) — these types are
  copied implicitly, no `.clone()` needed.

Here is a case where cloning is the right choice. A status tracker needs to
record each state before it changes — each snapshot must be an independent copy:

```rust
fn main() {
    let mut history = Vec::new();
    let mut status = String::from("pending");

    history.push(status.clone());  // snapshot before mutation
    status = String::from("processing");

    history.push(status.clone());
    status = String::from("complete");

    history.push(status);  // final value — move, no clone needed

    println!("History: {history:?}");
}
```

```
History: ["pending", "processing", "complete"]
```

Each `clone()` captures a snapshot that must survive independently of the
original value. The final push uses a move instead of a clone because `status`
is not needed afterward — a small detail that shows you are thinking about
ownership, not just silencing the compiler.

The rule of thumb: if you added `.clone()` because the compiler said "value
moved" or "cannot borrow," pause and ask whether restructuring the code would
eliminate the need.

## Reaching for Unwrap in Production Code

`unwrap()` converts a `Result` or `Option` into its inner value — or panics if
the value is an `Err` or `None`. It is the fastest way to prototype: skip the
error handling, get to the logic, and come back later to fix it.

The problem is that "later" often means "never."

### Why Unwrap Is Dangerous in Libraries

A library author who calls `unwrap()` takes the decision away from the caller.
If the operation fails, the entire program crashes with a message like
`called 'unwrap()' on an 'Err' value`. The caller has no chance to retry, fall
back, or log a meaningful error:

```rust
fn parse_port(input: &str) -> u16 {
    input.parse().unwrap() // panics on invalid input
}

fn main() {
    let port = parse_port("8080");
    println!("Port: {port}");

    // This would panic at runtime:
    // let bad = parse_port("abc");
}
```

```
Port: 8080
```

The function works when the input is valid. It destroys the program when it is
not. A library function should never make that choice — the caller should.

### Return Result and Let the Caller Decide

The fix is to return `Result` and let the caller choose how to handle failure:

```rust
use std::num::ParseIntError;

fn parse_port(input: &str) -> Result<u16, ParseIntError> {
    input.parse()
}

fn main() {
    match parse_port("8080") {
        Ok(port) => println!("Port: {port}"),
        Err(e) => println!("Invalid port: {e}"),
    }

    match parse_port("abc") {
        Ok(port) => println!("Port: {port}"),
        Err(e) => println!("Invalid port: {e}"),
    }
}
```

```
Port: 8080
Invalid port: invalid digit found in string
```

The caller can now handle the error however they see fit — retry, use a
default, or propagate it upward with `?`.

### The Unwrap Spectrum

Not every use of `unwrap` is wrong. The right choice depends on context:

| Method | Use when | Example |
|---|---|---|
| `?` | The caller should handle the error | `let port = input.parse()?;` |
| `unwrap_or` / `unwrap_or_else` | A sensible default exists | `port.unwrap_or(8080)` |
| `expect("reason")` | Failure indicates a bug, not a user error | `env::var("HOME").expect("HOME must be set")` |
| `unwrap()` | Tests, examples, and quick prototypes | `assert_eq!(parse_port("80").unwrap(), 80);` |

`expect` is strictly better than `unwrap` for documenting invariants. The
message should explain _why_ the value must be present, not restate what
happened:

```rust
fn main() {
    let config_dir = std::env::var("HOME")
        .expect("HOME environment variable must be set");

    println!("Config dir: {config_dir}");
}
```

```
Config dir: /Users/admin
```

If `HOME` is missing, the panic message reads: `HOME environment variable must
be set: NotPresent` — far more useful than `called 'unwrap()' on an 'Err'
value: NotPresent`.

## Writing Java in Rust Syntax

Every language has a dominant paradigm. Java defaults to classes and interfaces.
Python defaults to dictionaries and duck typing. C++ defaults to inheritance
hierarchies. When developers bring these defaults to Rust, they produce code
that compiles but misses the point.

### Getter and Setter Boilerplate

In Java, every field is private and exposed through `get` and `set` methods.
This is an established convention in Java because fields cannot have
side-effects on access. In Rust, this convention is usually unnecessary.

The anti-pattern looks like this:

```rust
struct User {
    name: String,
    email: String,
}

#[allow(dead_code)]
impl User {
    fn get_name(&self) -> &str { &self.name }
    fn set_name(&mut self, name: String) { self.name = name; }
    fn get_email(&self) -> &str { &self.email }
    fn set_email(&mut self, email: String) { self.email = email; }
}

fn main() {
    let mut user = User {
        name: String::from("Alice"),
        email: String::from("alice@example.com"),
    };

    println!("{} <{}>", user.get_name(), user.get_email());

    user.set_name(String::from("Bob"));
    println!("{}", user.get_name());
}
```

```
Alice <alice@example.com>
Bob
```

This code works, but the getters and setters add nothing. They do not enforce
invariants, they do not log access, they do not compute anything. They are pure
ceremony.

**If there are no invariants to protect, make the fields public:**

```rust
struct User {
    pub name: String,
    pub email: String,
}

fn main() {
    let mut user = User {
        name: String::from("Alice"),
        email: String::from("alice@example.com"),
    };

    println!("{} <{}>", user.name, user.email);

    user.name = String::from("Bob");
    println!("{}", user.name);
}
```

```
Alice <alice@example.com>
Bob
```

**If you do need a getter**, the Rust API Guidelines say: drop the `get_`
prefix. A method that returns a reference to a field should be named after the
field:

```rust
struct Account {
    balance: f64,
}

impl Account {
    fn new(balance: f64) -> Self {
        Self { balance }
    }

    // Getter: named after the field, no "get_" prefix
    fn balance(&self) -> f64 {
        self.balance
    }

    // Setter exists because it enforces an invariant
    fn deposit(&mut self, amount: f64) {
        if amount > 0.0 {
            self.balance += amount;
        }
    }
}

fn main() {
    let mut account = Account::new(100.0);
    account.deposit(50.0);
    account.deposit(-10.0); // silently ignored — invariant protected

    println!("Balance: {:.2}", account.balance());
}
```

```
Balance: 150.00
```

The getter exists because `balance` is private — external code should not be
able to set it directly. The setter is named `deposit` because it enforces a
business rule. This is meaningful encapsulation, not boilerplate.

### Overusing dyn Trait

In Java, every abstraction starts with an interface. Developers bring that habit
to Rust and reach for `dyn Trait` by default — heap-allocating every trait
object, introducing vtable overhead, and preventing the compiler from inlining.

```rust
trait Summarize {
    fn summary(&self) -> String;
}

struct Article { title: String }
struct Tweet { content: String }

impl Summarize for Article {
    fn summary(&self) -> String {
        format!("Article: {}", self.title)
    }
}

impl Summarize for Tweet {
    fn summary(&self) -> String {
        format!("Tweet: {}", self.content)
    }
}

// Anti-pattern: dyn when types are known at compile time
#[allow(dead_code)]
fn print_summary_dyn(item: &dyn Summarize) {
    println!("{}", item.summary());
}

// Idiomatic: generics enable inlining and monomorphization
fn print_summary(item: &impl Summarize) {
    println!("{}", item.summary());
}

fn main() {
    let article = Article { title: String::from("Rust 2024") };
    let tweet = Tweet { content: String::from("Loving Rust!") };

    print_summary(&article);
    print_summary(&tweet);
}
```

```
Article: Rust 2024
Tweet: Loving Rust!
```

With `&impl Summarize`, the compiler generates a specialized version of
`print_summary` for each concrete type. The function call is static, inlinable,
and carries no vtable overhead. With `&dyn Summarize`, every call goes through a
pointer indirection.

**When `dyn Trait` is the right choice:**

`dyn Trait` earns its place when you need a _heterogeneous collection_ — a
single container holding different concrete types:

```rust
trait Shape {
    fn area(&self) -> f64;
}

struct Circle { radius: f64 }
struct Rectangle { width: f64, height: f64 }

impl Shape for Circle {
    fn area(&self) -> f64 { std::f64::consts::PI * self.radius * self.radius }
}

impl Shape for Rectangle {
    fn area(&self) -> f64 { self.width * self.height }
}

fn main() {
    // Different types in one collection — dyn is correct here
    let shapes: Vec<Box<dyn Shape>> = vec![
        Box::new(Circle { radius: 3.0 }),
        Box::new(Rectangle { width: 4.0, height: 5.0 }),
    ];

    for shape in &shapes {
        println!("Area: {:.2}", shape.area());
    }
}
```

```
Area: 28.27
Area: 20.00
```

The rule is: prefer `impl Trait` or generics by default. Reach for
`Box<dyn Trait>` only when you need runtime polymorphism — when the concrete
types are not known until the program runs.

### Simulating Inheritance

Rust has no class inheritance. Some developers simulate it by implementing
`Deref` on a struct that wraps another struct, making the "child" transparently
expose the "parent's" methods through deref coercion. This is a misuse of
`Deref`.

`Deref` exists for smart pointer types — `Box<T>`, `Arc<T>`, `Rc<T>` — where
the wrapper _is_ a transparent handle to the inner value. Using it to fake
inheritance creates confusing APIs where methods appear on a type without being
defined in its `impl` block, and traits implemented on the inner type do not
automatically apply to the outer type.

The idiomatic alternative is _composition with explicit delegation_:

```rust
struct Engine {
    horsepower: u32,
}

impl Engine {
    fn start(&self) -> String {
        format!("Engine ({} HP) started", self.horsepower)
    }
}

struct Car {
    engine: Engine,
    model: String,
}

impl Car {
    fn new(model: &str, hp: u32) -> Self {
        Self {
            engine: Engine { horsepower: hp },
            model: String::from(model),
        }
    }

    // Explicit delegation — clear, greppable, no magic
    fn start(&self) -> String {
        format!("{}: {}", self.model, self.engine.start())
    }
}

fn main() {
    let car = Car::new("Model S", 670);
    println!("{}", car.start());
}
```

```
Model S: Engine (670 HP) started
```

If multiple types share behavior, define a trait. If a type wraps another type,
delegate explicitly. Rust's composition model is more verbose than inheritance
in the small, but it avoids the fragile base class problem, makes dependencies
visible, and scales better as the codebase grows.

## Fighting the Type System

Rust's type system is one of its greatest assets. Fighting it — using strings
where enums belong, booleans where named types would be clearer — throws away
compile-time guarantees and makes code harder to read.

### Stringly-Typed Code

Using strings as identifiers, states, or configuration keys means the compiler
cannot check them:

```rust
fn set_log_level(level: &str) {
    match level {
        "debug" | "info" | "warn" | "error" => {
            println!("Log level set to: {level}");
        }
        _ => println!("Unknown log level: {level}"),
    }
}

fn main() {
    set_log_level("info");
    set_log_level("wrn");  // Typo — caught only at runtime
}
```

```
Log level set to: info
Unknown log level: wrn
```

The typo `"wrn"` compiles without error. An enum catches it at compile time:

```rust
#[derive(Debug)]
#[allow(dead_code)]
enum LogLevel {
    Debug,
    Info,
    Warn,
    Error,
}

fn set_log_level(level: LogLevel) {
    println!("Log level set to: {level:?}");
}

fn main() {
    set_log_level(LogLevel::Info);
    set_log_level(LogLevel::Warn);

    // This would not compile:
    // set_log_level(LogLevel::Wrn);
}
```

```
Log level set to: Info
Log level set to: Warn
```

The compiler checks every variant. A typo in the variant name is a compile
error, not a silent runtime bug.

### Boolean Blindness

Functions that take multiple booleans are impossible to read at the call site:

```rust
fn create_report(verbose: bool, include_header: bool, color: bool) {
    println!(
        "Report: verbose={verbose}, header={include_header}, color={color}"
    );
}

fn main() {
    create_report(true, false, true); // What does each bool mean?
}
```

```
Report: verbose=true, header=false, color=true
```

At the call site, `create_report(true, false, true)` is opaque. You have to
look up the function signature to know which boolean controls what. Named types
make every call site self-documenting:

```rust
#[derive(Debug)]
#[allow(dead_code)]
enum Verbosity { Normal, Verbose }

#[derive(Debug)]
#[allow(dead_code)]
enum Header { Omit, Include }

#[derive(Debug)]
#[allow(dead_code)]
enum ColorMode { Plain, Colored }

fn create_report(verbosity: Verbosity, header: Header, color: ColorMode) {
    println!("Report: {verbosity:?}, {header:?}, {color:?}");
}

fn main() {
    create_report(Verbosity::Verbose, Header::Omit, ColorMode::Colored);
}
```

```
Report: Verbose, Omit, Colored
```

No ambiguity. The call site reads like a sentence: create a verbose report, omit
the header, use colored output. The compiler prevents passing `Verbosity` where
`ColorMode` is expected — an impossible mix-up with booleans.

### Making Invalid States Representable

The most insidious form of fighting the type system is using a combination of
fields where some combinations are logically impossible. The previous chapter
introduced the principle of making illegal states unrepresentable. The
anti-pattern looks like this:

```rust
struct Connection {
    use_tls: bool,
    cert_path: Option<String>,  // meaningless when use_tls is false
    connected: bool,
    error_message: Option<String>,  // meaningless when connected is true
}

fn describe(conn: &Connection) {
    if conn.connected {
        if conn.use_tls {
            println!(
                "Connected with TLS (cert: {})",
                conn.cert_path.as_deref().unwrap_or("none")
            );
        } else {
            println!("Connected without TLS");
        }
    } else {
        println!(
            "Disconnected: {}",
            conn.error_message.as_deref().unwrap_or("unknown error")
        );
    }
}

fn main() {
    let conn = Connection {
        use_tls: true,
        cert_path: Some(String::from("/etc/ssl/cert.pem")),
        connected: true,
        error_message: None,
    };

    describe(&conn);

    // This is a nonsensical state: TLS on but no cert, connected but has error
    let bad = Connection {
        use_tls: true,
        cert_path: None,
        connected: true,
        error_message: Some(String::from("oops")),
    };

    describe(&bad);
}
```

```
Connected with TLS (cert: /etc/ssl/cert.pem)
Connected with TLS (cert: none)
```

The second connection is in a logically impossible state — connected with TLS
but no certificate, and it has an error message while being connected. The
compiler accepts it because nothing in the types prevents it.

An enum makes the invalid states unrepresentable:

```rust
#[allow(dead_code)]
enum Connection {
    Plain,
    Tls { cert_path: String },
}

enum Status {
    Connected(Connection),
    Disconnected { reason: String },
}

fn describe(status: &Status) {
    match status {
        Status::Connected(Connection::Plain) => {
            println!("Connected without TLS");
        }
        Status::Connected(Connection::Tls { cert_path }) => {
            println!("Connected with TLS (cert: {cert_path})");
        }
        Status::Disconnected { reason } => {
            println!("Disconnected: {reason}");
        }
    }
}

fn main() {
    let status = Status::Connected(Connection::Tls {
        cert_path: String::from("/etc/ssl/cert.pem"),
    });

    describe(&status);

    let offline = Status::Disconnected {
        reason: String::from("timeout"),
    };

    describe(&offline);
}
```

```
Connected with TLS (cert: /etc/ssl/cert.pem)
Disconnected: timeout
```

There is no way to construct a "connected with TLS but no certificate" value.
There is no way to construct a "connected with an error message" value. The
illegal states do not exist in the type system. Exhaustive `match` guarantees
every valid state is handled.

## Reaching for Unsafe

The `unsafe` keyword tells the compiler: "I am opting out of your safety
guarantees for this block. I promise the invariants hold." It exists because
some operations — calling C functions, implementing custom allocators,
manipulating raw pointers for data structure internals — cannot be verified by
the compiler. But `unsafe` is a scalpel, not a hammer.

### The Temptation

When the borrow checker rejects code that you _know_ is correct, the temptation
is to reach for `unsafe` to force it through:

```rust
fn first_and_last(slice: &mut [i32]) -> Option<(&mut i32, &mut i32)> {
    if slice.len() < 2 {
        return None;
    }

    // Safe alternative using split_at_mut
    let (first_half, second_half) = slice.split_at_mut(1);
    Some((&mut first_half[0], second_half.last_mut().unwrap()))
}

fn main() {
    let mut data = vec![10, 20, 30, 40, 50];

    if let Some((first, last)) = first_and_last(&mut data) {
        *first += 100;
        *last += 100;
    }

    println!("{data:?}");
}
```

```
[110, 20, 30, 40, 150]
```

The standard library provides `split_at_mut` specifically for this case — it
splits a mutable slice into two non-overlapping mutable slices, which the borrow
checker can verify are disjoint. No `unsafe` needed.

### The Checklist Before Reaching for Unsafe

Before writing `unsafe`, exhaust these alternatives:

1. **Check the standard library.** Methods like `split_at_mut`, `chunks_mut`,
   slice patterns, and `Cell`/`RefCell` solve many problems that seem to require
   raw pointers.

2. **Restructure the code.** Often the borrow checker is telling you that your
   data structure has a design problem. Can you split the struct? Can you use
   indices instead of references?

3. **Check the ecosystem.** Crates like bytemuck (safe type casting), crossbeam
   (safe concurrent data structures), and parking_lot (optimized locks) provide
   safe APIs over internally sound `unsafe` code.

4. **If you must use unsafe, contain it.** Wrap the `unsafe` block in a function
   with a safe public API. Document the invariants that must hold. Keep the
   `unsafe` surface area as small as possible.

### Rust 2024 Tightens Unsafe

In the 2024 edition, the `unsafe_op_in_unsafe_fn` lint is warn-by-default.
Unsafe operations inside an `unsafe fn` must be wrapped in explicit `unsafe {}`
blocks. This forces you to identify exactly which operations are unsafe, rather
than treating the entire function body as an unsafe zone:

```rust
// Rust 2024: each unsafe operation must be explicitly marked
unsafe fn process_raw(ptr: *const i32, len: usize) -> i32 {
    let mut sum = 0;
    for i in 0..len {
        // Explicit unsafe block around each unsafe operation
        sum += unsafe { *ptr.add(i) };
    }
    sum
}

fn main() {
    let data = [1, 2, 3, 4, 5];

    // Caller must use unsafe to call an unsafe function
    let total = unsafe { process_raw(data.as_ptr(), data.len()) };
    println!("Total: {total}");
}
```

```
Total: 15
```

This is a deliberate design choice: even inside an `unsafe fn`, the compiler
wants you to think about each individual operation that steps outside the safety
model. The more precisely you mark your unsafe code, the easier it is to audit.

## Overusing Smart Pointers

`Rc<T>` and `Arc<T>` provide shared ownership — multiple parts of your program
can own the same value. Developers coming from garbage-collected languages
sometimes reach for them by default, wrapping everything in `Arc` "just to be
safe." But shared ownership has costs, and most Rust code does not need it.

### The Overhead

`Rc` maintains a reference count. Every `Rc::clone` increments it, every drop
decrements it. `Arc` does the same with atomic operations, which are more
expensive because they must synchronize across CPU cores. Neither can be
optimized away by the compiler the way a simple move can.

```rust
fn process(data: Vec<u8>) -> usize {
    data.len()
}

fn process_ref(data: &[u8]) -> usize {
    data.len()
}

fn main() {
    let data = vec![1, 2, 3, 4, 5];

    // Move: zero overhead, transfers ownership
    let len = process(data);
    println!("Length (moved): {len}");

    let data2 = vec![10, 20, 30];

    // Borrow: zero overhead, no ownership transfer
    let len2 = process_ref(&data2);
    println!("Length (borrowed): {len2}");
    println!("Still available: {data2:?}");
}
```

```
Length (moved): 5
Length (borrowed): 3
Still available: [10, 20, 30]
```

Move and borrow cover the vast majority of use cases with zero overhead. Smart
pointers are for the cases where neither works.

### When Smart Pointers Are Correct

Choose the simplest ownership model that satisfies your requirements:

| Situation | Use |
|---|---|
| One clear owner, no sharing needed | Move (default) |
| Temporary read access, owner retains data | `&T` borrow |
| Multiple owners in a single thread | `Rc<T>` |
| Multiple owners across threads | `Arc<T>` |
| Shared ownership with mutation | `Rc<RefCell<T>>` or `Arc<Mutex<T>>` |
| Recursive types or trait objects on the heap | `Box<T>` |

The decision tree is simple: start with ownership and borrowing. If the compiler
rejects it because the lifetime relationships are genuinely complex — multiple
owners, shared state, recursive data — then reach for the appropriate smart
pointer. Do not start with `Arc` and work backwards.

## Preferring Indices over Iterators

Developers with a C or Java background naturally write indexed loops:

```rust
fn main() {
    let words = vec!["hello", "beautiful", "world"];

    // Index-based: verbose, bounds-checked on each access
    let mut lengths_indexed = Vec::new();
    for i in 0..words.len() {
        lengths_indexed.push(words[i].len());
    }

    // Iterator-based: concise, no bounds checks, intent is clear
    let lengths_iter: Vec<usize> = words.iter().map(|w| w.len()).collect();

    println!("Indexed:  {lengths_indexed:?}");
    println!("Iterator: {lengths_iter:?}");
}
```

```
Indexed:  [5, 9, 5]
Iterator: [5, 9, 5]
```

The iterator version is not just shorter — it is faster. The compiler knows that
an iterator over a slice will never go out of bounds, so it can eliminate the
bounds check that `words[i]` requires. Iterator chains also compose: filtering,
mapping, and collecting in a single pass without intermediate allocations.

### Common Translations

Most index-based loops have a direct iterator equivalent:

| Index-based pattern | Iterator equivalent |
|---|---|
| `for i in 0..v.len() { v[i] }` | `for item in &v` |
| `for i in 0..v.len() { (i, v[i]) }` | `for (i, item) in v.iter().enumerate()` |
| Build new vec from old vec | `v.iter().map(\|x\| ...).collect()` |
| Check if any element matches | `v.iter().any(\|x\| ...)` |
| Find first matching element | `v.iter().find(\|x\| ...)` |
| Sum all elements | `v.iter().sum()` |

### When Indices Are Appropriate

Indices are the right tool when you need to mutate the collection based on
relationships between elements at different positions — for example, a sorting
algorithm that swaps elements by index. They are also necessary when the
algorithm requires random access to non-adjacent elements. But for sequential
processing — the most common case — iterators are shorter, faster, and more
expressive.

## Putting It All Together

Here is a small program that demonstrates several anti-patterns side by side
with their idiomatic replacements. The program processes a list of scores and
produces a report:

```rust
use std::fmt;

// Use an enum, not a string, for categories
#[derive(Debug)]
enum Grade {
    Excellent,
    Good,
    NeedsImprovement,
}

impl fmt::Display for Grade {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            Grade::Excellent => write!(f, "Excellent"),
            Grade::Good => write!(f, "Good"),
            Grade::NeedsImprovement => write!(f, "Needs Improvement"),
        }
    }
}

// Use a newtype for type safety, not a bare f64
#[derive(Debug, Clone, Copy, PartialEq, PartialOrd)]
struct Score(f64);

impl Score {
    fn new(value: f64) -> Option<Score> {
        if (0.0..=100.0).contains(&value) {
            Some(Score(value))
        } else {
            None
        }
    }

    fn grade(self) -> Grade {
        match self.0 as u32 {
            90..=100 => Grade::Excellent,
            70..=89 => Grade::Good,
            _ => Grade::NeedsImprovement,
        }
    }
}

impl fmt::Display for Score {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "{:.1}", self.0)
    }
}

fn generate_report(raw_scores: &[f64]) -> String {
    // Use iterators, not indexed loops
    // Use filter_map, not unwrap
    // Use combinators, not nested matches
    let scores: Vec<Score> = raw_scores
        .iter()
        .filter_map(|&s| Score::new(s))
        .collect();

    if scores.is_empty() {
        return String::from("No valid scores to report.");
    }

    let mut lines = Vec::new();

    for (i, score) in scores.iter().enumerate() {
        lines.push(format!(
            "  {}. {} — {}",
            i + 1,
            score,
            score.grade()
        ));
    }

    // Use combinators for aggregation
    let average = scores.iter().map(|s| s.0).sum::<f64>() / scores.len() as f64;
    let best = scores.iter().max_by(|a, b| a.0.total_cmp(&b.0));

    let header = format!("Score Report ({} valid scores):", scores.len());
    let footer = format!(
        "  Average: {:.1} | Best: {}",
        average,
        best.map(|s| s.to_string())
            .unwrap_or_else(|| String::from("none")),
    );

    format!("{header}\n{}\n{footer}", lines.join("\n"))
}

fn main() {
    let raw = [95.0, 82.0, 67.5, 101.0, -5.0, 88.0, 73.0];
    let report = generate_report(&raw);
    println!("{report}");
}
```

```
Score Report (5 valid scores):
  1. 95.0 — Excellent
  2. 82.0 — Good
  3. 67.5 — Needs Improvement
  4. 88.0 — Good
  5. 73.0 — Good
  Average: 81.1 | Best: 95.0
```

This example demonstrates several of the chapter's lessons:

- **No cloning:** scores are `Copy` (wrapping `f64`), so no heap allocations.
- **No unwrap in logic:** `Score::new` returns `Option`, invalid scores are
  filtered out with `filter_map`, not panicked on.
- **No stringly-typed code:** `Grade` is an enum, not a `&str`.
- **No boolean blindness:** the grade is determined by pattern matching on
  ranges, not by checking boolean flags.
- **No index-based loops:** iterators handle filtering, mapping, enumeration,
  and aggregation.
- **Newtype safety:** `Score(f64)` prevents accidentally passing a raw `f64`
  where a validated score is expected.

---

The anti-patterns in this chapter share a common root: they all involve working
_against_ Rust's design instead of _with_ it. The borrow checker is not an
obstacle — it is a code reviewer that catches bugs before they ship. The type
system is not a burden — it is a proof system that eliminates entire categories
of errors. `unsafe` is not a solution — it is an escape hatch for rare cases
where the compiler cannot verify your intent.

The key lessons from this chapter:

- **Do not clone to silence the borrow checker.** Restructure your code so the
  conflicting accesses do not overlap. Use borrows, narrow scopes, and accept
  `&str`/`&[T]` in function signatures.
- **Do not use `unwrap()` in library or production code.** Return `Result`, use
  `?` for propagation, and reserve `unwrap` for tests and prototypes.
- **Do not write Java/Python/C++ in Rust syntax.** Skip getter/setter
  boilerplate, prefer generics over `dyn Trait`, and use composition instead of
  faking inheritance with `Deref`.
- **Do not fight the type system.** Use enums instead of strings, named types
  instead of booleans, and make invalid states unrepresentable.
- **Do not reach for `unsafe` before exhausting safe alternatives.** Check the
  standard library, restructure your data, and contain `unsafe` when you must
  use it.
- **Do not default to `Rc`/`Arc`.** Start with ownership and borrowing; reach
  for smart pointers only when lifetimes are genuinely complex.
- **Do not prefer indices over iterators.** Iterator chains are shorter, faster,
  and more expressive for sequential processing.
