# Structs and Methods

So far, every value you have worked with has been a standalone piece of data — an
integer, a string, a boolean. Real programs model things that have multiple
related pieces of information: a user has a name and an email, a rectangle has a
width and a height, a sensor reading has a value and a timestamp. Keeping these
as separate variables works for a while, but it falls apart quickly:

```rust
fn main() {
    let user_name = String::from("Alice");
    let user_email = String::from("alice@example.com");
    let user_active = true;

    let user2_name = String::from("Bob");
    let user2_email = String::from("bob@example.com");
    let user2_active = true;

    println!("{} ({})", user_name, user_email);
    println!("{} ({})", user2_name, user2_email);
}
```

Output:

```
Alice (alice@example.com)
Bob (bob@example.com)
```

Nothing ties `user_name` to `user_email`. Nothing prevents you from accidentally
mixing up Alice's name with Bob's email. The compiler sees six unrelated
variables, not two users.

Rust solves this with _structs_ — custom types that group related data into a
single value. But structs in Rust are more than containers. Combined with
ownership and borrowing, they become the foundation for designing safe,
expressive APIs. The method you attach to a struct declares _how_ it accesses its
data — reading, modifying, or consuming — using the same `&self`, `&mut self`,
and `self` distinction you learned in the borrowing chapter.

## Defining a Struct

A _struct_ defines a new type by grouping named fields together:

```rust
struct User {
    name: String,
    email: String,
    active: bool,
}

fn main() {
    let user = User {
        name: String::from("Alice"),
        email: String::from("alice@example.com"),
        active: true,
    };

    println!("{} ({}) - active: {}", user.name, user.email, user.active);
}
```

Output: `Alice (alice@example.com) - active: true`

The `struct User` declaration creates a new type with three fields. Each field
has a name and a type. You create an instance by providing a value for every
field — the compiler will not let you forget one.

Fields are accessed with dot notation: `user.name`, `user.email`. This is
familiar from most languages, but with an important difference: the struct _owns_
its data. The `String` values live inside the struct, and when the struct is
dropped, its fields are dropped with it. No dangling references, no forgotten
cleanup.

### Field Init Shorthand

When a variable has the same name as a struct field, you can use the shorthand:

```rust
struct Point {
    x: f64,
    y: f64,
}

fn main() {
    let x = 3.0;
    let y = 4.0;

    let p = Point { x, y }; // same as Point { x: x, y: y }

    println!("({}, {})", p.x, p.y);
}
```

Output: `(3, 4)`

This is not a special syntax — it follows naturally from Rust's design. The field
name and the variable name match, so Rust lets you write it once instead of
twice.

### Mutation Requires the Whole Struct to Be Mutable

To change a field, the entire struct must be declared `mut`. Rust does not allow
individual fields to be selectively mutable:

```rust
struct User {
    name: String,
    email: String,
    active: bool,
}

fn main() {
    let mut user = User {
        name: String::from("Alice"),
        email: String::from("alice@example.com"),
        active: true,
    };

    user.email = String::from("alice@newdomain.com");

    println!("{} ({})", user.name, user.email);
}
```

Output: `Alice (alice@newdomain.com)`

This mirrors what you already know about `let mut` — mutability is a property of
the binding, not of individual pieces of data. It keeps things simple: if you can
see `let mut`, the entire value might change. If you see `let`, nothing will.

### Struct Update Syntax

When you want to create a new struct instance that is mostly the same as an
existing one, _struct update syntax_ lets you copy the unchanged fields:

```rust
struct User {
    name: String,
    email: String,
    active: bool,
}

fn main() {
    let user1 = User {
        name: String::from("Alice"),
        email: String::from("alice@example.com"),
        active: true,
    };

    let user2 = User {
        email: String::from("bob@example.com"),
        ..user1
    };

    // user1.name was moved into user2, so user1 is partially moved
    println!("{} - active: {}", user2.email, user2.active);
}
```

Output: `bob@example.com - active: true`

The `..user1` fills in any fields you did not specify. This uses _move_ semantics
for fields that are not `Copy` — `user1.name` (a `String`) moves into `user2`,
so `user1.name` is no longer usable. But `user1.active` (a `bool`, which is
`Copy`) is simply copied.

This is ownership in action. The struct update syntax does not magically clone
data — it moves it, following the same rules you learned in the ownership
chapter.

## Tuple Structs

Sometimes you want a struct with fields but no field names. A _tuple struct_
gives you a named type with positional fields:

```rust
struct Color(u8, u8, u8);
struct Meters(f64);

fn main() {
    let red = Color(255, 0, 0);
    let distance = Meters(42.5);

    println!("red: ({}, {}, {})", red.0, red.1, red.2);
    println!("distance: {} m", distance.0);
}
```

Output:

```
red: (255, 0, 0)
distance: 42.5 m
```

Tuple structs are especially useful for the _newtype pattern_ — wrapping an
existing type to create a distinct type. `Meters(f64)` and `f64` are different
types, so you cannot accidentally add meters to seconds. The compiler enforces
the distinction at zero runtime cost.

## Unit Structs

A struct with no fields is called a _unit struct_:

```rust
struct Marker;

fn main() {
    let _m = Marker;
    println!("unit struct created");
}
```

Output: `unit struct created`

Unit structs take up no space. They are useful as type-level markers — you will
see them used with traits in Part 4 to tag types with certain properties without
carrying any data.

## Methods: Behavior Attached to Data

A struct defines data. An `impl` block attaches behavior to it. Together, they
form a complete type: data and the operations you can perform on that data.

```rust
struct Rectangle {
    width: f64,
    height: f64,
}

impl Rectangle {
    fn area(&self) -> f64 {
        self.width * self.height
    }
}

fn main() {
    let rect = Rectangle { width: 10.0, height: 5.0 };
    println!("area: {}", rect.area());
}
```

Output: `area: 50`

The `impl Rectangle` block says: "here are the methods that belong to
`Rectangle`." The method `area` takes `&self` as its first parameter — a shared
reference to the `Rectangle` instance. You call it with dot notation:
`rect.area()`.

The `&self` parameter is shorthand for `self: &Rectangle`. You do not write the
type explicitly — Rust knows which type the `impl` block belongs to.

### The Three Method Receivers

The first parameter of a method determines how it accesses the struct. This maps
directly to the ownership and borrowing rules you already know:

| Receiver | Meaning | When to Use |
|---|---|---|
| `&self` | Shared reference — read the data | Most methods: queries, computations, formatting |
| `&mut self` | Mutable reference — modify the data | Methods that change the struct's fields |
| `self` | Ownership — consume the struct | Transformations where the original is no longer needed |

Here is each receiver in action:

```rust
struct Account {
    holder: String,
    balance: f64,
}

impl Account {
    // &self: read-only access
    fn summary(&self) -> String {
        format!("{}: ${:.2}", self.holder, self.balance)
    }

    // &mut self: modify in place
    fn deposit(&mut self, amount: f64) {
        self.balance += amount;
    }

    // self: take ownership, return something new
    fn close(self) -> String {
        format!("Closed {}. Final balance: ${:.2}", self.holder, self.balance)
    }
}

fn main() {
    let mut acct = Account {
        holder: String::from("Alice"),
        balance: 100.0,
    };

    println!("{}", acct.summary());

    acct.deposit(50.0);
    println!("{}", acct.summary());

    let farewell = acct.close();
    println!("{farewell}");
    // acct is consumed — using it here would be a compiler error
}
```

Output:

```
Alice: $100.00
Alice: $150.00
Closed Alice. Final balance: $150.00
```

Notice how each method receiver tells you exactly what happens:

- `summary(&self)` borrows the account — you can call it repeatedly without
  giving up ownership.
- `deposit(&mut self)` borrows mutably — the account must be declared `mut`,
  and no other references can exist during the call.
- `close(self)` takes ownership — after calling it, the `Account` is gone.
  The compiler enforces this: any attempt to use `acct` after `close` produces
  error `E0382` (use of moved value).

Choosing the right receiver is one of the most important design decisions in
Rust. The compiler will not let your callers get it wrong.

### Methods with Parameters

Methods can take additional parameters after `self`:

```rust
struct Rectangle {
    width: f64,
    height: f64,
}

impl Rectangle {
    fn can_contain(&self, other: &Rectangle) -> bool {
        self.width > other.width && self.height > other.height
    }
}

fn main() {
    let outer = Rectangle { width: 10.0, height: 8.0 };
    let inner = Rectangle { width: 5.0, height: 4.0 };

    println!("outer can contain inner: {}", outer.can_contain(&inner));
    println!("inner can contain outer: {}", inner.can_contain(&outer));
}
```

Output:

```
outer can contain inner: true
inner can contain outer: false
```

The method borrows `self` as `&self` (read-only) and borrows `other` as
`&Rectangle` (also read-only). Neither rectangle is consumed. After the calls,
both `outer` and `inner` are still valid.

## Associated Functions: No Self Required

Not every function in an `impl` block needs a `self` parameter. Functions without
`self` are called _associated functions_ — they belong to the type itself, not to
an instance:

```rust
struct Rectangle {
    width: f64,
    height: f64,
}

impl Rectangle {
    fn square(size: f64) -> Self {
        Self {
            width: size,
            height: size,
        }
    }

    fn area(&self) -> f64 {
        self.width * self.height
    }
}

fn main() {
    let sq = Rectangle::square(5.0);
    println!("square area: {}", sq.area());
}
```

Output: `square area: 25`

Associated functions are called with `::` syntax — `Rectangle::square(5.0)` —
because there is no instance to call them on. The `Self` keyword refers to the
type being implemented, so `Self { width: size, height: size }` is the same as
`Rectangle { width: size, height: size }`.

### The New Convention

Rust has no special constructor syntax. No `new` keyword, no constructor
functions with magic behavior. Instead, the community follows a convention: an
associated function named `new` that returns `Self`:

```rust
struct User {
    name: String,
    email: String,
    active: bool,
}

impl User {
    fn new(name: String, email: String) -> Self {
        Self {
            name,
            email,
            active: true, // sensible default
        }
    }
}

fn main() {
    let user = User::new(
        String::from("Alice"),
        String::from("alice@example.com"),
    );

    println!("{} ({}) - active: {}", user.name, user.email, user.active);
}
```

Output: `Alice (alice@example.com) - active: true`

The `new` function is just an associated function. It has no special status in
the language — the compiler treats it like any other function. But by convention,
`new` is the primary way to construct a type. It can set defaults (like `active:
true`), validate inputs, or perform any setup the type needs.

This convention is powerful because it is simple. There are no constructor
overloading rules, no initialization order subtleties, no `super()` chains. A
`new` function is a regular function that happens to return `Self`.

## Printing Your Types

If you try to print a struct with `{}` or `{:?}`, the compiler tells you it
does not know how to display it:

```rust,does_not_compile
struct Point {
    x: f64,
    y: f64,
}

fn main() {
    let p = Point { x: 3.0, y: 4.0 };
    println!("{:?}", p); // error: `Point` doesn't implement `Debug`
}
```

```
error[E0277]: `Point` doesn't implement `Debug`
 --> src/main.rs:8:22
  |
8 |     println!("{:?}", p);
  |                      ^ `Point` cannot be formatted using `{:?}`
  |                        because it doesn't implement `Debug`
  |
  = help: the trait `Debug` is not implemented for `Point`
  = note: add `#[derive(Debug)]` to `Point` or manually `impl Debug for Point`
```

Rust does not guess how to display your types. Instead, it provides two
mechanisms: `Debug` for developer-facing output and `Display` for user-facing
output.

### Debug: Developer Output with Derive

The fastest way to make a struct printable is to _derive_ the `Debug` trait. Add
`#[derive(Debug)]` above the struct definition, and Rust generates the
formatting code for you:

```rust
#[derive(Debug)]
struct Point {
    x: f64,
    y: f64,
}

fn main() {
    let p = Point { x: 3.0, y: 4.0 };
    println!("{:?}", p);
    println!("{:#?}", p); // pretty-printed version
}
```

Output:

```
Point { x: 3.0, y: 4.0 }
Point {
    x: 3.0,
    y: 4.0,
}
```

The `{:?}` format prints a compact representation. The `{:#?}` format prints a
pretty-printed version with indentation — useful for nested structs or
debugging complex data.

`#[derive(Debug)]` is an _attribute_ that tells the compiler to generate an
implementation of the `Debug` trait automatically. The word _derive_ means "the
compiler figures out the implementation based on your fields." You will see
`derive` used for other traits too — `Clone`, `PartialEq`, `Default` — and they
all work the same way: add the attribute, get the behavior for free.

### Display: User-Facing Output

`Debug` output is for developers — log files, debug sessions, error messages.
For output that users see, implement the `Display` trait manually:

```rust
use std::fmt;

struct Point {
    x: f64,
    y: f64,
}

impl fmt::Display for Point {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "({}, {})", self.x, self.y)
    }
}

fn main() {
    let p = Point { x: 3.0, y: 4.0 };
    println!("point: {}", p);
    println!("the origin is {}", Point { x: 0.0, y: 0.0 });
}
```

Output:

```
point: (3, 4)
the origin is (0, 0)
```

The `fmt::Display` trait has one required method: `fmt`. It takes `&self` (a
shared reference to your type) and a `Formatter` that handles the actual
writing. You use `write!` — the same macro as `println!`, but writing into the
formatter instead of to the screen.

`Display` is never derived — you always implement it by hand. This makes sense:
`Debug` has an obvious mechanical output (show every field with its name), but
user-facing formatting depends on your domain. A `Point` might display as
`(3, 4)`, a `Temperature` as `72°F`, a `Money` as `$42.50`. Only you know what
is appropriate.

A common pattern is to derive `Debug` for development and implement `Display`
for the public interface. Many types in the standard library do exactly this.

## Structs and Ownership

Structs follow the same ownership rules as every other value in Rust. When a
struct goes out of scope, it is dropped, and all of its fields are dropped with
it. When you move a struct, all of its fields move together:

```rust
#[derive(Debug)]
struct Config {
    name: String,
    max_retries: u32,
}

fn apply_config(config: Config) {
    println!("applying: {:?}", config);
}

fn main() {
    let config = Config {
        name: String::from("production"),
        max_retries: 3,
    };

    apply_config(config);
    // config is moved — cannot use it here
}
```

Output: `applying: Config { name: "production", max_retries: 3 }`

The function `apply_config` takes a `Config` by value, which moves it. After the
call, `config` is no longer available. If you need to keep the original, pass a
reference instead:

```rust
#[derive(Debug)]
struct Config {
    name: String,
    max_retries: u32,
}

fn print_config(config: &Config) {
    println!("config: {:?}", config);
}

fn main() {
    let config = Config {
        name: String::from("production"),
        max_retries: 3,
    };

    print_config(&config);
    print_config(&config); // still valid — we only borrowed
    println!("name: {}", config.name);
}
```

Output:

```
config: Config { name: "production", max_retries: 3 }
config: Config { name: "production", max_retries: 3 }
name: production
```

This is the same ownership and borrowing you have used all along. Structs do not
add new rules — they follow the existing ones. Choosing between `self`,
`&self`, and `&mut self` in your methods is the same decision you make every time
you pass a value to a function.

## Putting It All Together

Here is a complete example that uses everything from this chapter: struct
definition, field init shorthand, methods with all three receivers, an associated
function, and both `Debug` and `Display` formatting:

```rust
use std::fmt;

#[derive(Debug)]
struct Task {
    title: String,
    priority: u8,
    done: bool,
}

impl Task {
    fn new(title: String, priority: u8) -> Self {
        Self {
            title,
            priority,
            done: false,
        }
    }

    fn is_high_priority(&self) -> bool {
        self.priority >= 8
    }

    fn complete(&mut self) {
        self.done = true;
    }

    fn into_summary(self) -> String {
        let status = if self.done { "done" } else { "pending" };
        format!("[{}] {} (priority {})", status, self.title, self.priority)
    }
}

impl fmt::Display for Task {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        let marker = if self.done { "x" } else { " " };
        write!(f, "[{}] {}", marker, self.title)
    }
}

fn main() {
    let mut task = Task::new(String::from("Write chapter 3"), 9);
    println!("debug: {:?}", task);
    println!("display: {task}");
    println!("high priority: {}", task.is_high_priority());

    task.complete();
    println!("after completing: {task}");

    let summary = task.into_summary();
    println!("{summary}");
    // task is consumed — it lived a good life
}
```

Output:

```
debug: Task { title: "Write chapter 3", priority: 9, done: false }
display: [ ] Write chapter 3
high priority: true
after completing: [x] Write chapter 3
[done] Write chapter 3 (priority 9)
```

Every design choice here reflects what you have learned:

- `new` is a plain associated function that sets sensible defaults (`done:
  false`).
- `is_high_priority(&self)` only reads the data, so it takes a shared reference.
- `complete(&mut self)` modifies the struct, so it requires a mutable reference.
- `into_summary(self)` consumes the struct, signaling that the `Task` is used up
  after producing the summary. The `into_` prefix is a Rust convention for
  methods that take ownership.
- `Display` provides clean user-facing output (`[x] Write chapter 3`).
- `Debug` provides developer-facing output with all fields visible.

## Why This Matters

Structs are how you model your domain in Rust. They are not just containers for
data — combined with `impl` blocks, they become types with clear interfaces.
The method receiver (`&self`, `&mut self`, `self`) declares the contract between
the type and its callers: what access is needed, and whether ownership is
transferred.

This is what makes Rust APIs self-documenting. When you see a method that takes
`&self`, you know it will not modify or consume the value. When you see `self`,
you know the value is gone after the call. The compiler checks every call site
against these contracts. No documentation drift, no runtime surprises.

Key points from this chapter:

- **Structs** group related data into a named type. Fields are accessed with
  dot notation. The struct owns its fields.
- **Methods** are defined in `impl` blocks. The first parameter (`&self`,
  `&mut self`, or `self`) determines how the method accesses the struct.
- **Associated functions** (no `self` parameter) belong to the type, not an
  instance. The `new` convention is how Rust replaces constructors.
- **`#[derive(Debug)]`** generates developer-facing output for `{:?}`.
  Implement `Display` manually for user-facing output with `{}`.
- **Ownership applies to structs** just like every other value. Moving a struct
  moves its fields. Borrowing a struct borrows its fields.

---

With structs, you can model single concepts — a user, a rectangle, a task. But
what about values that can be _one of several_ things? A shape could be a circle
_or_ a rectangle. A network response could be success _or_ failure. In the next
chapter, you will learn how Rust models these alternatives with _enums_ —
types that carry data and guarantee you handle every case.
