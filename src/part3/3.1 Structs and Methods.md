# Structs and Methods

So far, every value you have worked with has been a standalone piece of data — an
integer, a string, a boolean. Real programs model things that have multiple
related pieces of information: a user has a name and an email, a rectangle has a
width and a height, a sensor reading has a value and a timestamp. Keeping these
as separate variables works for a while, but it falls apart quickly:

```rust
fn main() {
    let user_name = String::from("Alice");
    let user_email = String::from("alice@example.com");
    let user_active = true;

    let user2_name = String::from("Bob");
    let user2_email = String::from("bob@example.com");
    let user2_active = true;

    println!("{} ({}) - active: {}", user_name, user_email, user_active);
    println!("{} ({}) - active: {}", user2_name, user2_email, user2_active);
}
```

Output:

```
Alice (alice@example.com) - active: true
Bob (bob@example.com) - active: true
```

Nothing ties `user_name` to `user_email` to `user_active`. Nothing prevents you
from accidentally mixing up Alice's name with Bob's email. The compiler sees six
unrelated variables, not two users.

Rust solves this with _structs_ — custom types that group related data into a
single value. If you are coming from an object-oriented language, structs are
Rust's answer to classes. There is no `class` keyword, no inheritance, no
constructor syntax. Instead, you define data with a struct, attach behavior with
an `impl` block, and share interfaces with traits (covered in a later chapter).

But structs in Rust are more than containers. Combined with ownership and
borrowing, they become the foundation for designing safe, expressive APIs. The
method you attach to a struct declares _how_ it accesses its data — reading,
modifying, or consuming — using the same `&self`, `&mut self`, and `self`
distinction you learned in the borrowing chapter.

## Defining a Struct

A _struct_ defines a new type by grouping named fields together:

```rust
struct User {
    name: String,
    email: String,
    active: bool,
}

fn main() {
    let user = User {
        name: String::from("Alice"),
        email: String::from("alice@example.com"),
        active: true,
    };

    println!("{} ({}) - active: {}", user.name, user.email, user.active);
}
```

Output: `Alice (alice@example.com) - active: true`

The `struct User` declaration creates a new type with three fields. Each field
has a name and a type. You create an instance by providing a value for every
field — the compiler will not let you forget one.

Fields are accessed with dot notation: `user.name`, `user.email`. This is
familiar from most languages, but with an important difference: the struct _owns_
its data. The `String` values live inside the struct, and when the struct is
_dropped_ (Rust's term for cleanup when a value goes out of scope), its fields
are dropped with it. This is automatic — Rust inserts the cleanup code for you.
If you need custom cleanup logic (closing a file handle, releasing a network
connection), you can implement the `Drop` trait, but most structs never need it
because dropping each field is the right behavior. No dangling references, no
forgotten cleanup.

> **Why `String`, not `&str`, for struct fields**
>
> A natural instinct is to write `name: &str` to avoid allocating. It will not
> compile — the compiler will say `expected named lifetime parameter`. A
> struct that holds a reference must declare how long that reference is
> valid, which means adding a lifetime parameter to the struct itself
> (`struct User<'a> { name: &'a str, ... }`). This is a real and useful
> pattern, but it is the advanced choice. For now, use owned types
> (`String`, `Vec<T>`) for fields — the struct owns its data, no lifetimes
> required, and you do not have to track who outlives whom. We return to
> structs that borrow in a later chapter.

### Field Init Shorthand

When a variable has the same name as a struct field, you can use the shorthand:

```rust
struct Point {
    x: f64,
    y: f64,
}

fn main() {
    let x = 3.0;
    let y = 4.0;

    let p = Point { x, y }; // same as Point { x: x, y: y }

    println!("({}, {})", p.x, p.y);
}
```

Output: `(3, 4)`

This is not a special syntax — it follows naturally from Rust's design. The field
name and the variable name match, so Rust lets you write it once instead of
twice.

### Mutation Requires the Whole Struct to Be Mutable

To change a field, the entire struct must be declared `mut`. Rust does not allow
individual fields to be selectively mutable:

```rust
struct User {
    name: String,
    email: String,
    active: bool,
}

fn main() {
    let mut user = User {
        name: String::from("Alice"),
        email: String::from("alice@example.com"),
        active: true,
    };

    user.email = String::from("alice@newdomain.com");

    println!("{} ({}) - active: {}", user.name, user.email, user.active);
}
```

Output: `Alice (alice@newdomain.com) - active: true`

This mirrors what you already know about `let mut` — mutability is a property of
the binding, not of individual pieces of data. It keeps things simple: if you can
see `let mut`, the entire value might change. If you see `let`, nothing will.

### Struct Update Syntax

When you want to create a new struct instance that is mostly the same as an
existing one, _struct update syntax_ lets you copy the unchanged fields:

```rust
struct User {
    name: String,
    email: String,
    active: bool,
}

fn main() {
    let user1 = User {
        name: String::from("Alice"),
        email: String::from("alice@example.com"),
        active: true,
    };

    let user2 = User {
        email: String::from("bob@example.com"),
        ..user1
    };

    // user1.name was moved into user2, so user1 is partially moved
    println!("{} ({}) - active: {}", user2.name, user2.email, user2.active);
}
```

Output: `Alice (bob@example.com) - active: true`

The `..user1` fills in any fields you did not specify. This uses _move_ semantics
for fields that are not `Copy` — `user1.name` (a `String`) moves into `user2`,
so `user1.name` is no longer usable. But `user1.active` (a `bool`, which is
`Copy`) is simply copied.

This is ownership in action. The struct update syntax does not magically clone
data — it moves it, following the same rules you learned in the ownership
chapter.

## Tuple Structs

Sometimes you want a struct with fields but no field names. A _tuple struct_
gives you a named type with positional fields:

```rust
struct Color(u8, u8, u8);
struct Meters(f64);

fn main() {
    let red = Color(255, 0, 0);
    let distance = Meters(42.5);

    println!("red: ({}, {}, {})", red.0, red.1, red.2);
    println!("distance: {} m", distance.0);
}
```

Output:

```
red: (255, 0, 0)
distance: 42.5 m
```

Tuple structs are especially useful for the _newtype pattern_ — wrapping an
existing type to create a distinct type. `Meters(f64)` and `f64` are different
types, so you cannot accidentally add meters to seconds. The compiler enforces
the distinction at zero runtime cost.

## Unit Structs

A struct with no fields is called a _unit struct_:

```rust
struct Marker;

fn main() {
    let _m = Marker;
    println!("unit struct created");
}
```

Output: `unit struct created`

Unit structs take up no space. They are useful as type-level markers — a way to
tag types with certain properties without carrying any data. For example, you
might define `struct Ascending;` and `struct Descending;` to represent sort
orders, using different types to distinguish intent at compile time rather than
passing a boolean or string.

## Methods: Behavior Attached to Data

A struct defines data. An `impl` block attaches behavior to it. Together, they
form a complete type: data and the operations you can perform on that data.

```rust
struct Rectangle {
    width: f64,
    height: f64,
}

impl Rectangle {
    fn area(&self) -> f64 {
        self.width * self.height
    }
}

fn main() {
    let rect = Rectangle { width: 10.0, height: 5.0 };
    println!("area: {}", rect.area());
}
```

Output: `area: 50`

The `impl Rectangle` block says: "here are the methods that belong to
`Rectangle`." The method `area` takes `&self` as its first parameter — a shared
reference to the `Rectangle` instance. You call it with dot notation:
`rect.area()`.

The `&self` parameter is shorthand for `self: &Rectangle`. You do not write the
type explicitly — Rust knows which type the `impl` block belongs to.

### The Three Method Receivers

The first parameter of a method determines how it accesses the struct. This maps
directly to the ownership and borrowing rules you already know:

| Receiver | Meaning | When to Use |
|---|---|---|
| `&self` | Shared reference — read the data | Most methods: queries, computations, formatting |
| `&mut self` | Mutable reference — modify the data | Methods that change the struct's fields |
| `self` | Ownership — consume the struct | Transformations where the original is no longer needed |

Here is each receiver in action:

```rust
struct Account {
    holder: String,
    balance: f64,
}

impl Account {
    // &self: read-only access
    fn summary(&self) -> String {
        format!("{}: ${:.2}", self.holder, self.balance)
    }

    // &mut self: modify in place
    fn deposit(&mut self, amount: f64) {
        self.balance += amount;
    }

    // self: take ownership, return something new
    fn close(self) -> String {
        format!("Closed {}. Final balance: ${:.2}", self.holder, self.balance)
    }
}

fn main() {
    let mut acct = Account {
        holder: String::from("Alice"),
        balance: 100.0,
    };

    println!("{}", acct.summary());

    acct.deposit(50.0);
    println!("{}", acct.summary());

    let farewell = acct.close();
    println!("{farewell}");
    // acct is consumed — using it here would be a compiler error
}
```

Output:

```
Alice: $100.00
Alice: $150.00
Closed Alice. Final balance: $150.00
```

Notice how each method receiver tells you exactly what happens:

- `summary(&self)` borrows the account — you can call it repeatedly without
  giving up ownership.
- `deposit(&mut self)` borrows mutably — the account must be declared `mut`,
  and no other references can exist during the call.
- `close(self)` takes ownership — after calling it, the `Account` is gone.
  The compiler enforces this: any attempt to use `acct` after `close` produces
  error `E0382` (use of moved value).

Choosing the right receiver is one of the most important design decisions in
Rust. The compiler will not let your callers get it wrong.

### Methods with Parameters

Methods can take additional parameters after `self`:

```rust
struct Rectangle {
    width: f64,
    height: f64,
}

impl Rectangle {
    fn can_contain(&self, other: &Rectangle) -> bool {
        self.width > other.width && self.height > other.height
    }
}

fn main() {
    let outer = Rectangle { width: 10.0, height: 8.0 };
    let inner = Rectangle { width: 5.0, height: 4.0 };

    println!("outer can contain inner: {}", outer.can_contain(&inner));
    println!("inner can contain outer: {}", inner.can_contain(&outer));
}
```

Output:

```
outer can contain inner: true
inner can contain outer: false
```

The method borrows `self` as `&self` (read-only) and borrows `other` as
`&Rectangle` (also read-only). Neither rectangle is consumed. After the calls,
both `outer` and `inner` are still valid.

## Associated Functions: No Self Required

Not every function in an `impl` block needs a `self` parameter. Functions without
`self` are called _associated functions_ — they belong to the type itself, not to
an instance:

```rust
struct Rectangle {
    width: f64,
    height: f64,
}

impl Rectangle {
    fn square(size: f64) -> Self {
        Self {
            width: size,
            height: size,
        }
    }

    fn area(&self) -> f64 {
        self.width * self.height
    }
}

fn main() {
    let sq = Rectangle::square(5.0);
    println!("square area: {}", sq.area());
}
```

Output: `square area: 25`

Associated functions are called with `::` syntax — `Rectangle::square(5.0)` —
because there is no instance to call them on. The `Self` keyword refers to the
type being implemented, so `Self { width: size, height: size }` is the same as
`Rectangle { width: size, height: size }`.

### The New Convention

Rust has no special constructor syntax. No `new` keyword, no constructor
functions with magic behavior. Instead, the community follows a convention: an
associated function named `new` that returns `Self`:

```rust
struct User {
    name: String,
    email: String,
    active: bool,
}

impl User {
    fn new(name: String, email: String) -> Self {
        Self {
            name,
            email,
            active: true, // sensible default
        }
    }
}

fn main() {
    let user = User::new(
        String::from("Alice"),
        String::from("alice@example.com"),
    );

    println!("{} ({}) - active: {}", user.name, user.email, user.active);
}
```

Output: `Alice (alice@example.com) - active: true`

The `new` function is just an associated function. It has no special status in
the language — the compiler treats it like any other function. But by convention,
`new` is the primary way to construct a type. It can set defaults (like `active:
true`), validate inputs, or perform any setup the type needs.

This convention is powerful because it is simple. There are no constructor
overloading rules, no initialization order subtleties, no `super()` chains. A
`new` function is a regular function that happens to return `Self`.

You may notice that calling `String::from("Alice")` everywhere is verbose. A
common alternative is to accept `&str` and convert inside the function:

```rust
fn new(name: &str, email: &str) -> Self {
    Self {
        name: name.to_string(),
        email: email.to_string(),
        active: true,
    }
}
```

This makes call sites cleaner — `User::new("Alice", "alice@example.com")` — at
the cost of always allocating, even if the caller already has a `String`. Both
approaches are common in Rust. The exercises in this chapter use `String`
parameters to keep the focus on struct mechanics, but you will see `&str`
parameters frequently in real code.

## Printing Your Types

If you try to print a struct with `{}` or `{:?}`, the compiler tells you it
does not know how to display it:

```rust,does_not_compile
struct Point {
    x: f64,
    y: f64,
}

fn main() {
    let p = Point { x: 3.0, y: 4.0 };
    println!("{:?}", p); // error: `Point` doesn't implement `Debug`
}
```

```
error[E0277]: `Point` doesn't implement `Debug`
 --> src/main.rs:8:22
  |
8 |     println!("{:?}", p);
  |               ----   ^ `Point` cannot be formatted using `{:?}`
  |               |        because it doesn't implement `Debug`
  |               required by this formatting parameter
  |
  = help: the trait `Debug` is not implemented for `Point`
  = note: add `#[derive(Debug)]` to `Point` or manually `impl Debug for Point`
```

Rust does not guess how to display your types. Instead, it provides two
mechanisms: `Debug` for developer-facing output and `Display` for user-facing
output.

### Debug: Developer Output with Derive

The fastest way to make a struct printable is to _derive_ the `Debug` trait. Add
`#[derive(Debug)]` above the struct definition, and Rust generates the
formatting code for you:

```rust
#[derive(Debug)]
struct Point {
    x: f64,
    y: f64,
}

fn main() {
    let p = Point { x: 3.0, y: 4.0 };
    println!("{:?}", p);
    println!("{:#?}", p); // pretty-printed version
    println!("coordinates: {}, {}", p.x, p.y);
}
```

Output:

```
Point { x: 3.0, y: 4.0 }
Point {
    x: 3.0,
    y: 4.0,
}
coordinates: 3, 4
```

The `{:?}` format prints a compact representation. The `{:#?}` format prints a
pretty-printed version with indentation — useful for nested structs or
debugging complex data.

`#[derive(Debug)]` is an _attribute_ that tells the compiler to generate an
implementation of the `Debug` trait automatically. The word _derive_ means "the
compiler figures out the implementation based on your fields." You will see
`derive` used for other traits too — `Clone`, `PartialEq`, `Default` — and they
all work the same way: add the attribute, get the behavior for free.

### Display: User-Facing Output

`Debug` output is for developers — log files, debug sessions, error messages.
For output that users see, implement the `Display` trait manually. But how do
you know what to write? Start with an empty implementation and let the compiler
guide you:

```rust,does_not_compile
use std::fmt;

struct Point {
    x: f64,
    y: f64,
}

impl fmt::Display for Point {}
```

```
error[E0046]: not all trait items implemented, missing: `fmt`
 --> src/main.rs:7:1
  |
7 | impl fmt::Display for Point {}
  | ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ missing `fmt` in implementation
  |
  = help: implement the missing item:
      `fn fmt(&self, _: &mut Formatter<'_>) -> Result<(), Error> { todo!() }`
```

The compiler tells you exactly what method is missing and what its signature
should be. This is how experienced Rust programmers discover trait requirements —
write the stub, read the error, fill in the blanks. Now you know what to write:

```rust
use std::fmt;

struct Point {
    x: f64,
    y: f64,
}

impl fmt::Display for Point {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "({}, {})", self.x, self.y)
    }
}

fn main() {
    let p = Point { x: 3.0, y: 4.0 };
    println!("point: {}", p);
    println!("the origin is {}", Point { x: 0.0, y: 0.0 });
}
```

Output:

```
point: (3, 4)
the origin is (0, 0)
```

The `fmt::Display` trait has one required method: `fmt`. It takes `&self` (a
shared reference to your type) and a `Formatter` that handles the actual
writing. You use `write!` — similar to `format!`, but with an important
difference: `format!` creates a new `String`, while `write!` writes directly
into a destination (here, the formatter `f`). That is why `f` comes first:
`write!(f, "...")`. Do not use `format!` inside `fmt` — it would allocate a
`String` only to copy it into the formatter, which is wasteful and does not
return the right type.

> **Just Enough: `<'_>`**
>
> The `<'_>` in `fmt::Formatter<'_>` is an _anonymous lifetime_ — it tells the
> compiler "this type holds a reference; please figure out the lifetime from
> context." You saw _named_ lifetimes (`<'a>`) in chapter 2.4, used to tie
> input and output references together. `<'_>` is the version you write when
> you do not need to refer to the lifetime by name, only acknowledge it
> exists. Every `Display` and `Debug` impl uses this exact signature — copy
> it verbatim. We return to lifetimes later; for now, `<'_>` is required
> boilerplate.

`Display` is never derived — you always implement it by hand. This makes sense:
`Debug` has an obvious mechanical output (show every field with its name), but
user-facing formatting depends on your domain. A `Point` might display as
`(3, 4)`, a `Temperature` as `72°F`, a `Money` as `$42.50`. Only you know what
is appropriate.

A common pattern is to derive `Debug` for development and implement `Display`
for the public interface. Many types in the standard library do exactly this.

### Discovering What a Trait Requires

You saw the compiler reveal `Display`'s required method when you wrote an empty
`impl` block. This works for any trait, and it is the fastest way to get started.
Two other approaches are also useful:

- **Read the docs.** Every trait in the standard library lists its required
  methods at `doc.rust-lang.org/std`. Search for the trait name and look for the
  "Required Methods" section.
- **Run `cargo doc --open`.** This generates browsable documentation for your
  project and all its dependencies, available locally in your browser.

Traits split their methods into _required_ (you must implement) and _provided_
(have defaults you can optionally override). `Display` has one required method:
`fmt`. `Debug` also has one, but you rarely write it by hand because `derive`
generates it for you. You will see this required-vs-provided pattern throughout
Rust's standard library — most traits keep the required surface small.

## Structs and Ownership

Structs follow the same ownership rules as every other value in Rust. When a
struct goes out of scope, it is dropped, and all of its fields are dropped with
it. When you move a struct, all of its fields move together:

```rust
struct Config {
    name: String,
    max_retries: u32,
}

fn apply_config(config: Config) {
    println!("applying {} (max {} retries)", config.name, config.max_retries);
}

fn main() {
    let config = Config {
        name: String::from("production"),
        max_retries: 3,
    };

    apply_config(config);
    // config is moved — cannot use it here
}
```

Output: `applying production (max 3 retries)`

The function `apply_config` takes a `Config` by value, which moves it. After the
call, `config` is no longer available. If you need to keep the original, pass a
reference instead:

```rust
#[derive(Debug)]
struct Config {
    name: String,
    max_retries: u32,
}

fn print_config(config: &Config) {
    println!("config: {:?}", config);
}

fn main() {
    let config = Config {
        name: String::from("production"),
        max_retries: 3,
    };

    print_config(&config);
    print_config(&config); // still valid — we only borrowed
    println!("name: {}", config.name);
}
```

Output:

```
config: Config { name: "production", max_retries: 3 }
config: Config { name: "production", max_retries: 3 }
name: production
```

This is the same ownership and borrowing you have used all along. Structs do not
add new rules — they follow the existing ones. Choosing between `self`,
`&self`, and `&mut self` in your methods is the same decision you make every time
you pass a value to a function.

## Putting It All Together

Here is a complete example that uses everything from this chapter: struct
definition, field init shorthand, methods with all three receivers, an associated
function, and both `Debug` and `Display` formatting:

```rust
use std::fmt;

#[derive(Debug)]
struct Task {
    title: String,
    priority: u8,
    done: bool,
}

impl Task {
    fn new(title: String, priority: u8) -> Self {
        Self {
            title,
            priority,
            done: false,
        }
    }

    fn is_high_priority(&self) -> bool {
        self.priority >= 8
    }

    fn complete(&mut self) {
        self.done = true;
    }

    fn into_summary(self) -> String {
        let status = if self.done { "done" } else { "pending" };
        format!("[{}] {} (priority {})", status, self.title, self.priority)
    }
}

impl fmt::Display for Task {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        let marker = if self.done { "x" } else { " " };
        write!(f, "[{}] {}", marker, self.title)
    }
}

fn main() {
    let mut task = Task::new(String::from("Write chapter 3"), 9);
    println!("debug: {:?}", task);
    println!("display: {task}");
    println!("high priority: {}", task.is_high_priority());

    task.complete();
    println!("after completing: {task}");

    let summary = task.into_summary();
    println!("{summary}");
    // task is consumed — it lived a good life
}
```

Output:

```
debug: Task { title: "Write chapter 3", priority: 9, done: false }
display: [ ] Write chapter 3
high priority: true
after completing: [x] Write chapter 3
[done] Write chapter 3 (priority 9)
```

Every design choice here reflects what you have learned:

- `new` is a plain associated function that sets sensible defaults (`done:
  false`).
- `is_high_priority(&self)` only reads the data, so it takes a shared reference.
- `complete(&mut self)` modifies the struct, so it requires a mutable reference.
- `into_summary(self)` consumes the struct, signaling that the `Task` is used up
  after producing the summary. The `into_` prefix is a Rust convention for
  methods that take ownership.
- `Display` provides clean user-facing output (`[x] Write chapter 3`).
- `Debug` provides developer-facing output with all fields visible.

## Why This Matters

Structs are how you model your domain in Rust. They are not just containers for
data — combined with `impl` blocks, they become types with clear interfaces.
The method receiver (`&self`, `&mut self`, `self`) declares the contract between
the type and its callers: what access is needed, and whether ownership is
transferred.

This is what makes Rust APIs self-documenting. When you see a method that takes
`&self`, you know it will not modify or consume the value. When you see `self`,
you know the value is gone after the call. The compiler checks every call site
against these contracts. No documentation drift, no runtime surprises.

Key points from this chapter:

- **Structs** group related data into a named type. Fields are accessed with
  dot notation. The struct owns its fields.
- **Methods** are defined in `impl` blocks. The first parameter (`&self`,
  `&mut self`, or `self`) determines how the method accesses the struct.
- **Associated functions** (no `self` parameter) belong to the type, not an
  instance. The `new` convention is how Rust replaces constructors.
- **`#[derive(Debug)]`** generates developer-facing output for `{:?}`.
  Implement `Display` manually for user-facing output with `{}`.
- **Ownership applies to structs** just like every other value. Moving a struct
  moves its fields. Borrowing a struct borrows its fields.

### Exercise 3-1: Rectangle Toolkit

Build a `Rectangle` type that wraps its dimensions in a `Meters` tuple struct
for type safety. This practices struct definition, `new`, `&self` methods,
`Display`, tuple structs, and the method-parameter pattern from
`can_contain`.

Define:

- `Meters(f64)` — a tuple struct representing a length in meters. Tuple
  structs construct directly — `Meters(5.0)` — so no `new` function is
  needed. Access the inner value with `.0` (e.g., `self.width.0`).
- `Rectangle` — a regular struct with `width: Meters` and `height: Meters`.

Implement these methods on `Rectangle`:

- `new(width: f64, height: f64) -> Self` — associated function that wraps
  the raw `f64` values in `Meters` (e.g., `Meters(width)`).
- `area(&self) -> f64` — returns width × height (extract the `f64` values
  with `.0`).
- `perimeter(&self) -> f64` — returns 2 × (width + height).
- `is_square(&self) -> bool` — returns `true` if width and height are equal.
  Floating-point arithmetic can introduce tiny rounding errors, so comparing
  with `==` is unreliable. Instead, check that the absolute difference is
  smaller than a tiny threshold: `(a - b).abs() < f64::EPSILON`.
- `wider_than(&self, other: &Rectangle) -> bool` — compares widths.

Implement `Display` for `Rectangle` so it prints as `5m × 3m`.

In `main`, create two rectangles (one rectangular, one square) and print
their properties.

Expected output:

```
Room: 5m × 3m
  area: 15 sq m
  perimeter: 16 m
  is square: false
Square: 4m × 4m
  is square: true
Room wider than square: true
Square wider than room: false
```

<details>
<summary>Solution</summary>

```rust
use std::fmt;

struct Meters(f64);

struct Rectangle {
    width: Meters,
    height: Meters,
}

impl Rectangle {
    fn new(width: f64, height: f64) -> Self {
        Self {
            width: Meters(width),
            height: Meters(height),
        }
    }

    fn area(&self) -> f64 {
        self.width.0 * self.height.0
    }

    fn perimeter(&self) -> f64 {
        2.0 * (self.width.0 + self.height.0)
    }

    fn is_square(&self) -> bool {
        (self.width.0 - self.height.0).abs() < f64::EPSILON
    }

    fn wider_than(&self, other: &Rectangle) -> bool {
        self.width.0 > other.width.0
    }
}

impl fmt::Display for Rectangle {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "{}m × {}m", self.width.0, self.height.0)
    }
}

fn main() {
    let room = Rectangle::new(5.0, 3.0);
    let square = Rectangle::new(4.0, 4.0);

    println!("Room: {room}");
    println!("  area: {} sq m", room.area());
    println!("  perimeter: {} m", room.perimeter());
    println!("  is square: {}", room.is_square());

    println!("Square: {square}");
    println!("  is square: {}", square.is_square());

    println!("Room wider than square: {}", room.wider_than(&square));
    println!("Square wider than room: {}", square.wider_than(&room));
}
```

### Exercise 3-2: Student Gradebook

Build a `Student` struct that tracks a student's name and scores, practicing
`&mut self` mutation, `&self` queries, `Debug`, and `Display`. This builds on
the `Vec` and `push` patterns from earlier chapters.

Define a `Student` struct with:

- `name: String`
- `scores: Vec<u8>`

Derive `Debug` and implement these methods:

- `new(name: String) -> Self` — creates a student with an empty score list.
- `add_score(&mut self, score: u8)` — appends a score.
- `average(&self) -> f64` — returns the mean score. Return `0.0` if the
  score list is empty — dividing by zero produces `NaN`, which silently
  poisons any further arithmetic. Be careful summing `u8` values: three
  scores can already reach 255 (the `u8` maximum), so sum into a larger
  integer type first.
- `highest(&self) -> u8` — returns the highest score (return `0` if empty).
  Use `copied()` rather than `cloned()` when working with `Copy` types like
  `u8` — both work, but `copied()` signals that the type is trivially
  copyable.
- `passing(&self, threshold: f64) -> bool` — returns `true` if the average
  meets or exceeds the threshold. This is a simple `>=` comparison — do not
  use the epsilon pattern here. Epsilon is for testing whether two floats
  are *equal* despite rounding errors. "Is this value big enough?" is just
  `>=`.

Implement `Display` so it prints as `Alice: 85.0 avg, 3 scores [PASS]` (using
`60.0` as the pass threshold).

In `main`, create a student, add four scores (85, 92, 78, 45), and show the
student's progress.

Expected output:

```
New student: Student { name: "Alice", scores: [] }
Alice: 85.0 avg, 3 scores [PASS]
Highest score: 92
Alice: 75.0 avg, 4 scores [PASS]
Still passing (60.0): true
Passing at 80.0: false
```

<details>
<summary>Solution</summary>

```rust
use std::fmt;

#[derive(Debug)]
struct Student {
    name: String,
    scores: Vec<u8>,
}

impl Student {
    fn new(name: String) -> Self {
        Self {
            name,
            scores: Vec::new(),
        }
    }

    fn add_score(&mut self, score: u8) {
        self.scores.push(score);
    }

    fn average(&self) -> f64 {
        if self.scores.is_empty() {
            return 0.0;
        }
        let sum: u64 = self.scores.iter().copied().map(u64::from).sum();
        sum as f64 / self.scores.len() as f64
    }

    fn highest(&self) -> u8 {
        self.scores.iter().copied().max().unwrap_or(0)
    }

    fn passing(&self, threshold: f64) -> bool {
        self.average() >= threshold
    }
}

impl fmt::Display for Student {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        let status = if self.passing(60.0) { "PASS" } else { "FAIL" };
        write!(
            f,
            "{}: {:.1} avg, {} scores [{}]",
            self.name,
            self.average(),
            self.scores.len(),
            status,
        )
    }
}

fn main() {
    let mut student = Student::new(String::from("Alice"));
    println!("New student: {:?}", student);

    student.add_score(85);
    student.add_score(92);
    student.add_score(78);

    println!("{student}");
    println!("Highest score: {}", student.highest());

    student.add_score(45);
    println!("{student}");
    println!("Still passing (60.0): {}", student.passing(60.0));
    println!("Passing at 80.0: {}", student.passing(80.0));
}
```

</details>

### Exercise 3-3: Bank Account Ledger

Build a `BankAccount` that records transactions, practicing all three method
receivers (`&self`, `&mut self`, `self`), a free function that borrows the
struct, and consuming ownership with `close`. This extends the `Account`
example from the chapter into a more complete design.

Define:

- `Transaction` — a struct with `description: String` and `amount: f64`
  (positive for deposits, negative for withdrawals). Derive `Debug`.
- `BankAccount` — a struct with `holder: String`, `balance: f64`, and
  `transactions: Vec<Transaction>`. Derive `Debug`.

Implement these methods on `BankAccount`:

- `new(holder: String, opening_balance: f64) -> Self` — starts with an
  empty transaction list.
- `deposit(&mut self, amount: f64, description: &str)` — adds the amount
  and records a transaction. Keep this method focused on data — do not print
  inside it. Let the caller in `main` handle any output. Methods that mix
  data changes with printing are harder to reuse and test.
- `withdraw(&mut self, amount: f64, description: &str) -> bool` — subtracts
  the amount if sufficient funds exist, records a negative transaction, and
  returns `true`. Returns `false` without modifying anything if the balance
  is insufficient. Same rule: return the result, let the caller print.
- `transaction_count(&self) -> usize` — returns the number of transactions.
- `close(self) -> String` — consumes the account and returns a summary
  string.

Implement `Display` to show the holder and balance (e.g., `Alice: $365.50`).
Use `{:.2}` to format the balance to two decimal places — without it, `365.5`
prints instead of `365.50`.

Write a free function (not a method) `print_statement(account: &BankAccount)`
that borrows the account and prints each transaction with a `+`/`-` sign.
This is deliberately not a method — it practices passing a struct by reference
to code outside the `impl` block, showing that `&BankAccount` works the same
whether it appears as `&self` or as a regular parameter.

In `main`, create an account for "Alice" with $100, then:

1. Deposit $250.00 ("Paycheck") and $75.50 ("Freelance work").
2. Withdraw $60.00 ("Groceries") — should succeed.
3. Withdraw $1000.00 ("Vacation") — should fail (insufficient funds).
4. Print the transaction count, a full statement, and close the account.

Expected output:

```
Alice: $100.00
Deposit $250.00: Paycheck
Deposit $75.50: Freelance work
Withdraw $60.00: true
Withdraw $1000.00: false
Transactions: 3
--- Statement for Alice ---
  +250.00  Paycheck
  +75.50  Freelance work
  -60.00  Groceries
  Balance: $365.50
---
Account closed for Alice. Final balance: $365.50 (3 transactions)
```

<details>
<summary>Solution</summary>

```rust
use std::fmt;

#[derive(Debug)]
struct Transaction {
    description: String,
    amount: f64,
}

#[derive(Debug)]
struct BankAccount {
    holder: String,
    balance: f64,
    transactions: Vec<Transaction>,
}

impl BankAccount {
    fn new(holder: String, opening_balance: f64) -> Self {
        Self {
            holder,
            balance: opening_balance,
            transactions: Vec::new(),
        }
    }

    fn deposit(&mut self, amount: f64, description: &str) {
        self.balance += amount;
        self.transactions.push(Transaction {
            description: String::from(description),
            amount,
        });
    }

    fn withdraw(&mut self, amount: f64, description: &str) -> bool {
        if amount > self.balance {
            return false;
        }
        self.balance -= amount;
        self.transactions.push(Transaction {
            description: String::from(description),
            amount: -amount,
        });
        true
    }

    fn transaction_count(&self) -> usize {
        self.transactions.len()
    }

    fn close(self) -> String {
        format!(
            "Account closed for {}. Final balance: ${:.2} ({} transactions)",
            self.holder,
            self.balance,
            self.transactions.len(),
        )
    }
}

impl fmt::Display for BankAccount {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "{}: ${:.2}", self.holder, self.balance)
    }
}

fn print_statement(account: &BankAccount) {
    println!("--- Statement for {} ---", account.holder);
    for t in &account.transactions {
        let sign = if t.amount >= 0.0 { "+" } else { "" };
        println!("  {}{:.2}  {}", sign, t.amount, t.description);
    }
    println!("  Balance: ${:.2}", account.balance);
    println!("---");
}

fn main() {
    let mut account = BankAccount::new(String::from("Alice"), 100.0);
    println!("{account}");

    account.deposit(250.0, "Paycheck");
    println!("Deposit $250.00: Paycheck");
    account.deposit(75.50, "Freelance work");
    println!("Deposit $75.50: Freelance work");

    let success = account.withdraw(60.0, "Groceries");
    println!("Withdraw $60.00: {success}");

    let failed = account.withdraw(1000.0, "Vacation");
    println!("Withdraw $1000.00: {failed}");

    println!("Transactions: {}", account.transaction_count());

    print_statement(&account);

    let farewell = account.close();
    println!("{farewell}");
}
```

</details>

With structs, you can model single concepts — a user, a rectangle, a task. But
what about values that can be _one of several_ things? A shape could be a circle
_or_ a rectangle. A network response could be success _or_ failure. In the next
chapter, you will learn how Rust models these alternatives with _enums_ —
types that carry data and guarantee you handle every case.
