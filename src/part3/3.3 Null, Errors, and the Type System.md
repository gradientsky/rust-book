# Null, Errors, and the Type System

Most languages have a hidden trapdoor. In Java, any reference can be `null`. In
Python, any function can raise an exception. In JavaScript, `undefined` lurks
behind every property access. These trapdoors share a common problem: they are
_invisible_. The type system does not tell you a function might return nothing,
or that an operation might fail. You discover the trapdoor when your program
crashes at runtime.

Rust takes a different stance: _if a function can fail, its type signature must
say so_. If a value might not exist, the type tells you. If an operation might
produce an error, the return type carries that information. The compiler then
ensures you handle every case — no crashes from forgotten checks, no silent
swallowing of errors.

This chapter introduces the two types that make this possible: `Option<T>` for
values that might not exist, and `Result<T, E>` for operations that might fail.
Both are enums — the same kind you learned in the previous chapter. Combined
with pattern matching, they replace null pointers and exceptions with something
safer, more explicit, and just as ergonomic.

> **How to Read This Chapter**
>
> - Understand now: `Option` models absence, `Result` models failure, `?` keeps
>   the happy path readable, and well-chosen types can encode business rules.
> - Memorize: `Some`, `None`, `Ok`, `Err`, and `?`.
> - Use as reference: the main combinators — `map`, `and_then`, `or_else`,
>   `filter`, `flatten`, `ok_or`, and `ok`.
> - Skim on first pass: the longer combinator catalog. First learn the shape of
>   `Option` and `Result`, then return when you need a specific helper.

## The Billion-Dollar Mistake

Tony Hoare, the inventor of null references, famously called them his
"billion-dollar mistake." The problem is not the concept of absence — sometimes
a value genuinely does not exist. The problem is that null is _invisible_ to the
type system. A `String` in Java might be a string, or it might be `null`, and
the type does not distinguish between the two. The compiler cannot help you.

Rust's answer is `Option<T>` — an enum with exactly two variants:

```rust
// This is how Option is defined in the standard library.
// You do not write this yourself — it is always available.
enum Option<T> {
    Some(T),  // a value exists
    None,     // no value
}
```

The `<T>` is a _generic type parameter_ — a placeholder that lets `Option` work
with any type. An `Option<String>` is either `Some(String)` or `None`. An
`Option<i32>` is either `Some(i32)` or `None`. Read `Option<T>` as "an optional
value of type T."

`Option` is so fundamental that Rust includes it in every program automatically.
You do not need to import it, and you can use `Some` and `None` directly without
the `Option::` prefix.

## Using Option

Here is a function that searches a list and might not find what you are looking
for:

Example 3-1. Returning Option when a search might not find a value

```rust
fn find_first_negative(numbers: &[i32]) -> Option<i32> {
    for &n in numbers {
        if n < 0 {
            return Some(n);
        }
    }
    None
}

fn main() {
    let values = [3, 7, -2, 5, -8];

    match find_first_negative(&values) {
        Some(n) => println!("found negative: {n}"),
        None => println!("all positive"),
    }

    match find_first_negative(&[1, 2, 3]) {
        Some(n) => println!("found negative: {n}"),
        None => println!("all positive"),
    }
}
```

Output:

```
found negative: -2
all positive
```

The return type `Option<i32>` tells the caller: "this function might not return
a number." The caller must handle both possibilities — `Some` and `None` — or
the code will not compile. There is no way to accidentally use the result as if
it were always a number.

Compare this to a language with null: the function would return an `int` or
`null`, and nothing in the type signature would warn you. In Rust, the type
_is_ the warning.

### When There Is No Value

`None` is not null. It is not a pointer to nothing. It is a regular enum variant
that carries no data. You cannot accidentally dereference it, because `Option<T>`
and `T` are different types:

```rust,does_not_compile
fn main() {
    let maybe_number: Option<i32> = None;
    let result = maybe_number + 1; // error: cannot add Option<i32> and i32
}
```

```
error[E0369]: cannot add `{integer}` to `Option<i32>`
 --> src/main.rs:3:31
  |
3 |     let result = maybe_number + 1;
  |                  ------------ ^ - {integer}
  |                  |
  |                  Option<i32>
  |
note: `Option<i32>` does not implement `Add<{integer}>`
```

To use the value inside an `Option`, you must first check whether it exists.
Rust gives you several ways to do this, and the right choice depends on the
situation.

### Extracting Values with Match and If Let

You have already seen `match` on `Option`:

```rust
fn greet(name: Option<&str>) {
    match name {
        Some(n) => println!("hello, {n}!"),
        None => println!("hello, stranger!"),
    }
}

fn main() {
    greet(Some("Alice"));
    greet(None);
}
```

Output:

```
hello, Alice!
hello, stranger!
```

When you only care about the `Some` case, `if let` is more concise:

```rust
fn main() {
    let config_port: Option<u16> = Some(8080);

    if let Some(port) = config_port {
        println!("using configured port: {port}");
    } else {
        println!("using default port: 3000");
    }
}
```

Output: `using configured port: 8080`

### Providing a Default Value

When you want the inner value or a fallback, `unwrap_or` is the simplest
approach:

```rust
fn main() {
    let user_color: Option<&str> = None;
    let color = user_color.unwrap_or("blue");
    println!("color: {color}");

    let user_color: Option<&str> = Some("green");
    let color = user_color.unwrap_or("blue");
    println!("color: {color}");
}
```

Output:

```
color: blue
color: green
```

If computing the default is expensive, use `unwrap_or_else` with a closure — the
closure only runs when the value is `None`:

```rust
fn default_name() -> String {
    println!("computing default...");
    String::from("anonymous")
}

fn main() {
    let name: Option<String> = Some(String::from("Alice"));
    let result = name.unwrap_or_else(default_name);
    println!("name: {result}");
}
```

Output: `name: Alice`

Notice that `"computing default..."` was never printed — the closure was not
called because the `Option` was `Some`.

### Transforming with Map

The next several sections introduce **combinators** — methods on `Option` (and
later `Result`) that transform, chain, or inspect the wrapped value without
explicitly matching on it. Methods like `map`, `and_then`, `filter`, and
`unwrap_or` are all combinators. The name comes from the idea that they
_combine_ — you chain them together to build complex transformations from small,
composable pieces.

`map` applies a function to the inner value if it exists, and passes `None`
through unchanged:

```rust
fn main() {
    let maybe_name: Option<&str> = Some("alice");
    let shouted = maybe_name.map(|name| name.to_uppercase());
    println!("{shouted:?}");

    let nothing: Option<&str> = None;
    let shouted = nothing.map(|name| name.to_uppercase());
    println!("{shouted:?}");
}
```

Output:

```
Some("ALICE")
None
```

`map` lets you transform the value inside an `Option` without unpacking it first.
The result is still an `Option` — `Some` with the transformed value, or `None`
if there was nothing to transform.

### Chaining with And Then

`map` works when the transformation always succeeds — you have a value, you
transform it, done. But what happens when the transformation itself might fail?

```rust
fn parse_port(input: &str) -> Option<u16> {
    let trimmed = input.trim();
    if trimmed.is_empty() {
        return None;
    }
    trimmed.parse().ok()
}

fn main() {
    let input: Option<&str> = Some("8080");

    // map: the closure returns Option<u16>, so map wraps it in another Option
    let nested: Option<Option<u16>> = input.map(|s| parse_port(s));
    println!("map:      {nested:?}");

    // and_then: flattens automatically — one layer of Option
    let flat: Option<u16> = input.and_then(|s| parse_port(s));
    println!("and_then: {flat:?}");
}
```

Output:

```
map:      Some(Some(8080))
and_then: Some(8080)
```

When the closure returns `Option`, `map` wraps that in _another_ `Option` —
giving you `Some(Some(8080))`. You could fix this by chaining `.flatten()`
after `.map(...)`, but `and_then` does both in one step: it maps the closure
and flattens the result. Think of `and_then` as "`map` for fallible
transformations."

```rust
fn parse_port(input: &str) -> Option<u16> {
    let trimmed = input.trim();
    if trimmed.is_empty() {
        return None;
    }
    trimmed.parse().ok()
}

fn main() {
    let inputs = ["8080", "", "abc", " 443 "];
    for input in inputs {
        let result = Some(input).and_then(parse_port);
        println!("'{input}' -> {result:?}");
    }
}
```

Output:

```
'8080' -> Some(8080)
'' -> None
'abc' -> None
' 443 ' -> Some(443)
```

`and_then` is the key to chaining multiple operations that each might fail. If
any step returns `None`, the entire chain short-circuits to `None`.

### Trying an Alternative with Or

`and_then` chains operations that continue when a value is present — "if this
succeeds, try the next step." But sometimes you want the opposite: "if this is
`None`, try a different source." That is what `or` does:

```rust
fn main() {
    // Try the user's preference, fall back to the system default
    let user_choice: Option<&str> = None;
    let system_default: Option<&str> = Some("en");

    let language = user_choice.or(system_default);
    println!("language: {language:?}");

    // When the first source has a value, the fallback is ignored
    let user_choice: Option<&str> = Some("ja");
    let language = user_choice.or(system_default);
    println!("language: {language:?}");

    // Chain multiple fallbacks for a layered lookup
    let env_port: Option<u16> = None;
    let config_port: Option<u16> = None;
    let default_port: Option<u16> = Some(8080);

    let port = env_port.or(config_port).or(default_port);
    println!("port: {port:?}");
}
```

Output:

```
language: Some("en")
language: Some("ja")
port: Some(8080)
```

`or` returns the first `Some` it finds. If `self` is `Some`, the alternative is
never evaluated. If `self` is `None`, the alternative is returned as-is. Chaining
multiple `.or()` calls creates a layered fallback — the first source that has a
value wins.

When computing the alternative is expensive, use `or_else` with a closure — the
closure runs only when the original value is `None`, just as `unwrap_or_else`
runs its closure only when there is no value to unwrap:

```rust
fn look_up_in_database(_key: &str) -> Option<String> {
    // imagine a real database query here
    Some("found-value".to_string())
}

fn main() {
    let cached: Option<String> = None;

    // The database query runs only if the cache misses
    let value = cached.or_else(|| look_up_in_database("config.timeout"));
    println!("{value:?}");
}
```

Output: `Some("found-value")`

Together, `and_then` and `or` form a natural pair: `and_then` continues a chain
of successes, while `or` provides alternatives for failures.

### Keeping or Discarding with Filter

`map` transforms a value. `and_then` chains a fallible step. But sometimes you
do not want to transform the value at all — you want to _keep it or discard it_
based on a condition. That is what `filter` does:

```rust
fn main() {
    let port: Option<u16> = Some(8080);

    // Keep the port only if it is in the non-privileged range
    let valid = port.filter(|&p| p >= 1024);
    println!("8080 valid: {valid:?}");

    let low: Option<u16> = Some(80);
    let rejected = low.filter(|&p| p >= 1024);
    println!("80 valid:   {rejected:?}");

    let absent: Option<u16> = None;
    let also_none = absent.filter(|&p| p >= 1024);
    println!("None valid: {also_none:?}");
}
```

Output:

```
8080 valid: Some(8080)
80 valid:   None
None valid: None
```

When the `Option` is `Some`, `filter` runs the predicate. If the predicate
returns `true`, the value passes through unchanged. If it returns `false`, the
`Option` becomes `None`. When the `Option` is already `None`, `filter` does
nothing — there is no value to test.

Think of `filter` as a gate: values that satisfy the condition pass through;
everything else is turned away. This is the same concept you will see with
iterators, where `filter` keeps only the elements that match a predicate.

### Flattening Nested Options

Sometimes you end up with an `Option<Option<T>>` — two layers of optionality
where you only need one. This happens naturally when a collection's values are
themselves optional:

> **Just Enough: HashMap**
>
> The next example uses `HashMap<K, V>`, a key-value store from the standard
> library. Think of it as an array indexed by any type — strings, numbers,
> anything hashable — instead of only by position. `HashMap::new()` creates an
> empty map, `.insert(key, value)` adds an entry, and `.get(&key)` returns
> `Option<&V>` (the value if the key exists, or `None`). We cover `HashMap` in
> depth in a later chapter; for now this is all you need.

```rust
use std::collections::HashMap;

fn main() {
    // A config where values are themselves optional
    let mut overrides: HashMap<&str, Option<u16>> = HashMap::new();
    overrides.insert("port", Some(9090));
    overrides.insert("timeout", None); // explicitly set to "no value"

    // Looking up a key returns Option<&Option<u16>> — two layers of Option
    let port: Option<&Option<u16>> = overrides.get("port");
    println!("raw lookup: {port:?}");

    // flatten collapses Option<Option<T>> into Option<T>
    let port: Option<u16> = overrides.get("port").copied().flatten();
    println!("flattened:  {port:?}");

    // A key explicitly set to None flattens to None
    let timeout: Option<u16> = overrides.get("timeout").copied().flatten();
    println!("timeout:    {timeout:?}");

    // A missing key also flattens to None
    let missing: Option<u16> = overrides.get("workers").copied().flatten();
    println!("missing:    {missing:?}");
}
```

Output:

```
raw lookup: Some(Some(9090))
flattened:  Some(9090)
timeout:    None
missing:    None
```

`flatten` removes exactly one layer of nesting. Both `None` (key not found) and
`Some(None)` (key found but value absent) collapse to `None`, while
`Some(Some(v))` becomes `Some(v)`.

There is a useful relationship between `flatten` and `and_then`: calling
`and_then(f)` is equivalent to calling `map(f)` followed by `flatten`. When you
find yourself chaining a transformation that returns `Option` and then
flattening the result, reach for `and_then` directly — it does both steps in
one call.

### Checking with Is Some And

Sometimes you need a boolean answer: does this `Option` contain a value that
meets a condition?

```rust
fn main() {
    let threshold: Option<u32> = Some(75);

    if threshold.is_some_and(|t| t >= 50) {
        println!("threshold is high enough");
    }

    let missing: Option<u32> = None;
    println!("missing passes check: {}", missing.is_some_and(|t| t >= 50));
}
```

Output:

```
threshold is high enough
missing passes check: false
```

`is_some_and` returns `true` only if the `Option` is `Some` _and_ the predicate
holds. Notice that you never unwrap the `Option` — you get a `bool` directly,
without extracting the inner value. This is the right tool whenever you need to
_check_ an optional value rather than _use_ it.

Its counterpart, `is_none_or`, returns `true` if the `Option` is `None`
_or_ the predicate holds — useful for "default-allow" validation:

```rust
fn main() {
    let max_retries: Option<u32> = None;

    // None means "no limit" — always acceptable
    if max_retries.is_none_or(|n| n <= 10) {
        println!("retry limit is acceptable");
    }

    let max_retries: Option<u32> = Some(100);
    if max_retries.is_none_or(|n| n <= 10) {
        println!("this won't print");
    } else {
        println!("retry limit too high");
    }
}
```

Output:

```
retry limit is acceptable
retry limit too high
```

### Try It: Option Combinators

These short exercises practice the `Option` methods from this section. Each one
can be solved in a single `main` function.

#### Exercise 3-E1: Theme Picker

A desktop app resolves which UI theme to use. Two sources exist:
`load_user_theme` returns `Some("dark".to_string())` and `load_system_theme`
returns `Some("high-contrast".to_string())`.

Write a `resolve_theme(user_pref: Option<String>) -> String` function that
applies these rules, in order:

1. Use the user's preference if one is provided.
2. Otherwise, fall back to the system theme.
3. Reject `"high-contrast"` — if it is the only theme found, treat it as if no
   theme exists.
4. Prefix the surviving theme name with `"theme:"`.
5. If no usable theme remains after the previous steps, return
   `"theme:default"`.

Solve this with a single combinator chain — no `match` or `if let` needed.
You will need `or_else`, `filter`, `map`, and `unwrap_or`.

In `main`, call `resolve_theme` twice: once with `None` (no user preference)
and once with the result of `load_user_theme`.

Expected output:

```
theme:default
theme:dark
```

<details>
<summary>Solution</summary>

```rust
fn load_user_theme() -> Option<String> {
    Some("dark".to_string())
}

fn load_system_theme() -> Option<String> {
    Some("high-contrast".to_string())
}

fn resolve_theme(user_pref: Option<String>) -> String {
    user_pref
        .or_else(|| load_system_theme())
        .filter(|t| t != "high-contrast")
        .map(|t| format!("theme:{t}"))
        .unwrap_or(String::from("theme:default"))
}

fn main() {
    println!("{}", resolve_theme(None));
    println!("{}", resolve_theme(load_user_theme()));
}
```

Output:

```
theme:default
theme:dark
```

</details>

#### Exercise 3-E2: Safe Division Pipeline

Build a small calculator that chains multiple fallible steps with `and_then`.

Write three functions:

- `parse_number(input: &str) -> Option<f64>` — trims whitespace and parses the
  string as `f64`, returning `None` on failure.
- `safe_divide(a: f64, b: f64) -> Option<f64>` — returns `None` when the
  divisor is `0.0`.
- `divide_strings(a: &str, b: &str) -> Option<f64>` — parses both strings and
  divides the results. Each step can fail, so use `and_then` to chain them:
  a single `None` short-circuits the whole pipeline.

In `main`, call `divide_strings` four times to exercise the pipeline:

1. `"10"` / `"4"` — both parse, divisor is nonzero. Print the result with
   `unwrap_or(0.0)`.
2. `"10"` / `"0"` — divisor is zero. Print with the same fallback.
3. `"abc"` / `"4"` — first parse fails. Print with the same fallback.
4. `"100"` / `"4"` — check whether the result is greater than `20.0` and print
   the boolean directly, without extracting the inner value. (Hint:
   `is_some_and`.)

Expected output:

```
10 / 4 = 2.5
10 / 0 = 0
abc / 4 = 0
100 / 4 > 20: true
```

<details>
<summary>Solution</summary>

```rust
fn parse_number(input: &str) -> Option<f64> {
    input.trim().parse().ok()
}

fn safe_divide(a: f64, b: f64) -> Option<f64> {
    if b == 0.0 {
        None
    } else {
        Some(a / b)
    }
}

fn divide_strings(a: &str, b: &str) -> Option<f64> {
    parse_number(a).and_then(|x| {
        parse_number(b).and_then(|y| safe_divide(x, y))
    })
}

fn main() {
    println!("10 / 4 = {}", divide_strings("10", "4").unwrap_or(0.0));
    println!("10 / 0 = {}", divide_strings("10", "0").unwrap_or(0.0));
    println!("abc / 4 = {}", divide_strings("abc", "4").unwrap_or(0.0));
    println!(
        "100 / 4 > 20: {}",
        divide_strings("100", "4").is_some_and(|v| v > 20.0)
    );
}
```

Output:

```
10 / 4 = 2.5
10 / 0 = 0
abc / 4 = 0
100 / 4 > 20: true
```

</details>

Validate each setting as a boolean — without unwrapping — and print the result:

## When Operations Fail: Result

`Option` handles absence — a value that might not exist. But what about
operations that can go _wrong_? Reading a file might fail because the file does
not exist. Parsing a number might fail because the input is not a number. A
network request might fail because the server is down.

In many languages, these failures are communicated through _exceptions_ — a
completely separate control flow path that is invisible in the function's type
signature. A function that says it returns a `User` might actually throw a
`DatabaseException`, a `TimeoutException`, or an `IOException`. The caller has
no idea unless they read the documentation (if it exists) or learn the hard way
at runtime.

Rust's answer is `Result<T, E>` — an enum with two variants:

```rust
// This is how Result is defined in the standard library.
// You do not write this yourself — it is always available.
enum Result<T, E> {
    Ok(T),   // operation succeeded with value T
    Err(E),  // operation failed with error E
}
```

`T` is the success type. `E` is the error type. A function that returns
`Result<User, DatabaseError>` is telling you _in its type signature_ that it
might fail, and exactly what kind of failure to expect. The compiler will not
let you ignore it.

Like `Option`, `Result` is included in every Rust program automatically. You can
use `Ok` and `Err` without importing anything.

### Your First Result

A function that can fail returns `Result` with an explicit error type:

Example 3-2. Returning Result for an operation that can fail

```rust
fn divide(a: f64, b: f64) -> Result<f64, String> {
    if b == 0.0 {
        Err(String::from("division by zero"))
    } else {
        Ok(a / b)
    }
}

fn main() {
    match divide(10.0, 3.0) {
        Ok(result) => println!("10 / 3 = {result:.2}"),
        Err(e) => println!("error: {e}"),
    }

    match divide(10.0, 0.0) {
        Ok(result) => println!("10 / 0 = {result:.2}"),
        Err(e) => println!("error: {e}"),
    }
}
```

Output:

```
10 / 3 = 3.33
error: division by zero
```

The caller must handle both `Ok` and `Err`. If you forget, the compiler will
tell you. This is the same exhaustive matching you learned with enums — applied
to error handling.

### Parsing and Other Standard Library Results

Many standard library functions return `Result`. Parsing a string into a number
is one of the most common:

```rust
fn main() {
    let good: Result<i32, _> = "42".parse();
    let bad: Result<i32, _> = "not_a_number".parse();

    match good {
        Ok(value) => println!("good: Ok({value})"),
        Err(error) => println!("good: Err({error})"),
    }

    match bad {
        Ok(value) => println!("bad: Ok({value})"),
        Err(error) => println!("bad: Err({error})"),
    }
}
```

Output:

```
good: Ok(42)
bad: Err(invalid digit found in string)
```

The `_` in `Result<i32, _>` lets the compiler infer the error type — you do not
need to write it out. The `parse` method returns `Result<T, T::Err>`, where the
error type depends on what you are parsing into.

### Providing Defaults and Transforming Results

`Result` has methods that mirror `Option`:

```rust
fn main() {
    let port: Result<u16, _> = "8080".parse();
    let port_number = port.unwrap_or(3000);
    println!("port: {port_number}");

    let bad_port: Result<u16, _> = "abc".parse();
    let port_number = bad_port.unwrap_or(3000);
    println!("fallback port: {port_number}");
}
```

Output:

```

fallback port: 3000
```

Use `map` to transform the success value and `map_err` to transform the error:

```rust
fn main() {
    let input = "  42  ";
    let result: Result<i32, _> = input.trim().parse();

    println!("doubled: {doubled:?}");

        .map_err(|e| format!("failed to parse input: {e}"));

```

### Flattening Nested Results

Just as `Option` can nest inside itself, so can `Result`. When you `map` over a
`Result` with a function that itself returns a `Result`, you get two layers —
`Result<Result<T, E>, E>` — where only one is needed:

```rust
fn validate_port(n: u16) -> Result<u16, String> {
    if n == 0 {
        Err(String::from("port cannot be zero"))
    } else {
        Ok(n)
    }
}

fn main() {
    let inputs = ["8080", "abc", "0"];

    for input in inputs {
        // map with a fallible function produces nested Results
        let nested = input.parse::<u16>()
            .map_err(|e| format!("parse error: {e}"))
            .map(validate_port);

        // flatten collapses the nesting into a single Result
        let flat = input.parse::<u16>()
            .map_err(|e| format!("parse error: {e}"))
            .map(validate_port)
            .flatten();

        println!("{input:>4} -> nested: {nested:?}");
        println!("       flat:   {flat:?}");
    }
}
```

Output:

```
8080 -> nested: Ok(Ok(8080))
       flat:   Ok(8080)
 abc -> nested: Err("parse error: invalid digit found in string")
       flat:   Err("parse error: invalid digit found in string")
   0 -> nested: Ok(Err("port cannot be zero"))
       flat:   Err("port cannot be zero")
```

`flatten` removes exactly one layer of nesting. An outer `Err` stays an `Err`.
An `Ok(Err(e))` — where the inner step failed — becomes `Err(e)`. Only
`Ok(Ok(v))` becomes `Ok(v)`.

The same relationship from `Option` applies here: `and_then(f)` is equivalent to
`map(f).flatten()`. When you find yourself mapping with a function that returns
`Result` and then flattening, reach for `and_then` directly — it does both steps
in one call.

### Converting Between Option and Result

`Option` and `Result` are closely related. You can convert between them:

```rust
fn main() {
    // Option -> Result: ok_or provides the error for None
    let maybe: Option<i32> = Some(42);
    let result: Result<i32, &str> = maybe.ok_or("value was missing");
    println!("{result:?}");

    let nothing: Option<i32> = None;
    let result: Result<i32, &str> = nothing.ok_or("value was missing");
    println!("{result:?}");

    // Result -> Option: ok() discards the error
    let parsed: Result<i32, _> = "99".parse();
    let as_option: Option<i32> = parsed.ok();
    println!("{as_option:?}");
}
```

Output:

```
Ok(42)
Err("value was missing")
Some(99)
```

The `.ok()` method on `Result` converts `Ok(v)` to `Some(v)` and `Err(_)` to
`None` — discarding the error information. Use it when you care about whether
an operation succeeded but not _why_ it failed.

## The Question Mark Operator

Pattern matching on `Result` is explicit and safe, but when a function performs
several fallible operations in sequence, the nesting becomes painful:

Example 3-3. Propagating parse failures with explicit nested matches

```rust
use std::num::ParseIntError;

fn add_strings(a: &str, b: &str) -> Result<i32, ParseIntError> {
    match a.parse::<i32>() {
        Ok(x) => match b.parse::<i32>() {
            Ok(y) => Ok(x + y),
            Err(e) => Err(e),
        },
        Err(e) => Err(e),
    }
}

fn report(result: Result<i32, ParseIntError>) {
    match result {
        Ok(sum) => println!("Ok({sum})"),
        Err(error) => println!("Err({error})"),
    }
}

fn main() {
    report(add_strings("10", "20"));
    report(add_strings("10", "abc"));
}
```

Output:

```
Ok(30)
Err(invalid digit found in string)
```

The function works, but the nested `match` obscures the logic. Most of the code
is error propagation boilerplate — "if this fails, return the error." Rust
provides a single character that does exactly this: `?`.

Example 3-4. Replacing nested matches with the question mark operator

```rust
use std::num::ParseIntError;

fn add_strings(a: &str, b: &str) -> Result<i32, ParseIntError> {
    let x = a.parse::<i32>()?;
    let y = b.parse::<i32>()?;
    Ok(x + y)
}

fn report(result: Result<i32, ParseIntError>) {
    match result {
        Ok(sum) => println!("Ok({sum})"),
        Err(error) => println!("Err({error})"),
    }
}

fn main() {
    report(add_strings("10", "20"));
    report(add_strings("10", "abc"));
}
```

Output:

```
Ok(30)
Err(invalid digit found in string)
```

The `?` operator does three things:

1. If the `Result` is `Ok(v)`, it extracts `v` and execution continues.
2. If the `Result` is `Err(e)`, it returns `Err(e)` from the enclosing function
   immediately.
3. If the operation's error type differs from the function's return error type,
   `?` converts it automatically — as long as Rust knows how to create the
   target type from the source.

The function now reads like a straight-line sequence of operations: parse `a`,
parse `b`, add them, return the sum. The error handling is still there — it is
just compressed into `?`.

> **Sidebar**
>
> Read `?` as: "give me the success value, or return this failure now." That
> mental sentence is usually more useful than memorizing the desugared `match`.

> **Smell: the passthrough match arm**
>
> If you find yourself writing
>
> ```rust,ignore
> match result {
>     Ok(0) => Err(String::from("...")),
>     other => other,
> }
> ```
>
> the `other => other` arm is the giveaway. You are matching on a `Result`
> only to hand most of it back unchanged — that is `?`'s job. The shape
> below is shorter, reads top-to-bottom, and short-circuits on error
> before the second check runs:
>
> ```rust,ignore
> let n = result?;
> if n == 0 { return Err(String::from("...")); }
> Ok(n)
> ```
>
> Whenever a `match` arm just re-emits its input, reach for `?`,
> `and_then`, or `map_err` instead.

### Automatic Error Conversion

That third point is subtle but powerful. Consider a function that validates
input and returns `Result<u32, &str>`, while its caller returns
`Result<String, String>`:

```rust
fn validate(input: &str) -> Result<u32, &str> {
    if input.is_empty() {
        return Err("empty input");
    }
    input.parse::<u32>().map_err(|_| "not a number")
}

fn process(input: &str) -> Result<String, String> {
    let n = validate(input)?; // &str error → String error, automatically
    Ok(format!("processed: {n}"))
}

fn main() {
    println!("{:?}", process("42"));
    println!("{:?}", process(""));
    println!("{:?}", process("abc"));
}
```

Output:

```
Ok("processed: 42")
Err("empty input")
Err("not a number")
```

The `validate` function fails with a `&str`, but `process` returns
`Result<String, String>`. When `validate` fails, `?` converts the `&str` error
into a `String` automatically. Rust knows how to create a `String` from a
`&str`, so the conversion happens without any extra code from you.

Many standard library types register these conversions. When you define custom
error types, you register your own so that `?` works with them seamlessly. The
important takeaway is that `?` handles the plumbing — you write the logic, and
Rust bridges the types.

### Chaining Fallible Operations

`?` shines when you chain multiple operations that can each fail:

```rust
fn first_line_length(text: &str) -> Result<usize, String> {
    let first = text.lines().next().ok_or(String::from("empty text"))?;
    let trimmed = first.trim();
    if trimmed.is_empty() {
        return Err(String::from("first line is blank"));
    }
    Ok(trimmed.len())
}

fn main() {
    println!("{:?}", first_line_length("hello world\nsecond line"));
    println!("{:?}", first_line_length(""));
    println!("{:?}", first_line_length("   \ncontent"));
}
```

Output:

```
Ok(11)
Err("empty text")
Err("first line is blank")
```

Each `?` is an early exit point. If `text.lines().next()` returns `None`, the
`.ok_or(...)` converts it to an `Err`, and `?` returns that error immediately.
The remaining logic only runs if every step succeeds.

### The Question Mark with Option

`?` also works with `Option` inside functions that return `Option`:

```rust
fn parse_and_double(input: &str) -> Option<i32> {
    let number: i32 = input.trim().parse().ok()?;
    Some(number * 2)
}

fn main() {
    println!("{:?}", parse_and_double("21"));
    println!("{:?}", parse_and_double("abc"));
    println!("{:?}", parse_and_double(""));
}
```

Output:

```

None
None
```

When used inside a function that returns `Option`, `?` extracts the value from
`Some` or returns `None` from the function immediately. The same operator, the
same pattern — applied to absence instead of errors.

### When to Use Unwrap and Expect

You may have noticed that `Option` and `Result` both have an `unwrap` method
that extracts the value or panics:

```rust
fn main() {
    let value: Option<i32> = Some(42);
    println!("{}", value.unwrap()); // 42

    // This would panic:
    // let nothing: Option<i32> = None;
    // nothing.unwrap(); // thread 'main' panicked at 'called `Option::unwrap()` on a `None` value'
}
```

Output: `42`

`unwrap` should be used sparingly. It is appropriate in two situations:

- **Tests and prototyping:** When you are exploring an idea and want to focus on
  the happy path. Panicking is fine in test code.
- **When you have proof the value exists:** If the logic guarantees a `Some` or
  `Ok`, and you want to document _why_, use `expect` with a message:

The important trap is not the method itself, but what it communicates about your
program's contract.

> **Warning**
>
> `unwrap` turns a recoverable condition into a crash. If missing data or failure
> is part of normal program behavior, use `match`, a combinator, or `?` instead.

```rust
fn main() {
    let line = "host=localhost";
    let (_, host) = line
        .split_once('=')
        .expect("hard-coded config line must contain '='");
    println!("host: {host}");
}
```

Output: `host: localhost`

The `expect` message becomes part of the panic output, explaining the invariant
that was violated. In production code, prefer `?` to propagate errors to the
caller. Reserve `unwrap` and `expect` for cases where failure means the program
cannot continue in any meaningful way.

### When `?` Won't Compile

Two compile errors come up often when learning `?`. Knowing the shape of
each makes them quick to fix.

**1. Mismatched error types.** `?` propagates the error out of the
enclosing function, so the error type the expression produces must be
convertible to the function's declared error type. If they differ and no
conversion is registered, you will see something like:

```text
the trait `From<ParseIntError>` is not implemented for `String`
```

The fix is to align the types yourself with `map_err`:

```rust,ignore
let n: i32 = input.parse()
    .map_err(|e| e.to_string())?; // ParseIntError -> String
```

**2. `?` in a function that does not return `Result` or `Option`.** `?`
only works inside a function whose return type can carry the failure.
Inside a function returning `()` (or any non-`Result`/`Option` type),
you will see:

```text
the `?` operator can only be used in a function that returns `Result`
or `Option`
```

The fix is to change the return type to `Result<_, _>` (or `Option<_>`),
or handle the failure explicitly with `match` or `unwrap_or`. Note that
mixing is also an error: a function returning `Result` cannot use `?` on
an `Option` directly — convert with `.ok_or(err)` first. Same in the
other direction: use `.ok()` to drop the error.

**A subtle one: matching the wrong layer after `?`.** When `?` strips
one layer of the type, the value that remains is the _inner_ type. If
your function returns `Result<Option<T>, E>` and you write `expr?`, the
result is `Option<T>` — match on `Some`/`None`, not on `Ok`/`Err`. The
compiler error here will complain that `Ok` and `Err` are not
constructors of `Option`.

|---|---|---|
| Transform the success value | `map` | `map` |
| Transform the error value | — | `map_err` |
| Chain another fallible step | `and_then` | `and_then` |
| Provide a fallback (eager) | `or` | `or` |
| Provide a fallback (lazy) | `or_else` | `or_else` |
| Default value on failure (eager) | `unwrap_or` | `unwrap_or` |
| Default value on failure (lazy) | `unwrap_or_else` | `unwrap_or_else` |
| Discard if predicate fails | `filter` | — |
| Collapse nested type | `flatten` | `flatten` |
| Boolean check with predicate | `is_some_and` / `is_none_or` | `is_ok_and` / `is_err_and` |
| Convert between types | `ok_or` (→ `Result`) | `.ok()` (→ `Option`) |
| Propagate failure early | `?` | `?` |

Two patterns to keep in mind:

- **Eager vs. lazy.** Methods ending in `_else` take a closure that runs
  only when needed. Use the eager form (`or`, `unwrap_or`) when the
  alternative is a cheap value already in hand; use the lazy form
  (`or_else`, `unwrap_or_else`) when computing the alternative is
  expensive or has side effects.
- **`map` + `flatten` = `and_then`.** Whenever you find yourself mapping
  with a function that itself returns `Option` or `Result`, then
  flattening, reach for `and_then` directly. Same result, one call
  instead of two.

## Making Illegal States Unrepresentable

`Option` and `Result` are specific applications of a broader principle: _use the
type system so that invalid states cannot compile_. Instead of checking for bad
data at runtime, design your types so that bad data cannot exist.

### Replacing Boolean Flags with Enums

Consider a connection that can be either connected or disconnected. A common
approach uses a boolean flag and an optional field:

```rust,ignore
// Fragile: nothing prevents is_connected = false with Some(stream_id)
struct Connection {
    host: String,
    is_connected: bool,
    stream_id: Option<u64>,
}
```

This design allows impossible combinations: `is_connected` is `true` but
`stream_id` is `None`, or `is_connected` is `false` but `stream_id` is
`Some(42)`. You would need runtime checks to catch these inconsistencies.

An enum makes the illegal states unrepresentable:

```rust
enum ConnectionState {
    Disconnected,
    Connected { stream_id: u64 },
}

struct Connection {
    host: String,
    state: ConnectionState,
}

fn describe(conn: &Connection) -> String {
    match &conn.state {
        ConnectionState::Disconnected => {
            format!("{}: disconnected", conn.host)
        }
        ConnectionState::Connected { stream_id } => {
            format!("{}: connected (stream {stream_id})", conn.host)
        }
    }
}

fn main() {
    let active = Connection {
        host: String::from("example.com"),
        state: ConnectionState::Connected { stream_id: 42 },
    };
    println!("{}", describe(&active));

    let idle = Connection {
        host: String::from("backup.example.com"),
        state: ConnectionState::Disconnected,
    };
    println!("{}", describe(&idle));
}
```

Output:

```
example.com: connected (stream 42)
backup.example.com: disconnected
```

Now a connected connection _always_ has a stream ID, and a disconnected
connection _never_ does. The type system enforces the invariant. No runtime
checks needed.

### Preventing Mixups with Newtype Wrappers

When two values have the same underlying type but different meanings, a
_newtype_ wrapper prevents accidental mixups:

```rust
struct UserId(u64);
struct OrderId(u64);

fn process_order(user: &UserId, order: &OrderId) {
    println!("processing order {} for user {}", order.0, user.0);
}

fn main() {
    let user = UserId(1001);
    let order = OrderId(5042);

    process_order(&user, &order);
    // process_order(&order, &user); // compile error: expected &UserId, found &OrderId
}
```

Output: `processing order 5042 for user 1001`

Both `UserId` and `OrderId` are `u64` underneath, but the compiler treats them
as completely different types. You cannot accidentally pass an order ID where a
user ID is expected. This zero-cost abstraction — the newtype adds no runtime
overhead — prevents an entire category of bugs.

> **When newtypes don't earn their keep**
>
> Wrap a type when it does one of two jobs: (a) carries an _invariant_
> enforced by a private constructor (next section), or (b) prevents callers
> from mixing up arguments that share an underlying type (`UserId` vs.
> `OrderId`). If neither applies, the wrapper is decoration: it costs you
> `.0` field access and visual weight without buying safety. A `Port(u16)`
> wrapper that anyone can construct as `Port(0)` enforces nothing — the
> validation lives elsewhere, and the type is along for the ride. Reach
> for newtypes when they do work, not when they only look like they do.

### Validated Construction

Take the idea further: use a type whose constructor _guarantees_ its value is
valid:

```rust
#[derive(Debug)]
struct Percentage {
    value: u8,
}

#[derive(Debug)]
enum PercentageError {
    OutOfRange(u8),
}

impl Percentage {
    fn new(value: u8) -> Result<Self, PercentageError> {
        if value > 100 {
            Err(PercentageError::OutOfRange(value))
        } else {
            Ok(Self { value })
        }
    }

    fn value(&self) -> u8 {
        self.value
    }
}

fn apply_discount(price: f64, pct: &Percentage) -> f64 {
    price * (1.0 - pct.value() as f64 / 100.0)
}

fn main() {
    match Percentage::new(20) {
        Ok(discount) => {
            let final_price = apply_discount(100.0, &discount);
            println!("20% off: ${final_price:.2}");
        }
        Err(PercentageError::OutOfRange(n)) => {
            println!("invalid: {n} is out of range");
        }
    }

    match Percentage::new(150) {
        Ok(discount) => {
            let final_price = apply_discount(100.0, &discount);
            println!("price: ${final_price:.2}");
        }
        Err(PercentageError::OutOfRange(n)) => {
            println!("invalid: {n} is out of range");
        }
    }
}
```

Output:

```
20% off: $80.00
invalid: 150 is out of range
```

Once you have a `Percentage`, you _know_ it is between 0 and 100. Every function
that receives a `Percentage` can skip validation — the type already guarantees
the constraint. Validation happens once, at the boundary where data enters your
system, and the type carries the proof everywhere it goes.

#### Why the field is not public

Notice that the `value` field has no `pub` keyword — it is private. This is what
makes the pattern work. Without access to the field, the only way to create a
`Percentage` is through `Percentage::new`, which enforces the 0–100 range.

In Rust, privacy is enforced at the _module_ boundary. Code in the same module
can still access private fields directly, so it could write
`Percentage { value: 200 }` and bypass the constructor. For the guarantee to
hold in a real project, place the type in its own module:

```rust,ignore
// In a real project: src/percentage.rs (or a mod block)
mod percentage {
    #[derive(Debug)]
    pub struct Percentage {
        value: u8, // private — only this module can touch it
    }

    #[derive(Debug)]
    pub enum PercentageError {
        OutOfRange(u8),
    }

    impl Percentage {
        pub fn new(value: u8) -> Result<Self, PercentageError> {
            if value > 100 {
                Err(PercentageError::OutOfRange(value))
            } else {
                Ok(Self { value })
            }
        }

        pub fn value(&self) -> u8 {
            self.value
        }
    }
}

fn main() {
    use percentage::{Percentage, PercentageError};

    match Percentage::new(20) {
        Ok(pct) => println!("valid: {}%", pct.value()),
        Err(PercentageError::OutOfRange(n)) => println!("invalid: {n}"),
    }

    // This would not compile — `value` is private outside the module:
    // let bypass = Percentage { value: 200 };
}
```

Output: `valid: 20%`

This is a pattern you will see throughout production Rust: a _public type_ with
_private fields_ and a _validating constructor_. The module boundary is the wall;
the constructor is the gate. The key insight: private fields plus a validating
constructor is how
Rust enforces invariants at compile time.

## Putting It All Together

Here is a complete example that combines `Option`, `Result`, `?`, and the
principles of making illegal states unrepresentable. Notice one deliberate
choice in the second half: instead of collapsing "keep the valid scores" into a
compact iterator pipeline, the example writes that rule out step by step. At
this stage, clarity matters more than compression.

Example 3-5. Combining Option, Result, ?, and validated construction

```rust
use std::fmt;

#[derive(Debug)]
struct Score {
    value: u32,
}

#[derive(Debug)]
enum ScoreError {
    InvalidInput(String),
    OutOfRange(u32),
}

impl fmt::Display for ScoreError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            ScoreError::InvalidInput(s) => write!(f, "not a number: '{s}'"),
            ScoreError::OutOfRange(n) => write!(f, "score {n} is out of range (0-100)"),
        }
    }
}

impl Score {
    fn new(value: u32) -> Result<Self, ScoreError> {
        if value > 100 {
            Err(ScoreError::OutOfRange(value))
        } else {
            Ok(Self { value })
        }
    }

    fn grade(&self) -> &str {
        match self.value {
            90..=100 => "A",
            80..=89 => "B",
            70..=79 => "C",
            60..=69 => "D",
            _ => "F",
        }
    }
}

fn parse_score(input: &str) -> Result<Score, ScoreError> {
    let number: u32 = input.trim().parse()
        .map_err(|_| ScoreError::InvalidInput(input.to_string()))?;
    Score::new(number)
}

fn process_scores(inputs: &[&str]) -> Vec<Score> {
    let mut valid = Vec::new();

    for input in inputs {
        match parse_score(input) {
            Ok(score) => {
                println!("{:>3} -> grade {}", score.value, score.grade());
                valid.push(score);
            }
            Err(e) => println!("error: {e}"),
        }
    }

    valid
}

fn average(scores: &[Score]) -> Option<f64> {
    if scores.is_empty() {
        return None;
    }
    let sum: u32 = scores.iter().map(|s| s.value).sum();
    Some(sum as f64 / scores.len() as f64)
}

fn main() {
    let inputs = ["85", "92", "abc", "150", "73", "  60  "];
    let valid = process_scores(&inputs);

    println!();

    match average(&valid) {
        Some(avg) => println!("average of valid scores: {avg:.1}"),
        None => println!("no valid scores"),
    }

    // Empty case
    let empty: Vec<Score> = vec![];
    match average(&empty) {
        Some(avg) => println!("average: {avg:.1}"),
        None => println!("no scores to average"),
    }
}
```

Output:

```
 85 -> grade B
 92 -> grade A
error: not a number: 'abc'
error: score 150 is out of range (0-100)
 73 -> grade C
 60 -> grade D

average of valid scores: 77.5
no scores to average
```

This example demonstrates several key patterns:

- **`Result` for fallible operations:** `parse_score` returns `Result` because
  parsing and validation can both fail. The `?` operator chains the two fallible
  steps — parsing the string and validating the range — into a clean, linear
  flow.
- **`Option` for genuine absence:** `average` returns `Option` because the
  average of an empty list is not an error — it simply does not exist.
- **Custom error types with `Display`:** `ScoreError` is an enum whose variants
  describe exactly what went wrong. The `Display` implementation provides
  human-readable messages.
- **Validated construction:** Once you have a `Score`, you know it is between
  0 and 100. The `grade` method does not need to check for out-of-range
  values — the type guarantees they cannot exist.
- **An explicit collection loop:** `process_scores` pushes each successful
  `Score` into a `Vec`. The rule stays visible: print the error, or keep the
  validated value for later work.
- **Normalization lives inside the function:** `parse_score` calls
  `input.trim()` itself rather than asking every caller to trim first. When
  a transformation is part of a function's contract — trimming, lowercasing,
  resolving a path — do it once inside the function. Self-contained
  functions compose; functions that demand pre-processed input invite bugs
  at every call site.

## Why This Matters

`Option` and `Result` are not just replacements for null and exceptions — they
represent a fundamentally different approach to handling absence and failure.
Instead of invisible trapdoors that crash your program at runtime, the type
system makes every potential failure visible and forces you to address it.

This has a compounding effect. When every function that can fail says so in its
type signature, you can trust the functions that _do not_ return `Result` — they
genuinely cannot fail. When every value that might be absent is wrapped in
`Option`, a plain `T` genuinely always exists. The type system becomes a map
of where things can go wrong, and the compiler ensures you have a plan for each
case.

Key points from this chapter:

- **`Option<T>`** replaces null. A value is either `Some(T)` or `None`, and the
  compiler forces you to handle both cases.
- **`Result<T, E>`** replaces exceptions. An operation either succeeds with
  `Ok(T)` or fails with `Err(E)`, and the failure mode is visible in the type
  signature.
- **The `?` operator** propagates errors concisely: extract the success value
  or return early with the error. It works with both `Result` and `Option`.
- **`unwrap` and `expect`** should be reserved for tests, prototyping, and
  cases where you can prove the value exists. In production code, prefer `?`
  or explicit matching.
- **Make illegal states unrepresentable.** Use enums to encode valid state
  combinations, newtypes to prevent value mixups, and validated constructors
  to guarantee constraints. Let the type system enforce your invariants so
  runtime checks become unnecessary.

```rust
fn parse_line(line: &str) -> Result<(&str, &str), String> {
    let trimmed = line.trim();
    if trimmed.is_empty() || trimmed.starts_with('#') {
        return Err(String::from("skip"));
    }
    trimmed
        .split_once('=')
        .ok_or(format!("invalid line (no '='): '{trimmed}'"))
}

fn parse_port(value: &str) -> Result<u16, String> {
    let value = value.trim();
    let n: u16 = value.parse().map_err(|_| format!("invalid port: '{value}'"))?;
    if n == 0 {
        return Err(String::from("port must be nonzero"));
    }
    Ok(n)
}

fn load_config(lines: &[&str]) -> Result<(String, u16, bool), String> {
    let mut host: Option<String> = None;
    let mut port: Option<u16> = None;
    let mut verbose: Option<bool> = None;

    for line in lines {
        let (key, value) = match parse_line(line) {
            Ok(pair) => pair,
            Err(e) if e == "skip" => continue,
            Err(e) => return Err(e),
        };

        match key.trim() {
            "host" => host = Some(value.trim().to_string()),
            "port" => port = Some(parse_port(value)?),
            "verbose" => {
                verbose = Some(match value.trim() {
                    "true" => true,
                    "false" => false,
                    other => return Err(format!("invalid bool: '{other}'")),
                });
            }
            other => return Err(format!("unknown key: '{other}'")),
        }
    }

    let host = host.ok_or(String::from("missing key: 'host'"))?;
    let port = port.ok_or(String::from("missing key: 'port'"))?;
    let verbose = verbose.unwrap_or(false);

    Ok((host, port, verbose))
}

fn main() {
    let config_lines = [
        "# Server configuration",
        "host = localhost",
        "port = 8080",
        "verbose = true",
        "",
    ];

    match load_config(&config_lines) {
        Ok((host, port, verbose)) => {
            println!("host:    {host}");
            println!("port:    {port}");
            println!("verbose: {verbose}");
        }
        Err(e) => println!("config error: {e}"),
    }

    println!();

    // Missing required key
    let incomplete = ["host = example.com"];
    match load_config(&incomplete) {
        Ok((host, port, verbose)) => {
            println!("host: {host}, port: {port}, verbose: {verbose}");
        }
        Err(e) => println!("config error: {e}"),
    }

    println!();

    // Invalid port
    let bad_port = ["host = example.com", "port = abc"];
    match load_config(&bad_port) {
        Ok((host, port, verbose)) => {
            println!("host: {host}, port: {port}, verbose: {verbose}");
        }
        Err(e) => println!("config error: {e}"),
    }
}
```

> **Looking ahead: the `"skip"` sentinel**
>
> The solution above uses `Err(String::from("skip"))` to signal "this line
> is a comment or blank, please skip it." This works, but it mixes two
> different concepts in one channel: the `Err` variant should mean "real
> failure," not "control-flow signal." If a future error message ever
> happens to be the string `"skip"`, the loop will silently swallow it.
>
> A cleaner shape returns `Result<Option<(&str, &str)>, String>`:
> `Ok(Some(pair))` is a parsed line, `Ok(None)` is "intentionally skip,"
> and `Err(_)` is reserved for actual failures. The call site then uses
> `?` for errors and matches on `Some`/`None` for the skip case — no
> string comparison needed. Try refactoring the solution to that
> signature; it is a small change with a real readability win, and it
> shows how layered types (`Result<Option<...>, _>`) can carry two
> independent signals without overloading one of them.

fn find_in_warehouse(item: &str) -> Option<u32> {
    match item {
        "bolt" => Some(200),
        "washer" => Some(500),
        _ => None,
    }
}

    match item {
        "bolt" => Some(15),
        "nut" => Some(40),
        _ => None,

fn lookup(item: &str) -> Option<String> {
    find_in_warehouse(item)
        .or_else(|| find_in_store(item))
        .filter(|&qty| qty > 0)

    match lookup(item) {
        Some(msg) => println!("{msg}"),

        "washer" => Some(0.10),

    }

    let qty = find_in_warehouse(item)
        .or_else(|| find_in_store(item))?;

    Empty,
    MissingAt,
    MissingDomain,
    MissingLocal,
}

    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            EmailError::Empty => write!(f, "email cannot be empty"),
            EmailError::MissingAt => write!(f, "email must contain '@'"),
            EmailError::MissingDomain => write!(f, "email must have a domain after '@'"),
            EmailError::MissingLocal => write!(f, "email must have a name before '@'"),

    fn new(input: &str) -> Result<Self, EmailError> {
        let trimmed = input.trim();
        if trimmed.is_empty() {
            return Err(EmailError::Empty);

        let (local, domain) = trimmed
            .split_once('@')

        if domain.is_empty() {
            return Err(EmailError::MissingDomain);
        }

        self.value.split_once('@').expect("validated email must contain '@'").1
    }

    fn local(&self) -> &str {
        self.value.split_once('@').expect("validated email must contain '@'").0
    }
}

    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "{}", self.value)
    }
}

    if email.domain().ends_with(".edu") {
        "student"
    } else if email.domain().ends_with(".gov") {

    } else {
        "general"
    }
}

    }
}

use std::fmt;

#[derive(Debug)]
struct ProductId(u32);

#[derive(Debug)]
struct Quantity(u32);

#[derive(Debug)]
enum OrderError {
    UnknownProduct(u32),
    InvalidQuantity,
    OutOfStock(u32),
    ParseError(String),
}

impl fmt::Display for OrderError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            OrderError::UnknownProduct(id) => write!(f, "unknown product: #{id}"),
            OrderError::InvalidQuantity => write!(f, "quantity must be at least 1"),
            OrderError::OutOfStock(available) => {
                write!(f, "product is out of stock ({available} available)")
            }
            OrderError::ParseError(msg) => write!(f, "{msg}"),
        }
    }
}

impl Quantity {
    fn new(n: u32) -> Result<Self, OrderError> {
        if n == 0 {
            Err(OrderError::InvalidQuantity)
        } else {
            Ok(Self(n))
        }
    }
}

fn lookup_price(id: &ProductId) -> Option<f64> {
    match id.0 {
        1 => Some(10.00),
        2 => Some(25.50),
        3 => Some(5.75),
        _ => None,
    }
}

fn check_stock(id: &ProductId) -> Option<u32> {
    match id.0 {
        1 => Some(100),
        2 => Some(0),
        3 => Some(50),
        _ => None,
    }
}

fn parse_order(input: &str) -> Result<(ProductId, Quantity), OrderError> {
    let (id_str, qty_str) = input
        .split_once(':')
        .ok_or(OrderError::ParseError(String::from("expected 'id:qty' format")))?;

    let id: u32 = id_str.trim().parse()
        .map_err(|_| OrderError::ParseError(format!("invalid product id: '{}'", id_str.trim())))?;

    let qty: u32 = qty_str.trim().parse()
        .map_err(|_| OrderError::ParseError(format!("invalid quantity: '{}'", qty_str.trim())))?;

    let quantity = Quantity::new(qty)?;
    Ok((ProductId(id), quantity))
}

fn process_order(input: &str) -> Result<String, OrderError> {
    let (product_id, quantity) = parse_order(input)?;

    let price = lookup_price(&product_id)
        .ok_or(OrderError::UnknownProduct(product_id.0))?;

    let stock = check_stock(&product_id)
        .ok_or(OrderError::UnknownProduct(product_id.0))?;

    if stock < quantity.0 {
        return Err(OrderError::OutOfStock(stock));
    }

    let total = price * quantity.0 as f64;
    Ok(format!(
        "{}x product #{} @ ${:.2} = ${:.2}",
        quantity.0, product_id.0, price, total
    ))
}

fn main() {
    let inputs = ["1:3", "2:1", "99:1", "1:0", "bad"];

    for input in inputs {
        match process_order(input) {
            Ok(summary) => println!("{input:>5?} -> {summary}"),
            Err(e) => println!("{input:>5?} -> error: {e}"),
        }
    }
}

use std::collections::HashMap;

enum AccountStatus {
    Active { credit_limit: u32 },
    Suspended { reason: String },
    Closed,
}

fn describe_status(status: &AccountStatus) -> String {
    match status {
        AccountStatus::Active { credit_limit } => {
            format!("active (limit: ${credit_limit})")
        }
        AccountStatus::Suspended { reason } => {
            format!("suspended ({reason})")
        }
        AccountStatus::Closed => String::from("closed"),
    }
}

fn get_credit_limit(
    accounts: &HashMap<u32, AccountStatus>,
    id: u32,
) -> Option<u32> {
    accounts.get(&id).and_then(|status| match status {
        AccountStatus::Active { credit_limit } => Some(*credit_limit),
        _ => None,
    })
}

fn approve_transaction(
    accounts: &HashMap<u32, AccountStatus>,
    primary_id: u32,
    backup_id: u32,
    amount: u32,
) -> Option<String> {
    // Try primary, fall back to backup with eager .or
    let limit = get_credit_limit(accounts, primary_id)
        .or(get_credit_limit(accounts, backup_id))?;

    // Filter: only approve if amount is within limit
    Some(amount)
        .filter(|&a| a <= limit)
        .map(|a| format!("approved ${a} (limit: ${limit})"))
}

fn has_usable_account(accounts: &HashMap<u32, AccountStatus>, id: u32) -> bool {
    get_credit_limit(accounts, id).is_some_and(|limit| limit > 0)
}

fn within_policy(
    accounts: &HashMap<u32, AccountStatus>,
    id: u32,
    max_limit: u32,
) -> bool {
    get_credit_limit(accounts, id).is_none_or(|limit| limit <= max_limit)
}

fn main() {
    let mut accounts = HashMap::new();
    accounts.insert(1, AccountStatus::Active { credit_limit: 500 });
    accounts.insert(
        2,
        AccountStatus::Suspended { reason: String::from("fraud") },
    );
    accounts.insert(3, AccountStatus::Active { credit_limit: 1000 });
    accounts.insert(4, AccountStatus::Closed);

    println!("--- Account Status ---");
    for id in [1, 2, 3, 4] {
        let status = accounts.get(&id).unwrap();
        println!("account {id}: {}", describe_status(status));
    }

    println!("\n--- Transactions ---");
    let cases: [(u32, u32, u32); 4] = [
        (1, 3, 200),
        (2, 3, 800),
        (2, 4, 100),
        (1, 3, 9999),
    ];
    for (primary, backup, amount) in cases {
        match approve_transaction(&accounts, primary, backup, amount) {
            Some(msg) => println!("({primary},{backup},${amount}): {msg}"),
            None => println!("({primary},{backup},${amount}): denied"),
        }
    }

    println!("\n--- Eligibility Checks ---");
    println!("account 1 usable: {}", has_usable_account(&accounts, 1));
    println!("account 2 usable: {}", has_usable_account(&accounts, 2));
    println!("account 9 usable: {}", has_usable_account(&accounts, 9));

    println!(
        "account 1 within 500 policy: {}",
        within_policy(&accounts, 1, 500)
    );
    println!(
        "account 3 within 500 policy: {}",
        within_policy(&accounts, 3, 500)
    );
    println!(
        "account 9 within 500 policy: {}",
        within_policy(&accounts, 9, 500)
    );
}

    UnknownStudent(u32),
    ParseError(String),
    InvalidScore(i32),
}

impl fmt::Display for GradeError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            GradeError::UnknownStudent(id) => write!(f, "unknown student: #{id}"),
            GradeError::ParseError(msg) => write!(f, "parse error: {msg}"),
            GradeError::InvalidScore(n) => write!(f, "invalid score: {n}"),
        }
    }
}

fn lookup_scores(id: &StudentId) -> Option<Vec<Option<i32>>> {
    match id.0 {
        1 => Some(vec![Some(85), None, Some(92), Some(78)]),
        2 => Some(vec![Some(60), Some(55), None, None]),
        3 => Some(vec![None, None, None]),

    let mut count = 0;
    let mut sum = 0;

    for score in scores {
        if let Some(n) = score {
            sum += n;
            count += 1;
        }
    }

    if count == 0 {
        return None;
    }

    Some(sum as f64 / count as f64)
}

fn student_report(id: u32) -> Result<String, GradeError> {
    let student_id = StudentId(id);

    let name = lookup_name(&student_id)
        .ok_or(GradeError::UnknownStudent(id))?;

    let scores = lookup_scores(&student_id)
        .ok_or(GradeError::UnknownStudent(id))?;

    // Count valid scores using map + ok() + flatten:
    // .map(validate_score) -> Option<Result<i32, _>>
    // .map(|r| r.ok())     -> Option<Option<i32>>
    // .flatten()            -> Option<i32>
    let valid_count: usize = scores
        .iter()
        .filter(|opt| {
            opt.map(validate_score)
                .map(|r| r.ok())
                .flatten()
                .is_some()
        })
        .count();

    let submitted = scores.iter().filter(|s| s.is_some()).count();
    let missing = scores.iter().filter(|s| s.is_none()).count();

    let average = compute_average(&scores);

    let grade = average
        .map(|avg| match avg as u32 {
            90..=100 => "A",
            80..=89 => "B",
            70..=79 => "C",
            60..=69 => "D",
            _ => "F",
        })
        .unwrap_or("N/A");

    let avg_str = average
        .map(|a| format!("{a:.1}"))
        .unwrap_or(String::from("none"));

    Ok(format!(
        "{name}: {submitted} submitted ({valid_count} valid), \
         {missing} missing, avg {avg_str}, grade {grade}"
    ))
}

fn main() {
    // Student reports: flatten, ok_or, ok(), map on Option
    println!("--- Student Reports ---");
    for id in [1, 2, 3, 99] {
        match student_report(id) {
            Ok(report) => println!("{report}"),
            Err(e) => println!("error: {e}"),
        }
    }

    // parse_and_validate: and_then on Result
    println!("\n--- Parse and Validate ---");
    let inputs = ["85", "abc", "-10", "150", "72"];
    for input in inputs {
        let result = parse_and_validate(input);
        match result {
            Ok(n) => println!("{input:>5} -> Ok({n})"),
            Err(ref e) => println!("{input:>5} -> Err({e})"),
        }
    }

    // map on Result to transform success values
    println!("\n--- Doubled Scores ---");
    for input in ["40", "abc", "60"] {
        let doubled = parse_and_validate(input).map(|n| n * 2);
        match doubled {
            Ok(n) => println!("{input:>3} -> {n}"),
            Err(e) => println!("{input:>3} -> error: {e}"),
        }
    }

    // is_some_and / is_none_or on computed averages
    println!("\n--- Eligibility ---");
    for id in [1, 2, 3] {
        let student_id = StudentId(id);
        let avg = lookup_scores(&student_id)
            .and_then(|scores| compute_average(&scores));

        let honors = avg.is_some_and(|a| a >= 80.0);
        let not_failing = avg.is_none_or(|a| a >= 60.0);
        println!("student {id}: honors={honors}, not_failing={not_failing}");
    }
}
