# Enums and Pattern Matching

In the previous chapter, you learned to group related data into a struct — a user
has a name and an email, a rectangle has a width and a height. But not every
value _is_ a collection of fields. Some values are one thing _or_ another. A
traffic light is red _or_ yellow _or_ green. A web request either succeeds _or_
fails. A shape could be a circle _or_ a rectangle _or_ a triangle.

In many languages, you would reach for a string, an integer code, or a boolean
flag to represent these alternatives. But strings can be misspelled, integers can
hold invalid values, and booleans only give you two options. The compiler cannot
help you if you forget to handle a case.

Rust solves this with _enums_ — types that represent one of several possible
variants. But Rust enums are fundamentally different from enums in C or Java.
Each variant can carry its own data, making them one of the most powerful
modeling tools in the language. Combined with _pattern matching_, the compiler
guarantees you handle every possible case. No forgotten branches, no default
cases hiding bugs.

## Defining an Enum

An _enum_ defines a type whose value is exactly one of several named variants:

```rust
enum Direction {
    North,
    South,
    East,
    West,
}

fn main() {
    let directions = [
        Direction::North,
        Direction::South,
        Direction::East,
        Direction::West,
    ];

    for heading in &directions {
        match heading {
            Direction::North => println!("heading north"),
            Direction::South => println!("heading south"),
            Direction::East => println!("heading east"),
            Direction::West => println!("heading west"),
        }
    }
}
```

Output:

```
heading north
heading south
heading east
heading west
```

The `enum Direction` declaration creates a new type with four variants. You refer
to each variant through the type name: `Direction::North`, `Direction::South`,
and so on. A `Direction` value is always exactly one of these four — nothing
else.

This already improves on using strings or integers. You cannot misspell a
variant name — the compiler catches it. You cannot create a `Direction` with an
invalid value — the type system prevents it.

### Enums That Carry Data

Here is where Rust enums become something entirely different from what you may
have seen in other languages. Each variant can hold data, and different variants
can hold _different types_ of data:

```rust
enum Shape {
    Circle(f64),               // radius
    Rectangle(f64, f64),       // width, height
    Triangle(f64, f64, f64),   // three sides
}

fn area(shape: &Shape) -> f64 {
    match shape {
        Shape::Circle(radius) => std::f64::consts::PI * radius * radius,
        Shape::Rectangle(w, h) => w * h,
        Shape::Triangle(a, b, c) => {
            let s = (a + b + c) / 2.0;
            (s * (s - a) * (s - b) * (s - c)).sqrt()
        }
    }
}

fn main() {
    let shapes = [
        Shape::Circle(5.0),
        Shape::Rectangle(4.0, 6.0),
        Shape::Triangle(3.0, 4.0, 5.0),
    ];

    for shape in &shapes {
        println!("area: {:.2}", area(shape));
    }
}
```

Output:

```
area: 78.54
area: 24.00
area: 6.00
```

`Shape::Circle` holds one `f64`. `Shape::Rectangle` holds two. `Shape::Triangle`
holds three. They are all the same type — `Shape` — but they carry different
data. This is called an _algebraic data type_ (or _sum type_ in type theory),
and it is one of the most expressive ideas in programming language design.

Compare this to a struct-based approach: you would need separate structs for each
shape, a trait to unify them, and dynamic dispatch. With an enum, one type
handles all cases, and the compiler ensures every function that processes a
`Shape` handles all variants.

### Variants with Named Fields

Variants can also use named fields, just like structs:

```rust
enum Event {
    Click { x: i32, y: i32 },
    KeyPress(char),
    Resize { width: u32, height: u32 },
}

fn describe(event: &Event) -> String {
    match event {
        Event::Click { x, y } => format!("click at ({x}, {y})"),
        Event::KeyPress(c) => format!("key: {c}"),
        Event::Resize { width, height } => format!("resize to {width}x{height}"),
    }
}

fn main() {
    let events = [
        Event::Click { x: 100, y: 200 },
        Event::KeyPress('q'),
        Event::Resize { width: 1920, height: 1080 },
    ];

    for event in &events {
        println!("{}", describe(event));
    }
}
```

Output:

```
click at (100, 200)
key: q
resize to 1920x1080
```

Named fields make the data self-documenting. `Event::Click { x: 100, y: 200 }`
is immediately clear. `Event::Click(100, 200)` would require you to remember
which number is which.

### Methods on Enums

In the previous chapter, you attached behavior to structs with `impl` blocks.
The same syntax works for enums — and it is where enums become truly powerful.
Every method you have called on `Option` (`is_some`, `unwrap_or`) or `Result`
(`is_ok`, `map`) was defined in an `impl` block on an enum. Now you can write
your own:

```rust
enum TrafficLight {
    Red,
    Yellow,
    Green,
}

impl TrafficLight {
    fn duration_seconds(&self) -> u32 {
        match self {
            TrafficLight::Red => 60,
            TrafficLight::Yellow => 5,
            TrafficLight::Green => 45,
        }
    }

    fn is_stop(&self) -> bool {
        match self {
            TrafficLight::Red => true,
            TrafficLight::Yellow => true,
            TrafficLight::Green => false,
        }
    }

    fn next(self) -> TrafficLight {
        match self {
            TrafficLight::Red => TrafficLight::Green,
            TrafficLight::Yellow => TrafficLight::Red,
            TrafficLight::Green => TrafficLight::Yellow,
        }
    }
}

fn main() {
    let light = TrafficLight::Red;
    println!("{} seconds, stop: {}", light.duration_seconds(), light.is_stop());

    let light = light.next();
    println!("{} seconds, stop: {}", light.duration_seconds(), light.is_stop());

    let light = light.next();
    println!("{} seconds, stop: {}", light.duration_seconds(), light.is_stop());
}
```

Output:

```
60 seconds, stop: true
45 seconds, stop: false
5 seconds, stop: true
```

The pattern is always the same: `match self` inside the method body, with one arm
per variant. The three receiver types work exactly as they do on structs:

- `&self` borrows the enum — `duration_seconds` and `is_stop` inspect the
  current variant without consuming it.
- `self` takes ownership — `next` consumes the current light and returns a new
  one. After `light.next()`, the original `light` is gone, so the code
  rebinds the name with `let light = light.next()` (shadowing, from the
  variables chapter).
- `&mut self` borrows mutably — useful when a method needs to change the
  variant in place.

Notice how the earlier `area` function was a standalone function that took
`&Shape` as a parameter. You could rewrite it as a method —
`impl Shape { fn area(&self) -> f64 { ... } }` — so callers write
`shape.area()` instead of `area(&shape)`. As a rule of thumb, if a function's
first parameter is a reference to your type, it probably belongs in an `impl`
block.

## Exhaustive Matching with Match

The `match` expression is Rust's primary tool for working with enums. It
compares a value against a series of patterns and executes the code for the
first pattern that matches:

```rust
enum Coin {
    Penny,
    Nickel,
    Dime,
    Quarter,
}

fn value_in_cents(coin: &Coin) -> u32 {
    match coin {
        Coin::Penny => 1,
        Coin::Nickel => 5,
        Coin::Dime => 10,
        Coin::Quarter => 25,
    }
}

fn main() {
    let coins = [Coin::Penny, Coin::Nickel, Coin::Dime, Coin::Quarter];
    for coin in &coins {
        println!("{} cents", value_in_cents(coin));
    }
}
```

Output:

```
1 cents
5 cents
10 cents
25 cents
```

The critical property of `match` is _exhaustiveness_ — the compiler requires
that every possible variant is covered. If you remove one arm, the compiler
tells you exactly which case is missing:

```rust,does_not_compile
enum Coin {
    Penny,
    Nickel,
    Dime,
    Quarter,
}

fn value_in_cents(coin: &Coin) -> u32 {
    match coin {
        Coin::Penny => 1,
        Coin::Nickel => 5,
        Coin::Dime => 10,
        // forgot Quarter!
    }
}
```

```
error[E0004]: non-exhaustive patterns: `&Coin::Quarter` not covered
 --> src/main.rs:9:11
  |
9 |     match coin {
  |           ^^^^ pattern `&Coin::Quarter` not covered
  |
note: `Coin` defined here
 --> src/main.rs:1:6
  |
1 | enum Coin {
  |      ^^^^
...
5 |     Quarter,
  |     ------- not covered
  = note: the matched value is of type `&Coin`
help: ensure that all possible cases are being handled by adding a match arm with a
      wildcard pattern or an explicit pattern as shown
  |
12~         Coin::Dime => 10,
13~         &Coin::Quarter => todo!(),
  |
```

This is not a warning — it is a hard error. Your code will not compile until
every case is handled. The compiler even shows you _where_ the missing variant
is defined and suggests adding an arm with `todo!()` so you can get the code
compiling and fill in the logic later. When you add a new variant to an enum,
the compiler points you to every `match` expression that needs updating. No
forgotten branches, no silent bugs.

### Match Is an Expression

Like `if`, `match` is an expression that returns a value. Every arm must return
the same type:

```rust
enum Season {
    Spring,
    Summer,
    Autumn,
    Winter,
}

fn main() {
    let seasons = [Season::Spring, Season::Summer, Season::Autumn, Season::Winter];

    for season in seasons {
        let description = match season {
            Season::Spring => "flowers blooming",
            Season::Summer => "long sunny days",
            Season::Autumn => "leaves falling",
            Season::Winter => "cozy blankets",
        };

        println!("{description}");
    }
}
```

Output:

```
flowers blooming
long sunny days
leaves falling
cozy blankets
```

The result of the `match` is bound directly to `description`. This is the
expression-oriented style you have been using since Chapter 2.1 — no need for
a mutable variable that you assign inside each branch.

### The Catch-All Pattern

Sometimes you only care about a few variants and want to handle the rest with a
single arm. The _wildcard pattern_ `_` matches anything:

```rust
enum HttpStatus {
    Ok,
    NotFound,
    InternalError,
    BadRequest,
    Unauthorized,
}

fn category(status: &HttpStatus) -> &str {
    match status {
        HttpStatus::Ok => "success",
        HttpStatus::NotFound => "client error",
        HttpStatus::BadRequest => "client error",
        HttpStatus::Unauthorized => "client error",
        _ => "server error",
    }
}

fn main() {
    let statuses = [
        HttpStatus::Ok,
        HttpStatus::NotFound,
        HttpStatus::BadRequest,
        HttpStatus::Unauthorized,
        HttpStatus::InternalError,
    ];

    for status in &statuses {
        println!("{}", category(status));
    }
}
```

Output:

```
success
client error
client error
client error
server error
```

The `_` arm catches any variant not explicitly listed. It must come last —
patterns are checked in order, and the first match wins.

Use `_` when you genuinely do not need the value. If you want to bind the
remaining cases to a variable for logging or debugging, use a name instead:

```rust
enum Command {
    Quit,
    Echo(String),
    Move { x: i32, y: i32 },
    Save,
}

fn execute(cmd: &Command) {
    match cmd {
        Command::Echo(msg) => println!("echo: {msg}"),
        Command::Move { x, y } => println!("move to ({x}, {y})"),
        _ => println!("simple command received"),
    }
}

fn main() {
    let commands = [
        Command::Echo(String::from("hello")),
        Command::Move { x: 10, y: 20 },
        Command::Quit,
        Command::Save,
    ];

    for cmd in &commands {
        execute(cmd);
    }
}
```

Output:

```
echo: hello
move to (10, 20)
simple command received
simple command received
```

A word of caution: catch-all arms can hide missing cases. If you add a new
variant to an enum, a `_` arm silently absorbs it. When correctness matters more
than convenience, prefer listing every variant explicitly.

### Or Patterns

When multiple variants should produce the same result, combine them with `|`:

```rust
enum Day {
    Monday,
    Tuesday,
    Wednesday,
    Thursday,
    Friday,
    Saturday,
    Sunday,
}

fn describe(day: &Day) -> &str {
    match day {
        Day::Saturday | Day::Sunday => "weekend",
        Day::Monday | Day::Friday => "almost there",
        _ => "midweek",
    }
}

fn main() {
    let week = [
        Day::Monday, Day::Tuesday, Day::Wednesday,
        Day::Thursday, Day::Friday, Day::Saturday, Day::Sunday,
    ];

    for day in &week {
        println!("{}", describe(day));
    }
}
```

Output:

```
almost there
midweek
midweek
midweek
almost there
weekend
weekend
```

The `|` pattern means "match this _or_ that." You can combine as many variants
as you need in a single arm.

When you only need a boolean answer — "does this value match a pattern?" — the
`matches!` macro is even more concise:

```rust
enum Day {
    Monday, Tuesday, Wednesday, Thursday, Friday, Saturday, Sunday,
}

fn main() {
    let week = [
        Day::Monday, Day::Tuesday, Day::Wednesday,
        Day::Thursday, Day::Friday, Day::Saturday, Day::Sunday,
    ];

    for day in &week {
        if matches!(day, Day::Saturday | Day::Sunday) {
            println!("weekend!");
        }
    }
}
```

Output:

```
weekend!
weekend!
```

`matches!` returns `true` if the value matches the pattern and `false`
otherwise. It is the idiomatic choice when you do not need to extract data from
the match — just test whether a pattern applies.

## If Let: When You Care About One Case

Sometimes you have an enum value and you only care about one variant. Writing a
full `match` with a `_ => ()` arm for the rest feels excessive:

```rust
enum Notification {
    Email(String),
    Sms,
    Push,
}

fn main() {
    let notifications = [
        Notification::Email(String::from("you have a meeting")),
        Notification::Push,
        Notification::Sms,
        Notification::Email(String::from("your code shipped")),
    ];

    // Only print email notifications
    for n in &notifications {
        if let Notification::Email(subject) = n {
            println!("email: {subject}");
        }
    }
}
```

Output:

```
email: you have a meeting
email: your code shipped
```

`if let` is syntactic sugar for a `match` with one arm and a wildcard. The
pattern goes on the left of `=`, and the value on the right. If the pattern
matches, the variables inside it are bound and the body executes. If not, the
optional `else` branch runs (or execution continues past the block).

### When to Prefer Match Over If Let

You can chain `if let` / `else if let` branches to handle multiple variants:

```rust
enum Discount {
    Percentage(u32),
    Fixed(f64),
    None,
}

fn describe_discount(d: &Discount) -> String {
    if let Discount::Percentage(pct) = d {
        format!("{pct}% off")
    } else if let Discount::Fixed(amount) = d {
        format!("${amount:.2} off")
    } else {
        String::from("no discount")
    }
}

fn main() {
    println!("{}", describe_discount(&Discount::Percentage(20)));
    println!("{}", describe_discount(&Discount::Fixed(5.0)));
    println!("{}", describe_discount(&Discount::None));
}
```

Output:

```
20% off
$5.00 off
no discount
```

This works, but when you find yourself chaining multiple `if let` / `else if
let` branches, ask whether a `match` would be clearer. A `match` gives you
_exhaustiveness checking_ — if you add a fourth variant to `Discount`, the
compiler will remind you to handle it. An `if let` chain silently falls through
to the final `else`.

## Let Else: Keeping the Happy Path Flat

`if let` runs code when a pattern _matches_. But often you want the opposite —
check a pattern, bail out early if it _doesn't_ match, and continue with the
bound variables on the happy path. This is the _guard clause_ style, and Rust
has a dedicated construct for it: `let`-`else`.

```rust
fn parse_and_double(input: &str) -> Option<i64> {
    let Ok(n) = input.parse::<i64>() else {
        return None;
    };
    // n is available here — the happy path continues
    Some(n * 2)
}

fn main() {
    println!("{:?}", parse_and_double("21"));
    println!("{:?}", parse_and_double("abc"));
}
```

Output:

```
Some(42)
None
```

The pattern goes on the left of `=`, the value on the right. If the pattern
matches, the bound variables (`n` in this case) are available for the rest of the
enclosing block — not just inside a nested `if` body. If the pattern does not
match, the `else` block executes, and it _must diverge_: it must leave the
enclosing scope entirely with `return`, `break`, `continue`, or a call that
never returns (like `panic!`).

This is the critical difference from `if let`. With `if let`, the bound variables
live inside a nested block. With `let`-`else`, they live at the _same
indentation level_ as the surrounding code:

```rust,ignore
// if let — happy path is indented
if let Ok(n) = input.parse::<i64>() {
    // n only available here, inside the block
    println!("{n}");
}

// let-else — happy path stays flat
let Ok(n) = input.parse::<i64>() else {
    return;
};
// n available here, at the same level as everything else
println!("{n}");
```

When a function validates several conditions before doing its real work,
`let`-`else` keeps the logic flat instead of deeply nested:

```rust
struct Config {
    database_url: Option<String>,
    max_connections: Option<u32>,
    timeout_secs: Option<u64>,
}

fn connect(config: &Config) {
    let Some(url) = &config.database_url else {
        println!("error: database_url is required");
        return;
    };

    let Some(max_conn) = config.max_connections else {
        println!("error: max_connections is required");
        return;
    };

    let timeout = config.timeout_secs.unwrap_or(30);

    println!("connecting to {url} (max {max_conn} connections, {timeout}s timeout)");
}

fn main() {
    let good = Config {
        database_url: Some(String::from("postgres://localhost/mydb")),
        max_connections: Some(10),
        timeout_secs: None,
    };
    connect(&good);

    let bad = Config {
        database_url: None,
        max_connections: Some(5),
        timeout_secs: Some(60),
    };
    connect(&bad);
}
```

Output:

```
connecting to postgres://localhost/mydb (max 10 connections, 30s timeout)
error: database_url is required
```

Each `let`-`else` validates one condition and returns early on failure. The main
logic stays at the top level of the function, reading top to bottom without
nesting deeper with each check. Compare what the same function looks like with
nested `if let`:

```rust,ignore
// Without let-else — the pyramid of doom
fn connect(config: &Config) {
    if let Some(url) = &config.database_url {
        if let Some(max_conn) = config.max_connections {
            let timeout = config.timeout_secs.unwrap_or(30);
            println!("connecting to {url} (max {max_conn}, {timeout}s)");
        } else {
            println!("error: max_connections is required");
        }
    } else {
        println!("error: database_url is required");
    }
}
```

The `let`-`else` version is not just shorter — it is structurally clearer. Each
validation stands on its own, and the error handling sits right next to the
condition it checks.

### When to Use Each

Use `if let` when you want to do something _only if_ a pattern matches, and
nothing special otherwise. Use `let`-`else` when you want to _continue_ with the
bound value and bail out if the pattern fails. Use `match` when you need to
handle multiple variants or want exhaustiveness checking.

## Let Chains: Combining Patterns and Conditions

Rust 2024 introduces _let chains_ in `if` and `while` conditions — the ability
to combine `let` bindings with boolean expressions using `&&`:

```rust
fn process_score(input: &str) {
    if let Ok(score) = input.parse::<u32>() && score >= 50 {
        println!("{score} is a passing score");
    } else {
        println!("'{input}' is not a passing score");
    }
}

fn main() {
    process_score("85");
    process_score("30");
    process_score("abc");
}
```

Output:

```
85 is a passing score
'30' is not a passing score
'abc' is not a passing score
```

Without let chains, you would need nested `if let` and `if` blocks:

```rust
fn process_score_nested(input: &str) {
    if let Ok(score) = input.parse::<u32>() {
        if score >= 50 {
            println!("{score} is a passing score");
        } else {
            println!("'{input}' is not a passing score");
        }
    } else {
        println!("'{input}' is not a passing score");
    }
}

fn main() {
    process_score_nested("85");
}
```

Output: `85 is a passing score`

Let chains flatten the nesting. You can chain multiple `let` bindings and boolean
tests in a single condition:

```rust
enum Role {
    Admin,
    User(String),
    Guest,
}

struct Request {
    role: Role,
    path: String,
}

fn check_access(req: &Request) -> bool {
    // Admins always have access
    if let Role::Admin = &req.role {
        println!("admin accessing {}", req.path);
        return true;
    }

    // Regular users need a non-empty name and an /api path
    if let Role::User(name) = &req.role
        && !name.is_empty()
        && req.path.starts_with("/api")
    {
        println!("{name} accessing {}", req.path);
        true
    } else {
        false
    }
}

fn main() {
    let admin_req = Request {
        role: Role::Admin,
        path: String::from("/admin/settings"),
    };
    println!("access granted: {}", check_access(&admin_req));

    let user_req = Request {
        role: Role::User(String::from("alice")),
        path: String::from("/api/data"),
    };
    println!("access granted: {}", check_access(&user_req));

    let guest_req = Request {
        role: Role::Guest,
        path: String::from("/api/data"),
    };
    println!("access granted: {}", check_access(&guest_req));
}
```

Output:

```
admin accessing /admin/settings
access granted: true
alice accessing /api/data
access granted: true
access granted: false
```

Each `&&` in the condition is evaluated left to right. If any part fails — the
pattern does not match, or a boolean test is false — the entire condition is
false and the `else` branch (if any) executes.

## Destructuring

Pattern matching is not limited to simple variant names. You can _destructure_ —
pull apart — complex data structures to extract exactly the pieces you need.

### Destructuring Enums

You have already seen this with variants that carry data:

```rust
enum Message {
    Text(String),
    Position { x: i32, y: i32 },
    Quit,
}

fn handle(msg: &Message) {
    match msg {
        Message::Text(content) => println!("text: {content}"),
        Message::Position { x, y } => println!("move to ({x}, {y})"),
        Message::Quit => println!("quit"),
    }
}

fn main() {
    handle(&Message::Text(String::from("hello")));
    handle(&Message::Position { x: 10, y: 20 });
    handle(&Message::Quit);
}
```

Output:

```
text: hello
move to (10, 20)
quit
```

The pattern `Message::Text(content)` binds the inner `String` to `content`.
The pattern `Message::Position { x, y }` binds the named fields using the
field init shorthand you learned with structs — `x` means `x: x`.

### Destructuring Structs in Match

You can destructure structs directly in patterns:

```rust
struct Point {
    x: f64,
    y: f64,
}

fn classify(point: &Point) -> &str {
    match point {
        Point { x: 0.0, y: 0.0 } => "origin",
        Point { x, y: 0.0 } if *x > 0.0 => "positive x-axis",
        Point { x: 0.0, y } if *y > 0.0 => "positive y-axis",
        _ => "somewhere else",
    }
}

fn main() {
    println!("{}", classify(&Point { x: 0.0, y: 0.0 }));
    println!("{}", classify(&Point { x: 5.0, y: 0.0 }));
    println!("{}", classify(&Point { x: 0.0, y: 3.0 }));
    println!("{}", classify(&Point { x: 1.0, y: 2.0 }));
}
```

Output:

```
origin
positive x-axis
positive y-axis
somewhere else
```

The `if *x > 0.0` after the pattern is a _match guard_ — an additional boolean
condition that must be true for the arm to match. Guards let you express
conditions that patterns alone cannot, like numerical comparisons.

### Destructuring Tuples

Tuples destructure naturally in patterns:

```rust
fn describe_pair(pair: (i32, i32)) -> &'static str {
    match pair {
        (0, 0) => "both zero",
        (x, 0) | (0, x) if x != 0 => "one zero",
        (x, y) if x == y => "equal",
        _ => "different",
    }
}

fn main() {
    println!("{}", describe_pair((0, 0)));
    println!("{}", describe_pair((5, 0)));
    println!("{}", describe_pair((3, 3)));
    println!("{}", describe_pair((1, 2)));
}
```

Output:

```
both zero
one zero
equal
different
```

### Nested Destructuring

Patterns compose — you can destructure nested structures in a single pattern:

```rust
enum Expr {
    Num(f64),
    Add(Box<Expr>, Box<Expr>),
    Neg(Box<Expr>),
}

fn eval(expr: &Expr) -> f64 {
    match expr {
        Expr::Num(n) => *n,
        Expr::Add(left, right) => eval(left) + eval(right),
        Expr::Neg(inner) => -eval(inner),
    }
}

fn main() {
    // 1 + (-(2 + 3))
    let expr = Expr::Add(
        Box::new(Expr::Num(1.0)),
        Box::new(Expr::Neg(
            Box::new(Expr::Add(
                Box::new(Expr::Num(2.0)),
                Box::new(Expr::Num(3.0)),
            )),
        )),
    );

    println!("result: {}", eval(&expr));
}
```

Output: `result: -4`

This example uses `Box<Expr>` — a heap-allocated pointer — because an `Expr` can
contain other `Expr` values. Without `Box`, the type would be infinitely large.
`Box` allocates data on the heap behind a pointer, letting you create recursive
types. Patterns see through it transparently, matching the inner data as if the
box were not there.

### The Ignore Pattern with Double Dot

When destructuring structs, you can use `..` to ignore fields you do not care
about:

```rust
struct Config {
    name: String,
    debug: bool,
    max_retries: u32,
    timeout_ms: u64,
}

fn summary(config: &Config) -> String {
    let Config { name, debug, .. } = config;
    format!("{name}: debug={debug}")
}

fn run(config: &Config) {
    println!("{}", summary(config));
    println!("retries: {}, timeout: {}ms", config.max_retries, config.timeout_ms);
}

fn main() {
    let config = Config {
        name: String::from("production"),
        debug: false,
        max_retries: 3,
        timeout_ms: 5000,
    };

    run(&config);
}
```

Output:

```
production: debug=false
retries: 3, timeout: 5000ms
```

The `..` says "I do not care about the remaining fields." Without it, you would
have to list every field — even the ones you are ignoring. This is especially
valuable when structs have many fields and you only need a few.

### Binding While Matching with At

A range pattern like `200..=299` tests whether a value falls within a range, but
it does not give you a name to use in the arm body. The `@` (read "at") operator
solves this — it tests a pattern _and_ binds the entire matched value to a
variable:

```rust
fn describe_code(code: u16) {
    match code {
        c @ 200..=299 => println!("{c}: success"),
        c @ 400..=499 => println!("{c}: client error"),
        c @ 500..=599 => println!("{c}: server error"),
        c => println!("{c}: other"),
    }
}

fn main() {
    describe_code(200);
    describe_code(404);
    describe_code(503);
    describe_code(301);
}
```

Output:

```
200: success
404: client error
503: server error
301: other
```

Without `@`, you would need a match guard to achieve the same thing:
`c if (200..=299).contains(&c)`. The `@` operator is more concise and expresses
the intent directly — "bind this value, but only if it matches this pattern."

The `@` operator works anywhere in a pattern, including inside enum variants.
This is especially useful for validating data carried by a variant:

```rust
enum Command {
    SetVolume(u32),
    SetBrightness(u32),
    Quit,
}

fn handle(cmd: &Command) {
    match cmd {
        Command::SetVolume(v @ 0..=100) => println!("volume: {v}%"),
        Command::SetVolume(v) => println!("volume {v} is out of range (0-100)"),
        Command::SetBrightness(b @ 0..=100) => println!("brightness: {b}%"),
        Command::SetBrightness(b) => println!("brightness {b} is out of range (0-100)"),
        Command::Quit => println!("goodbye"),
    }
}

fn main() {
    handle(&Command::SetVolume(75));
    handle(&Command::SetVolume(150));
    handle(&Command::SetBrightness(30));
    handle(&Command::Quit);
}
```

Output:

```
volume: 75%
volume 150 is out of range (0-100)
brightness: 30%
goodbye
```

The first `SetVolume` arm matches only values in the valid range and binds to
`v`. The second arm catches everything else — values outside the range — with a
plain variable binding. The compiler checks arms top to bottom, so the more
specific `@` pattern goes first.

## Putting It All Together

Here is a complete example that combines enum variants with data, exhaustive
matching, destructuring, match guards, and let chains:

```rust
use std::fmt;

#[derive(Debug)]
enum Priority {
    Low,
    Medium,
    High,
    Critical,
}

#[derive(Debug)]
enum Ticket {
    Bug { title: String, priority: Priority },
    Feature { title: String, votes: u32 },
    Chore(String),
}

impl fmt::Display for Ticket {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            Ticket::Bug { title, priority } => {
                write!(f, "[BUG:{priority:?}] {title}")
            }
            Ticket::Feature { title, votes } => {
                write!(f, "[FEATURE] {title} ({votes} votes)")
            }
            Ticket::Chore(title) => write!(f, "[CHORE] {title}"),
        }
    }
}

fn should_prioritize(ticket: &Ticket) -> bool {
    match ticket {
        Ticket::Bug { priority: Priority::Critical | Priority::High, .. } => true,
        Ticket::Feature { votes, .. } if *votes > 100 => true,
        _ => false,
    }
}

fn process_tickets(tickets: &[Ticket]) {
    for ticket in tickets {
        let label = if should_prioritize(ticket) {
            ">>> PRIORITY"
        } else {
            "    normal"
        };

        println!("{label}: {ticket}");
    }
}

fn main() {
    let tickets = vec![
        Ticket::Bug {
            title: String::from("login page crash"),
            priority: Priority::Critical,
        },
        Ticket::Feature {
            title: String::from("dark mode"),
            votes: 250,
        },
        Ticket::Chore(String::from("update dependencies")),
        Ticket::Bug {
            title: String::from("slow query"),
            priority: Priority::Medium,
        },
        Ticket::Bug {
            title: String::from("typo in footer"),
            priority: Priority::Low,
        },
        Ticket::Feature {
            title: String::from("export to PDF"),
            votes: 12,
        },
        Ticket::Bug {
            title: String::from("data loss on save"),
            priority: Priority::High,
        },
    ];

    process_tickets(&tickets);

    // Let chain: find the first critical bug
    if let Some(Ticket::Bug { title, priority: Priority::Critical }) =
        tickets.iter().find(|t| matches!(t, Ticket::Bug { priority: Priority::Critical, .. }))
    {
        println!("\nCritical bug found: {title}");
    }
}
```

Output:

```
>>> PRIORITY: [BUG:Critical] login page crash
>>> PRIORITY: [FEATURE] dark mode (250 votes)
    normal: [CHORE] update dependencies
    normal: [BUG:Medium] slow query
    normal: [BUG:Low] typo in footer
    normal: [FEATURE] export to PDF (12 votes)
>>> PRIORITY: [BUG:High] data loss on save

Critical bug found: login page crash
```

This example demonstrates several key techniques:

- Enum variants with different data shapes (`Bug` has named fields, `Feature`
  has named fields with different types, `Chore` has a single unnamed field).
- Nested pattern matching: `Priority::Critical | Priority::High` inside a
  `Ticket::Bug` pattern.
- Match guards: `if *votes > 100` to add conditions beyond what patterns
  express.
- The `matches!` macro: a convenient way to check if a value matches a pattern
  without extracting data — it returns `true` or `false`.
- Let chains with `if let` and `Some` to combine iterator results with pattern
  matching.
- `Display` implementation on an enum, matching each variant to format it
  differently.

## Why This Matters

Enums and pattern matching together form one of Rust's most powerful ideas: you
define all the possible states a value can be in, and the compiler _guarantees_
you handle each one. This is fundamentally different from representing states
with strings, integers, or boolean flags — approaches where the compiler cannot
help you.

When you add a new variant to an enum, every `match` without a wildcard arm
becomes a compile error, pointing you to exactly the code that needs updating.
When you forget to handle a case, the compiler catches it before your code ever
runs. This is the type system working _for_ you — turning runtime bugs into
compile-time errors.

Key points from this chapter:

- **Enums** define types with multiple variants. Each variant can carry different
  data — tuples, named fields, or nothing at all.
- **`impl` blocks** on enums work just like on structs. Use `match self` inside
  methods to handle each variant.
- **`match`** is exhaustive: the compiler requires you to handle every variant.
  Every arm must return the same type.
- **`if let`** is concise shorthand for matching a single pattern. Use it when
  you only care about one variant.
- **`let`-`else`** is the guard clause complement to `if let` — bind the value
  and bail out early if the pattern fails, keeping the happy path flat.
- **Let chains** (Rust 2024) combine `let` bindings with `&&` boolean tests in
  a single `if` or `while` condition, eliminating nested blocks.
- **Destructuring** pulls apart enums, structs, and tuples in patterns. Use
  `..` to ignore fields you do not need.
- **The `@` operator** binds a value to a variable while simultaneously testing
  it against a pattern — useful for ranges and validated data inside variants.
- **Match guards** (`if condition` after a pattern) add boolean tests that
  patterns alone cannot express.
