# Modules and Project Structure

A single file works fine for small programs, but real software grows. A web
server has routing, database access, authentication, and serialization. A CLI
tool has argument parsing, business logic, and output formatting. Without
organization, a thousand-line file becomes a maze where every change risks
breaking something unrelated.

Rust's answer is the _module system_: a way to split code into namespaced units
with explicit visibility boundaries. Modules let you decide _what_ is public
and _what_ is hidden — not by convention or documentation, but enforced by the
compiler. If a function is private, no code outside its module can call it.
Period.

This chapter covers how to organize code as projects grow: the `mod` and `use`
keywords, visibility with `pub`, file layout conventions, the structure of
_Cargo.toml_, and workspaces for multi-crate projects.

## Modules Define Boundaries

A _module_ is a named container for functions, types, constants, and other
modules. You create one with the `mod` keyword:

```rust
mod math {
    pub fn square(x: i32) -> i32 {
        x * x
    }

    fn is_even(x: i32) -> bool {
        x % 2 == 0
    }

    pub fn describe(x: i32) -> &'static str {
        if is_even(x) { "even" } else { "odd" }
    }
}

fn main() {
    println!("3 squared = {}", math::square(3));
    println!("4 is {}", math::describe(4));

    // math::is_even(4); // ERROR: `is_even` is private
}
```

Output:

```
3 squared = 9
4 is even
```

`square` and `describe` are marked `pub`, so code outside the module can call
them. `is_even` has no `pub` — it is _private by default_. Code within `math`
can call `is_even` freely (as `describe` does), but code outside the module
cannot. This is Rust's philosophy: everything is hidden unless you explicitly
choose to expose it. Visibility is not an afterthought; it is a design decision
you make for every item.

The path `math::square` uses `::` to navigate the module tree, just like `/` in
a filesystem path. Every item in Rust lives at a path, and modules are the
directories.

## Bringing Names into Scope with Use

Typing full paths everywhere gets verbose. The `use` keyword imports a name
into the current scope:

```rust
mod geometry {
    pub fn circle_area(radius: f64) -> f64 {
        std::f64::consts::PI * radius * radius
    }

    pub fn rectangle_area(width: f64, height: f64) -> f64 {
        width * height
    }
}

use geometry::circle_area;

fn main() {
    // Imported name — no prefix needed
    println!("circle: {:.2}", circle_area(5.0));

    // Full path still works too
    println!("rectangle: {:.2}", geometry::rectangle_area(3.0, 4.0));
}
```

Output:

```
circle: 78.54
rectangle: 12.00
```

### What to Import

Rust has a convention about what to `use`:

- **Functions**: import the _parent module_, then call with one level of
  qualification. This makes it clear the function comes from somewhere else:
  `geometry::rectangle_area(3.0, 4.0)`.
- **Types** (structs, enums): import the type directly. Writing `HashMap::new()`
  is clear enough — you do not need `collections::HashMap::new()`.
- **When names collide**: keep the parent module for disambiguation, or rename
  with `as`.

```rust
use std::collections::HashMap;
use std::collections::HashSet;

// Renaming to avoid collision
use std::fmt::Result as FmtResult;
use std::io::Result as IoResult;

fn main() {
    let _map: HashMap<&str, i32> = HashMap::new();
    let _set: HashSet<i32> = HashSet::new();

    // Use the renamed types
    let _fmt: FmtResult = Ok(());
    let _io: IoResult<()> = Ok(());

    println!("types imported successfully");
}
```

Output:

```
types imported successfully
```

### Grouping Imports

When importing multiple items from the same module, use curly braces to group
them:

```rust
use std::collections::{HashMap, HashSet};
use std::io::{self, Write};

fn main() {
    let mut data = HashMap::new();
    data.insert("language", "Rust");

    let tags: HashSet<&str> = HashSet::from(["fast", "safe"]);

    // io::stdout() uses the `self` import
    let mut out = io::stdout();
    writeln!(out, "data: {data:?}").unwrap();
    writeln!(out, "tags: {tags:?}").unwrap();
}
```

Output (tag order may vary):

```
data: {"language": "Rust"}
tags: {"safe", "fast"}
```

The `self` keyword in `use std::io::{self, Write}` imports both the `io`
module itself and the `Write` trait from it. This is a common pattern when you
need both the module namespace and specific items from it.

## Visibility in Depth

Rust gives you fine-grained control over what is visible and where. The default
is private — visible only within the current module and its children.

### Struct Field Visibility

Marking a struct `pub` makes the _type_ visible, but its fields remain private
unless individually marked:

```rust
mod account {
    pub struct User {
        pub name: String,
        pub(crate) email: String,  // visible within this crate only
        login_count: u32,          // private to this module
    }

    impl User {
        pub fn new(name: &str, email: &str) -> Self {
            Self {
                name: String::from(name),
                email: String::from(email),
                login_count: 0,
            }
        }

        pub fn login_count(&self) -> u32 {
            self.login_count
        }

        pub fn record_login(&mut self) {
            self.login_count += 1;
        }
    }
}

fn main() {
    let mut user = account::User::new("Alice", "alice@example.com");
    user.record_login();
    user.record_login();

    // pub(crate) fields are accessible within the same crate
    println!("{} ({}) has logged in {} times",
        user.name, user.email, user.login_count());

    // user.login_count = 100; // ERROR: field `login_count` is private
}
```

Output:

```
Alice (alice@example.com) has logged in 2 times
```

This is the idiomatic pattern: expose fields that are safe for direct access,
and hide fields that maintain invariants. The `login_count` can only be changed
through `record_login`, which ensures it only ever increases. The compiler
enforces this — no discipline required.

### The Visibility Spectrum

Rust provides several levels of visibility between fully private and fully
public:

| Modifier | Visible To |
|---|---|
| _(no modifier)_ | Current module and its children |
| `pub(super)` | Parent module |
| `pub(crate)` | Anywhere within the current crate |
| `pub` | Everyone, including external crates |

`pub(crate)` is the most commonly used restricted visibility. It makes an item
available throughout your crate — for internal coordination between modules —
without exposing it to users of your library.

```rust
mod internal {
    // Only this crate can call this function
    pub(crate) fn helper() -> &'static str {
        "internal helper"
    }
}

fn main() {
    // Works: we are in the same crate
    println!("{}", internal::helper());
}
```

Output:

```
internal helper
```

### Re-exports with Pub Use

Sometimes your module structure is an implementation detail. You want users of
your library to write `my_crate::Config`, not
`my_crate::internal::settings::Config`. Re-exports solve this:

```rust
mod config {
    pub struct Settings {
        pub max_retries: u32,
        pub timeout_secs: u64,
    }

    impl Settings {
        pub fn new() -> Self {
            Self {
                max_retries: 3,
                timeout_secs: 30,
            }
        }
    }
}

// Re-export: users see Settings at the top level
pub use config::Settings;

fn main() {
    // No need to know about the `config` module
    let s = Settings::new();
    println!("retries: {}, timeout: {}s", s.max_retries, s.timeout_secs);
}
```

Output:

```
retries: 3, timeout: 30s
```

`pub use` brings an item into the current module's public API. The `config`
module can remain private — it is an organizational detail, not part of your
public interface. This pattern is how the standard library keeps its public API
clean despite having a deep internal module hierarchy.

## Splitting Code into Files

Inline modules work for small programs, but production code lives in separate
files. In Rust, each file is a module. The file's _path_ determines its place
in the module tree.

### The Two-File Convention

When you write `mod math;` (with a semicolon instead of curly braces), the
compiler looks for the module's contents in a file:

```
src/
  main.rs       ← contains: mod math;
  math.rs       ← the contents of the math module
```

_src/main.rs:_

```rust,ignore
mod math;

fn main() {
    println!("5 squared = {}", math::square(5));
}
```

_src/math.rs:_

```rust,ignore
pub fn square(x: i32) -> i32 {
    x * x
}
```

The `mod math;` declaration tells the compiler: "there is a module named `math`;
find its source in _math.rs_." The file name _is_ the module name.

### Submodules and Directories

When a module has its own submodules, it gets a directory. The file for the
parent module sits next to the directory:

```
src/
  main.rs           ← mod network;
  network.rs        ← mod client; mod server;
  network/
    client.rs       ← contents of network::client
    server.rs       ← contents of network::server
```

_src/main.rs:_

```rust,ignore
mod network;

fn main() {
    network::client::connect();
    network::server::listen();
}
```

_src/network.rs:_

```rust,ignore
pub mod client;
pub mod server;
```

_src/network/client.rs:_

```rust,ignore
pub fn connect() {
    println!("client connected");
}
```

_src/network/server.rs:_

```rust,ignore
pub fn listen() {
    println!("server listening");
}
```

The pattern is consistent: _network.rs_ declares submodules with `pub mod`, and
the compiler looks for their source files inside the _network/_ directory.
Submodules of submodules follow the same rule, nesting directories as deep as
you need.

An older convention places the parent module's code in a file named _mod.rs_
inside the directory (_network/mod.rs_ instead of _network.rs_). Both styles
work, but you cannot use both for the same module. The named-file style shown
here is the modern convention — it avoids having many files all named _mod.rs_
in your editor's tab bar.

## Library and Binary Crates

Cargo recognizes two kinds of _crate_ (compilation unit) in a single package:

- **Library crate** (_src/lib.rs_): code meant to be used by other crates.
  A package can have at most one library crate.
- **Binary crate** (_src/main.rs_): an executable with a `fn main()`.
  A package can have any number of binary crates.

### A Library Crate

When your project is primarily a library, _src/lib.rs_ is the crate root:

```
my_library/
  Cargo.toml
  src/
    lib.rs      ← crate root
    utils.rs    ← mod utils;
```

Everything `pub` in _lib.rs_ (or re-exported through it) becomes the library's
public API. Users of your library access it with `use my_library::SomeType;`.

### Library Plus Binary

A common pattern is to put business logic in a library crate and keep
_main.rs_ as a thin entry point:

```
my_project/
  Cargo.toml
  src/
    lib.rs       ← library crate: the real logic
    main.rs      ← binary crate: calls into the library
```

The binary crate uses the library by its package name. If _Cargo.toml_ says
`name = "my_project"`, then:

_src/lib.rs:_

```rust,ignore
pub fn greet(name: &str) -> String {
    format!("Hello, {name}!")
}
```

_src/main.rs:_

```rust,ignore
use my_project::greet;

fn main() {
    println!("{}", greet("world"));
}
```

This separation has a practical benefit: integration tests (in the _tests/_
directory) can only access your library crate's public API, not the binary.
Putting logic in the library means it is testable by design.

### Multiple Binaries

Additional binaries live in _src/bin/_:

```
my_project/
  Cargo.toml
  src/
    lib.rs
    main.rs           ← default binary
    bin/
      tool.rs          ← cargo run --bin tool
      migrate.rs       ← cargo run --bin migrate
```

Each file in _src/bin/_ becomes a separate binary with its own `fn main()`.
Run them with **`cargo run --bin tool`** or **`cargo run --bin migrate`**.

## Cargo.toml in Depth

_Cargo.toml_ is more than a dependency list. It defines your package's
identity, edition, dependencies, and quality standards.

### Package Metadata

Every _Cargo.toml_ starts with a `[package]` table that identifies your crate:

```toml
[package]
name = "my-project"
version = "0.1.0"
edition = "2024"
rust-version = "1.85"
```

- **`name`**: the crate name. Hyphens in the name become underscores in code:
  `my-project` is imported as `use my_project::...`.
- **`version`**: follows semantic versioning (major.minor.patch).
- **`edition`**: the Rust edition. `"2024"` targets the latest edition
  (requires Rust 1.85+). Every project created with **`cargo new`** defaults
  to the current edition.
- **`rust-version`**: the minimum Rust version your crate supports. The
  resolver uses this to avoid pulling dependencies that require a newer
  compiler.

### Dependencies

Cargo separates dependencies by how they are used:

```toml
[dependencies]
serde = { version = "1.0", features = ["derive"] }
anyhow = "1"

[dev-dependencies]
assert_cmd = "2"

[build-dependencies]
cc = "1"
```

- **`[dependencies]`**: crates your code needs to compile and run.
- **`[dev-dependencies]`**: crates needed only for tests, examples, and
  benchmarks. Not included when someone depends on your library.
- **`[build-dependencies]`**: crates used by a build script (_build.rs_).

Version requirements follow Cargo's version syntax: `"1.0"` means "any 1.x.y
that is >= 1.0.0". `"=1.4.3"` pins to an exact version. `">=0.5, <0.8"` uses a
range.

### The Lints Section

You can configure compiler and Clippy lints directly in _Cargo.toml_ instead of
scattering `#[allow(...)]` and `#[deny(...)]` across your code:

```toml
[lints.rust]
unsafe_code = "forbid"

[lints.clippy]
pedantic = { level = "warn", priority = -1 }
must_use_candidate = "allow"
```

The `priority` field controls ordering when lint groups and individual lints
overlap. A lower priority is overridden by a higher one. In the example above,
`pedantic` (a broad group) has priority -1, so the individual
`must_use_candidate = "allow"` (default priority 0) takes precedence.

### The Resolver

Setting `edition = "2024"` implies `resolver = "3"`. The v3 resolver is
_MSRV-aware_: it reads the `rust-version` field and avoids selecting dependency
versions that require a newer compiler than your stated minimum. You do not
need to write `resolver = "3"` explicitly — it is the default for all 2024
edition packages.

## Cargo Manages Its Own Cache

Every crate you depend on gets downloaded, extracted, and stored under
_~/.cargo/_. Over months of development, this directory grows — registry
archives, extracted source trees, git clones, and checkout directories
accumulate silently. Before Rust 1.88, cleaning this was entirely your
responsibility. Most developers never did, and _~/.cargo/_ quietly consumed
gigabytes of disk space.

Cargo now cleans up after itself automatically.

### How It Works

Starting with Rust 1.78, Cargo silently records a timestamp every time it reads
or writes a cached file. Starting with Rust 1.88, Cargo checks these timestamps
once per day and deletes files that have not been accessed recently:

| Cached Data | Location | Default Retention |
|---|---|---|
| Compressed `.crate` archives | _registry/cache/_ | 3 months |
| Extracted source trees | _registry/src/_ | 1 month |

The logic is intuitive: files that require a network download to replace are
kept longer (3 months). Files that Cargo can regenerate locally from the
archive are kept shorter (1 month). If you run **`cargo build`** and a deleted
source tree is needed again, Cargo simply re-extracts it from the archive — or
re-downloads the archive if that was also cleaned.

Cleanup runs automatically during normal commands like **`cargo build`** and
**`cargo fetch`**. It never runs with `--offline` or `--frozen`, since those
modes depend on the cache being intact. Errors during cleanup produce warnings
but never abort your build.

### Configuring the Frequency

The cleanup frequency is controlled in _~/.cargo/config.toml_:

```toml
[cache]
auto-clean-frequency = "1 day"
```

The default is `"1 day"` — Cargo checks at most once per day. Other accepted
values:

| Value | Meaning |
|---|---|
| `"never"` | Disable automatic cleanup entirely |
| `"always"` | Check on every Cargo invocation |
| `"1 day"` | Check at most once per day (default) |
| `"7 days"`, `"2 weeks"` | Any integer plus a time unit |

You can also set this through the environment variable
`CARGO_CACHE_AUTO_CLEAN_FREQUENCY`.

**When to disable cleanup.** If your CI pipeline caches _~/.cargo/_ between
runs, restored files may have old timestamps and could be deleted prematurely.
Set `auto-clean-frequency = "never"` in your CI configuration, or run
**`cargo fetch`** immediately after restoring the cache to refresh timestamps.

## Workspaces for Multi-Crate Projects

As projects grow, splitting into multiple crates provides faster compilation
(only changed crates recompile), clearer boundaries, and independent
versioning. A _workspace_ groups related crates under a single _Cargo.lock_
and a shared _target/_ directory.

### Setting Up a Workspace

A workspace is defined by a _Cargo.toml_ at the project root:

```toml
[workspace]
members = [
    "crates/core",
    "crates/cli",
    "crates/api",
]
```

The directory layout:

```
my-workspace/
  Cargo.toml        ← workspace root
  Cargo.lock        ← shared lock file
  crates/
    core/
      Cargo.toml
      src/lib.rs
    cli/
      Cargo.toml
      src/main.rs
    api/
      Cargo.toml
      src/main.rs
```

All crates in the workspace share a single _target/_ directory, so a dependency
compiled for one member is reused by the others. **`cargo build`** at the
workspace root builds everything. **`cargo test`** runs all tests across all
members. **`cargo run -p cli`** runs a specific binary.

### Shared Dependencies

Duplicate dependency versions across workspace members lead to drift. The
`[workspace.dependencies]` table defines versions once:

```toml
# Workspace Cargo.toml
[workspace]
members = ["crates/core", "crates/cli"]

[workspace.dependencies]
serde = { version = "1.0", features = ["derive"] }
anyhow = "1"
core-lib = { path = "crates/core" }
```

Members inherit with `workspace = true`:

```toml
# crates/cli/Cargo.toml
[package]
name = "cli"
version = "0.1.0"
edition = "2024"

[dependencies]
serde.workspace = true
anyhow.workspace = true
core-lib.workspace = true
```

Members can add extra features to an inherited dependency but cannot remove
features that the workspace definition includes. In the 2024 edition, writing
`default-features = false` in a member when the workspace does not also set
it is a hard error — Cargo now catches this inconsistency instead of silently
ignoring it.

### Shared Lint Configuration

Just like dependencies, lint policy can be defined once:

```toml
# Workspace Cargo.toml
[workspace.lints.rust]
unsafe_code = "forbid"

[workspace.lints.clippy]
pedantic = { level = "warn", priority = -1 }
```

Members opt in:

```toml
# crates/core/Cargo.toml
[lints]
workspace = true
```

This keeps your entire project consistent without duplicating lint
configuration in every crate.

## Choosing and Evaluating Crates

Rust's standard library is deliberately small. For most tasks beyond the
basics — HTTP, serialization, logging, async — you will reach for third-party
crates. Here is how to evaluate them:

- **docs.rs**: Every published crate has automatically generated documentation
  at _docs.rs/crate-name_. Read the API docs before committing.
- **crates.io**: The official registry. Check download counts and recent
  releases. A crate with millions of downloads and monthly releases is
  actively maintained.
- **lib.rs**: An alternative discovery site with categorization and popularity
  rankings.
- **Repository activity**: Look at the GitHub/GitLab repository. Recent
  commits, responsive issue triage, and a clear changelog are positive
  signals.
- **Dependency count**: Fewer dependencies mean a smaller attack surface and
  faster compilation. Prefer crates that are focused and lean.

For the most common tasks, the ecosystem has strong defaults:

| Task | Crate |
|---|---|
| Serialization | serde |
| Error handling (libraries) | thiserror |
| Error handling (applications) | anyhow |
| HTTP client | reqwest |
| Web framework | axum |
| CLI argument parsing | clap |
| Async runtime | tokio |
| Logging | tracing |

You do not need to memorize this list. As you build real projects, the right
crates will become familiar quickly.

## Putting It All Together

Here is a complete single-file example that demonstrates module organization,
visibility, and re-exports. In a real project these modules would live in
separate files, but the structure is identical:

```rust
mod storage {
    use std::collections::HashMap;

    pub struct KeyValueStore {
        data: HashMap<String, String>,
    }

    impl KeyValueStore {
        pub fn new() -> Self {
            Self {
                data: HashMap::new(),
            }
        }

        pub fn set(&mut self, key: &str, value: &str) {
            self.data.insert(String::from(key), String::from(value));
        }

        pub fn get(&self, key: &str) -> Option<&str> {
            self.data.get(key).map(|v| v.as_str())
        }

        pub fn keys(&self) -> Vec<&str> {
            let mut keys: Vec<&str> = self.data.keys().map(|k| k.as_str()).collect();
            keys.sort();
            keys
        }

        pub(crate) fn len(&self) -> usize {
            self.data.len()
        }
    }
}

mod commands {
    use super::storage::KeyValueStore;

    pub fn populate(store: &mut KeyValueStore) {
        store.set("language", "Rust");
        store.set("edition", "2024");
        store.set("level", "beginner-friendly");
    }

    pub fn display(store: &KeyValueStore) {
        for key in store.keys() {
            if let Some(value) = store.get(key) {
                println!("  {key} = {value}");
            }
        }
    }
}

// Re-export so users do not need to know about the `storage` module
pub use storage::KeyValueStore;

fn main() {
    let mut store = KeyValueStore::new();
    commands::populate(&mut store);

    println!("store contents ({} entries):", store.len());
    commands::display(&store);

    store.set("status", "complete");
    println!("\nafter update ({} entries):", store.len());
    commands::display(&store);
}
```

Output:

```
store contents (3 entries):
  edition = 2024
  language = Rust
  level = beginner-friendly

after update (4 entries):
  edition = 2024
  language = Rust
  level = beginner-friendly
  status = complete
```

This example demonstrates the key patterns from this chapter:

- **`mod` for organization.** `storage` handles data; `commands` handles
  operations. Each module has a clear responsibility.
- **`pub` for the public API.** `KeyValueStore` and its methods are public.
  The internal `data` field is private — callers interact through methods.
- **`pub(crate)` for internal sharing.** The `len` method is available within
  the crate but would not be visible to external crate users.
- **`super` for upward navigation.** The `commands` module uses
  `super::storage::KeyValueStore` to reference a sibling module's type.
- **`pub use` for a clean API.** `KeyValueStore` is re-exported at the crate
  root, so `main` can use it directly without knowing about the `storage`
  module.

## Why This Matters

Modules are not just about organizing files — they are about defining
boundaries. Every `pub` is a promise: "this is part of my API, and I will
maintain it." Every private item is freedom: "I can change this without
breaking anyone."

The key ideas from this chapter:

- **Private by default.** Items are visible only within their module unless
  marked `pub`. This forces you to think about what you expose.
- **`mod` declares, `use` imports.** `mod math;` creates a module from a file.
  `use math::square;` brings a name into scope.
- **File layout follows the module tree.** _network.rs_ plus _network/_
  directory for submodules. One module per file, one file per module.
- **Library crate for logic, binary crate for the entry point.** Put your
  real code in _src/lib.rs_ so it is testable and reusable.
- **_Cargo.toml_ is your project's identity.** Edition, dependencies, lint
  configuration, and minimum Rust version — all in one file.
- **Workspaces for multi-crate projects.** Shared dependencies, shared
  lints, and a single _Cargo.lock_ keep everything consistent.
- **Cargo cleans its own cache.** Downloaded crates and extracted sources are
  automatically deleted when unused, so _~/.cargo/_ stays manageable.
- **`pub use` creates a clean public API.** Internal module structure is an
  implementation detail; re-exports let you present the interface you want.
