# Testing as a First-Class Citizen

Most languages treat testing as an afterthought — an external framework you
install, configure, and wire up before you can write your first test. Rust takes
the opposite approach. Testing is built into the language and toolchain from the
ground up. The `#[test]` attribute, the `assert!` family of macros, and the
**`cargo test`** command are all part of the standard distribution. No extra
dependencies, no configuration files, no setup ceremony.

This design choice reflects a deeper philosophy: _if something matters, it
should be easy_. Writing a test should be as natural as writing a function. And
in Rust, it is — a test _is_ a function, annotated with a single attribute.
When testing is effortless, you write more tests. When you write more tests, you
catch more bugs. When you catch more bugs at compile time and test time, fewer
reach production.

This chapter covers the three kinds of tests Rust supports — unit tests,
integration tests, and documentation tests — along with the **`cargo test`**
workflows that tie them together.

## Unit Tests Live with Your Code

A unit test in Rust is a function annotated with `#[test]` inside a module
annotated with `#[cfg(test)]`. The `cfg(test)` attribute tells the compiler:
"only compile this module when running tests." Your test code adds zero bytes to
your release binary.

```rust
fn celsius_to_fahrenheit(c: f64) -> f64 {
    c * 1.8 + 32.0
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn freezing_point() {
        assert_eq!(celsius_to_fahrenheit(0.0), 32.0);
    }

    #[test]
    fn boiling_point() {
        assert_eq!(celsius_to_fahrenheit(100.0), 212.0);
    }

    #[test]
    fn body_temperature() {
        let result = celsius_to_fahrenheit(37.0);
        assert!((result - 98.6).abs() < 0.01, "expected ~98.6, got {result}");
    }
}
```

Output from **`cargo test`**:

```
running 3 tests
test tests::boiling_point ... ok
test tests::body_temperature ... ok
test tests::freezing_point ... ok

test result: ok. 3 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out
```

Three things to notice:

- **`use super::*`** imports everything from the parent module. Tests live
  inside a child module, so they need this import to access the functions they
  are testing. Because tests are inside the same crate, they can test private
  functions — no need to make something `pub` just to test it.
- **`#[test]`** marks a function as a test case. The test runner discovers these
  automatically — no registration step.
- **`#[cfg(test)]`** on the module ensures the entire `tests` module is
  stripped from non-test builds. You can put helper functions, test fixtures,
  and even `use` statements inside it without affecting your production binary.

### The Assert Family

Rust provides three assertion macros for tests:

| Macro | Purpose | Requires |
|---|---|---|
| `assert!(expr)` | Checks that `expr` is `true` | — |
| `assert_eq!(left, right)` | Checks that `left == right` | `PartialEq + Debug` |
| `assert_ne!(left, right)` | Checks that `left != right` | `PartialEq + Debug` |

All three accept an optional message as additional arguments:

```rust
fn clamp(value: i32, min: i32, max: i32) -> i32 {
    if value < min {
        min
    } else if value > max {
        max
    } else {
        value
    }
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn clamp_within_range() {
        assert_eq!(clamp(5, 0, 10), 5);
    }

    #[test]
    fn clamp_below_minimum() {
        assert_eq!(clamp(-3, 0, 10), 0, "values below min should be clamped");
    }

    #[test]
    fn clamp_above_maximum() {
        let result = clamp(15, 0, 10);
        assert_ne!(result, 15, "value should have been clamped, got {result}");
        assert_eq!(result, 10);
    }
}
```

Output:

```
running 3 tests
test tests::clamp_above_maximum ... ok
test tests::clamp_below_minimum ... ok
test tests::clamp_within_range ... ok

test result: ok. 3 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out
```

When `assert_eq!` fails, it prints both values so you can see what went wrong.
This is why it requires `Debug` on both sides — the failure message uses `{:?}`
to format them. For your own types, `#[derive(Debug, PartialEq)]` is usually
all you need.

### Testing for Panics

Some functions are _supposed_ to panic under certain conditions. The
`#[should_panic]` attribute marks a test that is expected to panic:

```rust
fn divide(a: i32, b: i32) -> i32 {
    if b == 0 {
        panic!("cannot divide {a} by zero");
    }
    a / b
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn normal_division() {
        assert_eq!(divide(10, 2), 5);
    }

    #[test]
    #[should_panic(expected = "cannot divide")]
    fn division_by_zero_panics() {
        divide(10, 0);
    }
}
```

Output:

```
running 2 tests
test tests::division_by_zero_panics - should panic ... ok
test tests::normal_division ... ok

test result: ok. 2 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out
```

The `expected` parameter is a substring match against the panic message. Always
include it — without `expected`, _any_ panic makes the test pass, including
panics from unrelated bugs. With `expected`, only a panic whose message
contains that substring counts as success.

### Returning Result from Tests

Tests that call fallible operations can return `Result` instead of panicking.
This lets you use the `?` operator for concise error propagation:

```rust
use std::num::ParseIntError;

fn parse_pair(s: &str) -> Result<(i32, i32), ParseIntError> {
    let mut parts = s.split(',');
    let a: i32 = parts.next().unwrap_or("").trim().parse()?;
    let b: i32 = parts.next().unwrap_or("").trim().parse()?;
    Ok((a, b))
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn valid_pair() -> Result<(), ParseIntError> {
        let (a, b) = parse_pair("10, 20")?;
        assert_eq!(a, 10);
        assert_eq!(b, 20);
        Ok(())
    }

    #[test]
    fn invalid_input_returns_error() {
        let result = parse_pair("abc, 20");
        assert!(result.is_err());
    }
}
```

Output:

```
running 2 tests
test tests::invalid_input_returns_error ... ok
test tests::valid_pair ... ok

test result: ok. 2 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out
```

A `Result`-returning test fails if it returns `Err`. This is cleaner than
calling `.unwrap()` on every fallible operation — you get a meaningful error
message instead of a panic backtrace. For tests that need to handle multiple
error types, use `Box<dyn std::error::Error>` as the error type:

```rust
#[cfg(test)]
mod tests {
    #[test]
    fn multiple_error_types() -> Result<(), Box<dyn std::error::Error>> {
        let n: i32 = "42".parse()?;
        let content = std::fs::read_to_string("Cargo.toml")?;
        assert_eq!(n, 42);
        assert!(content.contains("[package]"));
        Ok(())
    }
}
```

Output:

```
running 1 test
test tests::multiple_error_types ... ok

test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out
```

### Ignoring Expensive Tests

Some tests are slow — they hit a database, make network requests, or process
large data sets. The `#[ignore]` attribute skips them during normal test runs:

```rust
fn fibonacci(n: u64) -> u64 {
    match n {
        0 => 0,
        1 => 1,
        _ => fibonacci(n - 1) + fibonacci(n - 2),
    }
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn small_fibonacci() {
        assert_eq!(fibonacci(10), 55);
    }

    #[test]
    #[ignore = "slow recursive calculation"]
    fn large_fibonacci() {
        assert_eq!(fibonacci(40), 102334155);
    }
}
```

Output from **`cargo test`**:

```
running 2 tests
test tests::large_fibonacci ... ignored
test tests::small_fibonacci ... ok

test result: ok. 1 passed; 0 failed; 1 ignored; 0 measured; 0 filtered out
```

Run ignored tests explicitly with **`cargo test -- --ignored`**. To run
_everything_ — both normal and ignored tests — use
**`cargo test -- --include-ignored`**.

The reason string after `#[ignore = "..."]` documents why the test is skipped.
It appears when listing tests and helps future readers understand the intent.

## Integration Tests

Unit tests verify individual functions and modules. _Integration tests_ verify
that your library's public API works correctly from the outside — exactly as a
user of your crate would call it.

Integration tests live in the _tests/_ directory at the project root, next to
_src/_:

```
my_project/
  Cargo.toml
  src/
    lib.rs
  tests/
    temperature_test.rs
```

Each file in _tests/_ is compiled as a separate crate that depends on your
library. This means integration tests can only access your public API — the
same items a downstream user would see.

_src/lib.rs:_

```rust,ignore
/// Converts Celsius to Fahrenheit.
pub fn celsius_to_fahrenheit(c: f64) -> f64 {
    c * 1.8 + 32.0
}

/// Converts Fahrenheit to Celsius.
pub fn fahrenheit_to_celsius(f: f64) -> f64 {
    (f - 32.0) / 1.8
}
```

_tests/temperature_test.rs:_

```rust,ignore
use my_project::{celsius_to_fahrenheit, fahrenheit_to_celsius};

#[test]
fn roundtrip_conversion() {
    let original = 37.0;
    let fahrenheit = celsius_to_fahrenheit(original);
    let back = fahrenheit_to_celsius(fahrenheit);
    assert!((back - original).abs() < 0.001, "roundtrip failed: {back}");
}

#[test]
fn known_conversions() {
    assert_eq!(celsius_to_fahrenheit(0.0), 32.0);
    assert_eq!(celsius_to_fahrenheit(100.0), 212.0);
    assert_eq!(fahrenheit_to_celsius(32.0), 0.0);
}
```

Notice: no `#[cfg(test)]`, no `mod tests`. Files in _tests/_ are always test
code — Cargo knows to compile them only when testing. Each file is its own
crate, so each one starts with `use` statements to import from your library.

### Sharing Test Helpers

When multiple integration test files need shared setup or helper functions,
place them in a subdirectory:

```
tests/
  temperature_test.rs
  helpers/
    mod.rs
```

_tests/helpers/mod.rs:_

```rust,ignore
pub fn approx_eq(a: f64, b: f64) -> bool {
    (a - b).abs() < 0.001
}
```

_tests/temperature_test.rs:_

```rust,ignore
mod helpers;

use helpers::approx_eq;
use my_project::celsius_to_fahrenheit;

#[test]
fn body_temperature() {
    assert!(approx_eq(celsius_to_fahrenheit(37.0), 98.6));
}
```

Files inside subdirectories of _tests/_ are not compiled as standalone test
crates — they will not generate empty test suite output. This is why helper
modules go in a subdirectory rather than at the top level of _tests/_.

### Binary Crates and Testing

Integration tests can only test library crates (_src/lib.rs_). If your project
is a binary-only crate (_src/main.rs_ without _src/lib.rs_), integration tests
have nothing to import.

This is why the lib-plus-binary pattern from the previous chapter matters for
testing: put your logic in _src/lib.rs_, keep _src/main.rs_ as a thin entry
point, and integration tests can exercise the logic through the library's
public API.

## Documentation Tests

Rust has a remarkable feature: code examples in your documentation comments are
compiled and run as tests. This means your examples can never silently go out
of date. If the API changes and an example breaks, **`cargo test`** catches it.

```rust
/// Calculates the factorial of a non-negative integer.
///
/// # Examples
///
/// ```
/// assert_eq!(factorial(0), 1);
/// assert_eq!(factorial(5), 120);
/// ```
///
/// # Panics
///
/// Panics if `n` is greater than 20 (overflow for `u64`).
///
/// ```should_panic
/// factorial(21); // overflows u64
/// ```
pub fn factorial(n: u64) -> u64 {
    match n {
        0 | 1 => 1,
        _ => n * factorial(n - 1),
    }
}

fn main() {
    println!("5! = {}", factorial(5));
    println!("10! = {}", factorial(10));
}
```

Output from running the program:

```
5! = 120
10! = 3628800
```

Output from **`cargo test`** (doc tests section):

```
running 2 tests
test src/lib.rs - factorial (line 5) ... ok
test src/lib.rs - factorial (line 13) ... ok

test result: ok. 2 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out
```

Every code block inside `///` comments is compiled and executed. If the code
does not compile or if an assertion fails, the test fails. This keeps
documentation honest — _forever_.

### Hiding Setup Lines

Sometimes a doc example needs setup code that would distract the reader. Lines
starting with `#` are compiled but hidden in the rendered documentation:

```rust
/// Sums a list of prices, applying a discount.
///
/// ```
/// # fn discount(price: f64) -> f64 { price * 0.9 }
/// let prices = vec![10.0, 20.0, 30.0];
/// let total: f64 = prices.iter().map(|&p| discount(p)).sum();
/// assert!((total - 54.0).abs() < 0.001);
/// ```
pub fn apply_discounts(prices: &[f64], rate: f64) -> Vec<f64> {
    prices.iter().map(|&p| p * (1.0 - rate)).collect()
}

fn main() {
    let discounted = apply_discounts(&[10.0, 20.0, 30.0], 0.1);
    println!("discounted: {discounted:?}");
}
```

Output:

```
discounted: [9.0, 18.0, 27.0]
```

The `# fn discount(...)` line is invisible in the rendered docs but present
when the test runs. Use hidden lines for imports, helper functions, and setup
that the reader does not need to see to understand the example.

### Code Block Annotations

Not every code block should run as a test. Rust provides annotations to control
how doc examples are handled:

| Annotation | Behavior |
|---|---|
| ` ```rust ` or ` ``` ` | Compiled and executed (default) |
| ` ```no_run ` | Compiled but not executed |
| ` ```compile_fail ` | Must fail to compile |
| ` ```should_panic ` | Must panic at runtime |
| ` ```ignore ` | Skipped entirely |
| ` ```text ` | Not treated as code |

```rust
/// Opens a configuration file.
///
/// Compiles but does not run during testing (no file on disk):
///
/// ```no_run
/// let config = std::fs::read_to_string("config.toml").unwrap();
/// println!("config length: {}", config.len());
/// ```
///
/// Demonstrates a compile-time error:
///
/// ```compile_fail
/// let x: i32 = "not a number";
/// ```
pub fn load_config(path: &str) -> Result<String, std::io::Error> {
    std::fs::read_to_string(path)
}

fn main() {
    match load_config("config.toml") {
        Ok(content) => println!("loaded {} bytes", content.len()),
        Err(e) => println!("no config file: {e}"),
    }
}
```

Output:

```
no config file: No such file or directory (os error 2)
```

`no_run` is perfect for examples that involve files, network, or other
resources that may not exist in the test environment. The example still
compiles — so API changes are caught — but it does not execute.

`compile_fail` is the opposite: it verifies that _incorrect_ code is
correctly rejected by the compiler. This is useful for documenting what
your API _prevents_.

### Combined Doctests in Rust 2024

In the 2024 edition, the compiler merges compatible doc tests into a single
binary instead of compiling each one separately. This dramatically speeds up
**`cargo test --doc`** for crates with many examples.

You do not need to change your code — the merging happens automatically. In
rare cases where a doc test depends on being compiled in isolation (for
example, because it inspects its own line numbers via
`std::panic::Location`), you can opt out with the `standalone_crate`
annotation:

```text
/// ```standalone_crate
/// // This test must be compiled separately
/// let loc = std::panic::Location::caller();
/// assert_eq!(loc.line(), 4);
/// ```
```

For the vast majority of doc tests, the default behavior is exactly what you
want — faster test runs with no changes required.

## Running Tests with Cargo Test

**`cargo test`** is the single command that runs all three kinds of tests:
unit tests, integration tests, and doc tests. Understanding its options lets
you run exactly the tests you need.

### Filtering Tests by Name

Pass a string to **`cargo test`** to filter tests by name. It matches any test
whose fully qualified name contains the string:

```rust
fn is_palindrome(s: &str) -> bool {
    let s: String = s.chars().filter(|c| c.is_alphanumeric()).collect();
    let s = s.to_lowercase();
    s == s.chars().rev().collect::<String>()
}

fn is_anagram(a: &str, b: &str) -> bool {
    let mut a: Vec<char> = a.to_lowercase().chars().collect();
    let mut b: Vec<char> = b.to_lowercase().chars().collect();
    a.sort();
    b.sort();
    a == b
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn palindrome_simple() {
        assert!(is_palindrome("racecar"));
    }

    #[test]
    fn palindrome_with_spaces() {
        assert!(is_palindrome("A man a plan a canal Panama"));
    }

    #[test]
    fn not_a_palindrome() {
        assert!(!is_palindrome("hello"));
    }

    #[test]
    fn anagram_basic() {
        assert!(is_anagram("listen", "silent"));
    }

    #[test]
    fn not_an_anagram() {
        assert!(!is_anagram("hello", "world"));
    }
}
```

test tests::palindrome_simple ... ok

### Common Cargo Test Options

Arguments before `--` go to Cargo. Arguments after `--` go to the test binary:

| Command | What It Does |
|---|---|
| **`cargo test`** | Run all tests |
| **`cargo test name`** | Run tests matching "name" |
| **`cargo test -- --show-output`** | Show `println!` from passing tests |
| **`cargo test -- --test-threads=1`** | Run tests sequentially |
| **`cargo test -- --ignored`** | Run only `#[ignore]` tests |
| **`cargo test -- --include-ignored`** | Run all tests, including ignored |
| **`cargo test --doc`** | Run only doc tests |
| **`cargo test --lib`** | Run only unit tests |
| **`cargo test --test name`** | Run a specific integration test file |
| **`cargo test --no-fail-fast`** | Continue after the first failure |
| **`cargo test -p crate_name`** | Test a specific workspace member |

By default, tests run in parallel using multiple threads. If your tests share
state (files, environment variables, a database), use `--test-threads=1` to
run them one at a time. Better yet, design your tests to be independent — tests
that can run in parallel finish faster.

## Putting It All Together

Here is a complete example that demonstrates unit tests, `Result`-returning
tests, `#[should_panic]`, `#[ignore]`, and doc tests — all in one file:

```rust
/// A simple calculator for integer arithmetic.
///
/// # Examples
///
/// ```
/// let result = Calculator::new()
///     .push(10)
///     .push(3)
///     .add();
/// assert_eq!(result.top(), Some(13));
/// ```
pub struct Calculator {
    stack: Vec<i64>,
}

impl Calculator {
    pub fn new() -> Self {
        Self { stack: Vec::new() }
    }

    pub fn push(mut self, value: i64) -> Self {
        self.stack.push(value);
        self
    }

    pub fn top(&self) -> Option<i64> {
        self.stack.last().copied()
    }

    pub fn add(mut self) -> Self {
        let b = self.stack.pop().expect("stack underflow");
        let a = self.stack.pop().expect("stack underflow");
        self.stack.push(a + b);
        self
    }

    pub fn subtract(mut self) -> Self {
        let b = self.stack.pop().expect("stack underflow");
        let a = self.stack.pop().expect("stack underflow");
        self.stack.push(a - b);
        self
    }

    pub fn multiply(mut self) -> Self {
        let b = self.stack.pop().expect("stack underflow");
        let a = self.stack.pop().expect("stack underflow");
        self.stack.push(a * b);
        self
    }

    pub fn stack_size(&self) -> usize {
        self.stack.len()
    }
}

fn main() {
    let calc = Calculator::new()
        .push(100)
        .push(58)
        .subtract()
        .push(20)
        .add();

    println!("result: {}", calc.top().unwrap());
    println!("stack size: {}", calc.stack_size());
}

#[cfg(test)]
mod tests {
    use super::*;

    // --- Basic operations ---

    #[test]
    fn addition() {
        let calc = Calculator::new().push(2).push(3).add();
        assert_eq!(calc.top(), Some(5));
    }

    #[test]
    fn subtraction() {
        let calc = Calculator::new().push(10).push(4).subtract();
        assert_eq!(calc.top(), Some(6));
    }

    #[test]
    fn multiplication() {
        let calc = Calculator::new().push(7).push(6).multiply();
        assert_eq!(calc.top(), Some(42));
    }

    // --- Chained operations ---

    #[test]
    fn chained_operations() -> Result<(), String> {
        let calc = Calculator::new()
            .push(5)
            .push(3)
            .add()
            .push(2)
            .multiply();

        let result = calc.top().ok_or("empty stack")?;
        assert_eq!(result, 16); // (5 + 3) * 2
        Ok(())
    }

    // --- Edge cases ---

    #[test]
    fn empty_stack_returns_none() {
        let calc = Calculator::new();
        assert_eq!(calc.top(), None);
    }

    #[test]
    fn single_value() {
        let calc = Calculator::new().push(42);
        assert_eq!(calc.top(), Some(42));
        assert_eq!(calc.stack_size(), 1);
    }

    // --- Panic behavior ---

    #[test]
    #[should_panic(expected = "stack underflow")]
    fn add_with_empty_stack_panics() {
        Calculator::new().add();
    }

    #[test]
    #[should_panic(expected = "stack underflow")]
    fn add_with_one_element_panics() {
        Calculator::new().push(1).add();
    }

    // --- Ignored (slow) tests ---

    #[test]
    #[ignore = "stress test with many operations"]
    fn many_operations() {
        let mut calc = Calculator::new();
        for i in 1..=1000 {
            calc = calc.push(i);
        }
        assert_eq!(calc.stack_size(), 1000);

        // Sum them all: push pairs and add
        for _ in 0..999 {
            calc = calc.add();
        }
        assert_eq!(calc.top(), Some(500500)); // sum of 1..=1000
    }
}
```

```

Output from **`cargo test`**:

```

This example demonstrates the key patterns from this chapter:

- **Unit tests in `#[cfg(test)] mod tests`.** They live alongside the code
  they test and can access private fields (like `stack`) through
  `use super::*`.
- **`assert_eq!` for value comparisons.** Clear, concise, and the failure
  message shows both values automatically.
- **`Result`-returning tests.** The `chained_operations` test uses `?` to
  propagate errors instead of calling `.unwrap()`.
- **`#[should_panic(expected = "...")]` for panic tests.** Both underflow tests
  pin the expected panic message to avoid false positives.
- **`#[ignore]` for slow tests.** The stress test runs only when explicitly
  requested with **`cargo test -- --ignored`**.
- **Doc tests on the struct.** The `Calculator` documentation includes a
  working example that is compiled and tested automatically.

## Why This Matters

Testing in Rust is not a chore you do after writing code — it is a tool you
use _while_ writing code. The design makes this natural: tests live next to the
code they test, assertions are built into the language, and documentation
examples are real tests. The barriers are so low that there is no excuse not
to test.

The key ideas from this chapter:

- **`#[test]` and `#[cfg(test)]`.** Mark test functions and test modules.
  Test code is stripped from release builds.
- **`assert!`, `assert_eq!`, `assert_ne!`.** The core assertion macros. Add
  custom messages for clarity.
- **`#[should_panic(expected = "...")]`.** Test that code panics with the
  right message.
- **`Result`-returning tests.** Use `?` in tests for cleaner error handling.
- **`#[ignore]` for slow tests.** Run them only when you ask for them.
- **Integration tests in _tests/_.** Each file is a separate crate that tests
  your public API.
- **Doc tests.** Code examples in `///` comments are compiled and executed.
  Use `no_run` for examples that need files or network. Use hidden lines
  (`# ...`) for setup code.
- **`cargo test` filtering.** Run subsets of tests by name, kind, or
  characteristics. Show output with `--show-output`.
