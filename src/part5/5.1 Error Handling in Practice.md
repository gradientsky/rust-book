# Error Handling in Practice

In Part 3, you learned that Rust makes errors visible in the type system:
`Result<T, E>` for operations that can fail, `Option<T>` for values that might
not exist, and the `?` operator for propagating errors concisely. You also saw
how to define a simple error enum with a `Display` implementation.

That foundation is enough for small programs. But as your code grows, you will
face questions that `Result` and `?` alone do not answer. A function calls three
libraries, each with its own error type — how does `?` reconcile them? Your
application prints an error message, but the user needs to know _why_ the
configuration file failed to load, not just that parsing failed. Your library
exposes an API, and callers need to match on specific failure modes without
depending on string messages.

This chapter is about the practical engineering of errors. You will learn to
define custom error types that compose cleanly with `?`, add contextual
information that makes failures actionable, and choose the right tool for the
job — typed errors for libraries, ergonomic catch-all errors for applications.

## The Standard Error Trait

Every error in Rust's ecosystem shares a common foundation: the `Error` trait
from the standard library. Understanding it explains why thiserror and anyhow
work the way they do.

```rust
use std::error::Error;
use std::fmt;

#[derive(Debug)]
struct ConfigError {
    path: String,
    line: usize,
    message: String,
}

impl fmt::Display for ConfigError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "{}:{}: {}", self.path, self.line, self.message)
    }
}

impl Error for ConfigError {}

fn main() {
    let err = ConfigError {
        path: String::from("config.toml"),
        line: 12,
        message: String::from("invalid port number"),
    };

    println!("{err}");
    println!("{err:?}");
}
```

Output:

```
config.toml:12: invalid port number
ConfigError { path: "config.toml", line: 12, message: "invalid port number" }
```

The `Error` trait requires two things: `Debug` (so the error can be printed for
developers) and `Display` (so it can be printed for users). The trait itself
provides one important method — `source()` — which returns the underlying cause,
if any. It defaults to `None`, meaning "this error has no deeper cause."

### Error Chains

Real failures have layers. A file read fails because the path does not exist.
A configuration load fails because the file read failed. An application startup
fails because the configuration load failed. Each layer adds meaning.

The `source` method connects these layers into a chain:

```rust
use std::error::Error;
use std::fmt;
use std::num::ParseIntError;

#[derive(Debug)]
struct PortError {
    input: String,
    cause: ParseIntError,
}

impl fmt::Display for PortError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "invalid port: '{}'", self.input)
    }
}

impl Error for PortError {
    fn source(&self) -> Option<&(dyn Error + 'static)> {
        Some(&self.cause)
    }
}

fn parse_port(input: &str) -> Result<u16, PortError> {
    input.parse::<u16>().map_err(|e| PortError {
        input: input.to_string(),
        cause: e,
    })
}

fn main() {
    let err = parse_port("not_a_port").unwrap_err();

    // Walk the error chain
    println!("error: {err}");
    let mut source = err.source();
    while let Some(cause) = source {
        println!("caused by: {cause}");
        source = cause.source();
    }
}
```

Output:

```
error: invalid port: 'not_a_port'
caused by: invalid digit found in string
```

The outer error says _what_ went wrong in your domain. The inner error says
_why_ at a technical level. Together, they give the user enough information to
fix the problem.

This pattern — implement `Display`, implement `Error` with `source()`,
implement `From` for the underlying error — is correct but verbose. Every
custom error type requires the same boilerplate. This is where thiserror comes
in.

## Custom Error Types with thiserror

The thiserror crate generates the `Display`, `Error`, and `From`
implementations you would write by hand. You describe the shape of your error
with a derive macro; thiserror fills in the mechanical code.

Add thiserror to your project:

```toml
[dependencies]
thiserror = "2"
```

Then define your error type:

```rust
use thiserror::Error;

#[derive(Error, Debug)]
enum DataError {
    #[error("file not found: {path}")]
    NotFound { path: String },

    #[error("invalid format at line {line}: {message}")]
    InvalidFormat { line: usize, message: String },

    #[error("I/O error")]
    Io(#[from] std::io::Error),

    #[error("parse error")]
    Parse(#[from] std::num::ParseIntError),
}

fn read_value(path: &str) -> Result<i32, DataError> {
    // In a real program, you would read from the file system.
    // Here we simulate the error to demonstrate the pattern.
    if path.ends_with(".missing") {
        return Err(DataError::NotFound {
            path: path.to_string(),
        });
    }

    let content = "not_a_number";
    let value: i32 = content.parse()?; // ? uses From<ParseIntError> for DataError
    Ok(value)
}

fn main() {
    match read_value("data.missing") {
        Ok(v) => println!("value: {v}"),
        Err(e) => println!("error: {e}"),
    }

    match read_value("data.txt") {
        Ok(v) => println!("value: {v}"),
        Err(e) => {
            println!("error: {e}");
            if let Some(source) = std::error::Error::source(&e) {
                println!("caused by: {source}");
            }
        }
    }
}
```

Output:

```
error: file not found: data.missing
error: parse error
caused by: invalid digit found in string
```

Three attributes do all the work:

- `#[error("...")]` generates the `Display` implementation. Use `{field_name}`
  or `{0}` to interpolate fields into the message.
- `#[from]` on a field generates `impl From<FieldType> for YourError`, which
  is what lets `?` convert the underlying error automatically. It also sets
  that field as the `source()`.
- `#[source]` marks a field as the return value of `source()` without
  generating a `From` implementation. Use this when multiple variants could
  wrap the same error type.

### Choosing Variant Shapes

Error variants follow the same rules as enum variants. Use named fields when
the error carries domain-specific context, and tuple fields when wrapping
another error type:

```rust
use thiserror::Error;

#[derive(Error, Debug)]
enum AppError {
    // Named fields: carries domain context
    #[error("user {user_id} not found")]
    UserNotFound { user_id: u64 },

    // Named fields: wrapping with added context
    #[error("failed to load config from {path}")]
    Config {
        path: String,
        #[source]
        cause: std::io::Error,
    },

    // Tuple field with #[from]: simple wrapping
    #[error("parse error")]
    Parse(#[from] std::num::ParseIntError),
}

fn main() {
    let err = AppError::UserNotFound { user_id: 42 };
    println!("{err}");

    let err = AppError::Config {
        path: String::from("/etc/app/config.toml"),
        cause: std::io::Error::new(
            std::io::ErrorKind::NotFound,
            "file not found",
        ),
    };
    println!("{err}");
    if let Some(source) = std::error::Error::source(&err) {
        println!("caused by: {source}");
    }

    // ? on a ParseIntError auto-converts via #[from]
    let err: AppError = "abc".parse::<i32>().unwrap_err().into();
    println!("{err}");
}
```

Output:

```
user 42 not found
failed to load config from /etc/app/config.toml
caused by: file not found
parse error
```

The three variants illustrate a clear pattern. `UserNotFound` carries only
domain data — no underlying error. `Config` wraps an `std::io::Error` but adds
context (the file path), so it uses `#[source]` to mark the cause without
generating a `From` implementation — you construct it explicitly with `map_err`.
`Parse` does nothing but wrap a `ParseIntError`, so `#[from]` generates the
`From` implementation that lets `?` convert automatically.

### The Transparent Variant

Sometimes you want a catch-all variant that delegates entirely to the wrapped
error — its `Display` output, its `source()` chain, everything:

```rust
use thiserror::Error;

#[derive(Error, Debug)]
enum ServiceError {
    #[error("authentication failed")]
    Auth,

    #[error("rate limit exceeded")]
    RateLimit,

    #[error(transparent)]
    Other(Box<dyn std::error::Error + Send + Sync>),
}

fn main() {
    let io_err = std::io::Error::new(
        std::io::ErrorKind::ConnectionRefused,
        "connection refused",
    );
    let err = ServiceError::Other(Box::new(io_err));
    println!("{err}");
}
```

Output:

```
connection refused
```

The `#[error(transparent)]` attribute makes the variant invisible — `Display`
and `source()` are forwarded to the inner error. This is useful for errors you
do not want to enumerate but still want to carry through your error type.

The `Send + Sync` bounds on the `Box` ensure the error can be safely shared
across threads — a requirement you will understand fully in Part 6. For now,
treat `Box<dyn std::error::Error + Send + Sync>` as the standard type for
"any error, anywhere."

## Ergonomic Errors with anyhow

The thiserror crate is the right choice for library code, where callers need to
match on specific error variants. But application code has different needs. You are
writing a CLI tool, a web server, or a data pipeline. Errors flow upward to be
logged or displayed to the user. You rarely match on error types — you just need
a readable message and a chain of context explaining what went wrong.

The anyhow crate is built for this use case. It provides a single error type,
`anyhow::Error`, that can hold any error implementing `std::error::Error`. No
custom error enums, no `From` implementations, no boilerplate.

Add anyhow to your project:

```toml
[dependencies]
anyhow = "1"
```

### Your First anyhow Function

With anyhow in your dependencies, you can return `anyhow::Result<T>` and attach context to every error:

```rust
use anyhow::{Context, Result};

fn read_config(path: &str) -> Result<String> {
    let content = std::fs::read_to_string(path)
        .with_context(|| format!("failed to read config from '{path}'"))?;
    Ok(content)
}

fn main() {
    match read_config("missing.toml") {
        Ok(content) => println!("config: {content}"),
        Err(e) => {
            // Print the full error chain
            println!("error: {e}");
            for cause in e.chain().skip(1) {
                println!("caused by: {cause}");
            }
        }
    }
}
```

Output:

```
error: failed to read config from 'missing.toml'
caused by: No such file or directory (os error 2)
```

`anyhow::Result<T>` is shorthand for `Result<T, anyhow::Error>`. The `?`
operator converts any `std::error::Error` into `anyhow::Error` automatically.
The `Context` trait adds the `.context()` and `.with_context()` methods, which
wrap the error with a human-readable message.

The `.with_context()` method takes a closure that is only called when the error
actually occurs — use it when building the context string is not trivial.
`.context()` takes a value directly and is simpler when the message is a
literal string.

### Creating Errors on the Fly

Sometimes you need to produce an error that does not come from another error.
The anyhow crate provides two macros for this:

```rust
use anyhow::{anyhow, bail, ensure, Result};

fn validate_age(age: u32) -> Result<u32> {
    ensure!(age > 0, "age must be positive");
    if age > 150 {
        bail!("age {age} is unreasonable");
    }
    Ok(age)
}

fn lookup_host(name: &str) -> Result<String> {
    if name.is_empty() {
        return Err(anyhow!("hostname cannot be empty"));
    }
    Ok(format!("192.168.1.1")) // simulated
}

fn main() {
    println!("{:?}", validate_age(0));
    println!("{:?}", validate_age(200));
    println!("{:?}", validate_age(30));
    println!("{:?}", lookup_host(""));
    println!("{:?}", lookup_host("example.com"));
}
```

Output:

```
Err(age must be positive)
Err(age 200 is unreasonable)
Ok(30)
Err(hostname cannot be empty)
Ok("192.168.1.1")
```

- `bail!("...")` is shorthand for `return Err(anyhow!("..."))`. Use it to exit
  a function with an error message.
- `ensure!(condition, "...")` is like `assert!`, but returns an `Err` instead
  of panicking. Use it for precondition checks.
- `anyhow!("...")` creates an ad-hoc error value. Use it when you need an
  error as an expression rather than an early return.

### Context Layering

The real power of anyhow is _context layering_ — each function adds a sentence
explaining what it was doing when the error occurred:

```rust
use anyhow::{Context, Result};

fn parse_port(input: &str) -> Result<u16> {
    input.parse::<u16>()
        .context("invalid port number")
}

fn load_address(host: &str, port_str: &str) -> Result<String> {
    let port = parse_port(port_str)
        .with_context(|| format!("in address '{host}:{port_str}'"))?;
    Ok(format!("{host}:{port}"))
}

fn start_server(config_host: &str, config_port: &str) -> Result<()> {
    let addr = load_address(config_host, config_port)
        .context("failed to configure server address")?;
    println!("starting server on {addr}");
    Ok(())
}

fn main() {
    if let Err(e) = start_server("localhost", "abc") {
        println!("error: {e}");
        for cause in e.chain().skip(1) {
            println!("  caused by: {cause}");
        }
    }
}
```

Output:

```
error: failed to configure server address
  caused by: in address 'localhost:abc'
  caused by: invalid port number
  caused by: invalid digit found in string
```

Read from top to bottom, the chain tells a story: the server failed to start
because the address was invalid, because the port was not a number, because
the parser found a non-digit character. Each layer is added by a different
function, and each describes the failure at its own level of abstraction.

## Composing Error Types with the Question Mark

In Part 3, you learned that `?` extracts the `Ok` value or returns the `Err`
early. But there is a crucial detail: `?` calls `From::from()` on the error
before returning it. This is what makes different error types compose.

### How the Question Mark Converts Errors

When you write:

```rust,ignore
let value = some_operation()?;
```

The compiler expands it roughly to:

```rust,ignore
let value = match some_operation() {
    Ok(v) => v,
    Err(e) => return Err(From::from(e)),
};
```

The `From::from(e)` call converts the operation's error type into the
function's return error type. If those types are the same, the conversion is
a no-op. If they differ, Rust looks for a `From` implementation to bridge
them — and that is exactly what thiserror's `#[from]` attribute generates.

### Multiple Error Types in One Function

Here is the common scenario: a function calls operations that produce different
error types:

```rust
use std::collections::HashMap;
use std::num::ParseIntError;
use thiserror::Error;

#[derive(Error, Debug)]
enum ConfigError {
    #[error("missing key: {0}")]
    MissingKey(String),

    #[error("invalid value for '{key}'")]
    InvalidValue {
        key: String,
        #[source]
        cause: ParseIntError,
    },
}

fn load_config(raw: &HashMap<&str, &str>) -> Result<(String, u16), ConfigError> {
    let host = raw.get("host")
        .ok_or_else(|| ConfigError::MissingKey("host".to_string()))?;

    let port_str = raw.get("port")
        .ok_or_else(|| ConfigError::MissingKey("port".to_string()))?;

    let port: u16 = port_str.parse().map_err(|e| ConfigError::InvalidValue {
        key: "port".to_string(),
        cause: e,
    })?;

    Ok((host.to_string(), port))
}

fn main() {
    let mut config = HashMap::new();
    config.insert("host", "localhost");
    config.insert("port", "8080");
    println!("{:?}", load_config(&config));

    let empty = HashMap::new();
    println!("{:?}", load_config(&empty));

    let mut bad = HashMap::new();
    bad.insert("host", "localhost");
    bad.insert("port", "abc");
    match load_config(&bad) {
        Ok(v) => println!("{v:?}"),
        Err(e) => {
            println!("error: {e}");
            if let Some(source) = std::error::Error::source(&e) {
                println!("caused by: {source}");
            }
        }
    }
}
```

Output:

```
Ok(("localhost", 8080))
Err(MissingKey("host"))
error: invalid value for 'port'
caused by: invalid digit found in string
```

When you need to attach context to an error — like which configuration key
failed to parse — construct the variant explicitly with `map_err`. When a
variant simply wraps another error with no extra context, `#[from]` and `?`
handle the conversion automatically.

### Observing Errors in Flight

When debugging a chain of `?` calls, you often want to see which step failed
_without_ changing the error or breaking the chain. One approach is to split
the `?` into a `match` or `if let`, add a log line, and then return the
error — but that trades one line for five and obscures the linear flow.

The `inspect_err` method on `Result` solves this. It calls a closure with a
reference to the error value, then returns the original `Result` unchanged.
The closure can log, record a metric, or print a debug message — it observes
the error without consuming or transforming it:

```rust
fn parse_port(input: &str) -> Result<u16, String> {
    input.parse::<u16>()
        .map_err(|e| format!("invalid port '{input}': {e}"))
}

fn parse_host(input: &str) -> Result<String, String> {
    if input.is_empty() {
        return Err("host cannot be empty".to_string());
    }
    Ok(input.to_string())
}

fn connect(host_str: &str, port_str: &str) -> Result<String, String> {
    let host = parse_host(host_str)
        .inspect_err(|e| println!("  [log] {e}"))?;

    let port = parse_port(port_str)
        .inspect_err(|e| println!("  [log] {e}"))?;

    Ok(format!("{host}:{port}"))
}

fn main() {
    println!("attempt 1:");
    match connect("localhost", "8080") {
        Ok(addr) => println!("  connected to {addr}"),
        Err(e) => println!("  failed: {e}"),
    }

    println!("attempt 2:");
    match connect("localhost", "abc") {
        Ok(addr) => println!("  connected to {addr}"),
        Err(e) => println!("  failed: {e}"),
    }

    println!("attempt 3:");
    match connect("", "8080") {
        Ok(addr) => println!("  connected to {addr}"),
        Err(e) => println!("  failed: {e}"),
    }
}
```

Output:

```
attempt 1:
  connected to localhost:8080
attempt 2:
  [log] invalid port 'abc': invalid digit found in string
  failed: invalid port 'abc': invalid digit found in string
attempt 3:
  [log] host cannot be empty
  failed: host cannot be empty
```

In the successful case, `inspect_err` does nothing — the closure is never
called. In the failing case, the log line prints _and_ the error continues to
propagate through `?`. The error reaches `main` exactly as it would without
the inspection.

The symmetric counterpart, `inspect`, calls its closure on the `Ok` value:

```rust,ignore
let port = parse_port(input)
    .inspect(|p| println!("parsed port: {p}"))
    .inspect_err(|e| println!("port error: {e}"))?;
```

Both methods follow the same principle — observe without modifying, then pass
the `Result` through unchanged. `Option` has its own `inspect` method that
works identically on `Some` values. In production, the `println!` calls
would typically be replaced by the tracing crate's `tracing::warn!` or
`tracing::debug!` macros.

## When to Use Which Approach

Error handling in Rust is not one-size-fits-all. The right approach depends on
who consumes the error.

### Libraries: Use thiserror

A library is code that other code calls. Your callers need to _react_ to
specific failure modes — retry on a timeout, prompt the user on invalid input,
fall back to a default on a missing file. This requires typed errors they can
match on:

```rust,ignore
use thiserror::Error;

// Library error type — callers can match on variants
#[derive(Error, Debug)]
pub enum StorageError {
    #[error("item not found: {id}")]
    NotFound { id: String },

    #[error("permission denied")]
    PermissionDenied,

    #[error("storage backend error")]
    Backend(#[from] std::io::Error),
}
```

Design guidelines for library error types:

- **One error enum per logical boundary.** A crate might have one top-level
  error type, or a few if it has distinct subsystems.
- **Use `#[from]` for simple wrapping.** When a variant does nothing but carry
  another error type, let thiserror generate the conversion.
- **Use `#[source]` with named fields for context.** When a variant adds domain
  information (a file path, a key name, a user ID), construct it explicitly.
- **Keep error messages lowercase and without trailing punctuation.** This is
  the Rust convention — errors are often printed as part of a larger sentence
  like `"failed to load config: invalid port number"`.

### Applications: Use anyhow

An application is the top-level program. Errors are logged, printed to the
user, or sent to an error tracking service. You rarely match on the error
type — you just need a clear, human-readable chain of context:

```rust,ignore
use anyhow::{Context, Result};

// Application code — errors carry context, not type information
fn run() -> Result<()> {
    let config = load_config("app.toml")
        .context("failed to initialize application")?;
    start_server(&config)?;
    Ok(())
}
```

### The Boundary Between Them

In a well-structured Rust project, library code (your domain logic, your data
layer) uses thiserror, and application code (your `main` function, your HTTP
handlers, your CLI entry points) uses anyhow. The `?` operator bridges the
two seamlessly — anyhow accepts any error that implements `std::error::Error`,
which thiserror errors do.

```rust,ignore
// Library crate (uses thiserror)
pub fn fetch_user(id: u64) -> Result<User, DatabaseError> { ... }

// Application crate (uses anyhow)
fn handle_request(id: u64) -> anyhow::Result<Response> {
    let user = fetch_user(id)
        .context("failed to fetch user for request")?;  // DatabaseError → anyhow::Error
    Ok(build_response(user))
}
```

This layering gives you the best of both worlds: typed, matchable errors where
callers need them, and ergonomic context chains where humans read them.

## The Fallible Main Function

Throughout this chapter, every `main` function has used `match` to handle
errors from fallible operations. When a program chains several fallible calls,
this creates a pattern of mechanical ceremony — matching, printing, and
returning at every step:

```rust
use std::error::Error;

fn parse_port(input: &str) -> Result<u16, Box<dyn Error>> {
    let port: u16 = input.parse()?;
    if port == 0 {
        return Err("port must be nonzero".into());
    }
    Ok(port)
}

fn connect(host: &str, port: u16) -> Result<String, Box<dyn Error>> {
    Ok(format!("{host}:{port}"))
}

// Without Result-returning main: match ceremony at every step
fn main() {
    let port = match parse_port("8080") {
        Ok(p) => p,
        Err(e) => {
            eprintln!("error: {e}");
            return;
        }
    };

    let addr = match connect("localhost", port) {
        Ok(a) => a,
        Err(e) => {
            eprintln!("error: {e}");
            return;
        }
    };

    println!("connected to {addr}");
}
```

```
connected to localhost:8080
```

The logic is buried under error-handling boilerplate. Rust solves this with
the same mechanism you already know: `main` can return `Result`.

### Returning Result from Main

When `main` returns `Result<(), E>` where `E` implements `Debug`, Rust handles
the error automatically. If `main` returns `Ok(())`, the program exits with
code 0. If it returns `Err(e)`, Rust prints `Error: {e:?}` to standard error
and exits with code 1:

```rust
use std::error::Error;

fn parse_port(input: &str) -> Result<u16, Box<dyn Error>> {
    let port: u16 = input.parse()?;
    if port == 0 {
        return Err("port must be nonzero".into());
    }
    Ok(port)
}

fn main() -> Result<(), Box<dyn Error>> {
    let port = parse_port("8080")?;
    let timeout: u64 = "30".parse()?;
    println!("server on port {port}, timeout {timeout}s");
    Ok(())
}
```

```
server on port 8080, timeout 30s
```

The `?` operator works exactly as it does in any other function. If
`parse_port` or `parse` returns an error, `main` returns `Err` immediately
and Rust prints the error. No `match`, no `eprintln!`, no manual exit code.

This works because of the _Termination_ trait, which Rust implements for
several types including `Result<(), E>` and `()`. You never need to implement
`Termination` yourself — it powers this pattern behind the scenes.

### Choosing the Error Type for Main

The error type determines how failures are reported. The three most common
choices:

| Return type | Error output | Best for |
|---|---|---|
| `Result<(), Box<dyn Error>>` | `Debug` format of the error | Quick scripts, prototypes |
| `Result<(), anyhow::Error>` | Full context chain (anyhow's `Debug` prints each layer) | Production applications |
| `Result<(), YourError>` | `Debug` format of your type | When you need specific exit behavior |

For applications using anyhow, a `Result`-returning `main` is the idiomatic
entry point:

```rust,ignore
use anyhow::{Context, Result};

fn main() -> Result<()> {
    let config = load_config("app.toml")
        .context("failed to initialize")?;
    start_server(&config)?;
    Ok(())
}
```

If the program fails, anyhow prints the full context chain — every layer of
`context` and every underlying error — because anyhow's `Debug` implementation
is designed for human-readable error reporting.

### Controlling the Exit Code

When `main` returns `Err`, Rust always exits with code 1. If you need a
specific exit code — to conform to a CLI convention or to communicate with a
parent process — use `std::process::ExitCode`:

```rust
use std::process::ExitCode;

fn run() -> Result<(), String> {
    let port: u16 = "8080".parse().map_err(|e| format!("bad port: {e}"))?;
    println!("server on port {port}");
    Ok(())
}

fn main() -> ExitCode {
    match run() {
        Ok(()) => ExitCode::SUCCESS,
        Err(e) => {
            eprintln!("error: {e}");
            ExitCode::FAILURE
        }
    }
}
```

```
server on port 8080
```

This pattern extracts the program logic into a `run` function that returns
`Result`, keeping `main` as a thin wrapper that translates success or failure
into an exit code. The `ExitCode` type also supports `ExitCode::from(n)` for
numeric codes beyond 0 and 1.

## When to Panic

Not every failure should be a `Result`. Some failures mean the program is in a
state so broken that continuing is worse than stopping. These are
_unrecoverable_ errors, and Rust handles them with _panics_.

A panic immediately unwinds the stack (by default), running destructors along
the way, and terminates the thread. If the main thread panics, the program
exits. Panics are loud, disruptive, and intentional — they are Rust's way of
saying "this should never happen, and if it does, stop everything."

### When Panicking Is Right

Use `unwrap` and `expect` when you have a logical guarantee that the value
exists:

```rust
fn main() {
    // We know this regex is valid — it is a compile-time constant.
    // If it somehow is not, the program cannot function at all.
    let digits = "abc123def456";
    let first_digit = digits.chars()
        .find(|c| c.is_ascii_digit())
        .expect("input is known to contain digits");
    println!("first digit: {first_digit}");

    // Environment setup that is a hard requirement
    let home = std::env::var("HOME")
        .expect("HOME environment variable must be set");
    println!("home: {home}");
}
```

Output:

```
first digit: 1
home: /Users/admin
```

The `expect` message documents the invariant: _why_ you believe this cannot
fail. When it does fail (because your assumption was wrong), the message
explains the programmer's intent, making the bug easier to find.

### The Unwrap Spectrum

Here is the guidance from most to least preferred:

1. **`?` (propagate)** — the default choice. Let the caller decide what to do
   with the error. Use this in library code and in any function that returns
   `Result` or `Option`.

2. **`unwrap_or` / `unwrap_or_else` / `unwrap_or_default`** — provide a
   fallback value. Use when absence or failure has a natural default.

3. **`expect("reason")`** — panic with an explanation. Use when you have proof
   the value exists and failure means a bug in your code, not bad input.

4. **`unwrap()`** — panic without explanation. Acceptable in tests and
   throwaway prototyping. Avoid in production code — if it panics, the message
   gives no clue about what went wrong.

## Putting It All Together

Here is a complete example that demonstrates the library/application split.
The "library" side defines a typed error with thiserror. The "application"
side uses anyhow for context layering:

```rust
use std::collections::HashMap;
use thiserror::Error;

// --- Library side: typed errors ---

#[derive(Error, Debug)]
enum InventoryError {
    #[error("item '{name}' not found")]
    NotFound { name: String },

    #[error("insufficient stock for '{name}': requested {requested}, available {available}")]
    InsufficientStock {
        name: String,
        requested: u32,
        available: u32,
    },
}

struct Inventory {
    stock: HashMap<String, u32>,
}

impl Inventory {
    fn new() -> Self {
        let mut stock = HashMap::new();
        stock.insert("widget".to_string(), 100);
        stock.insert("gadget".to_string(), 5);
        Self { stock }
    }

    fn reserve(&mut self, name: &str, qty: u32) -> Result<u32, InventoryError> {
        let available = self.stock.get_mut(name)
            .ok_or_else(|| InventoryError::NotFound {
                name: name.to_string(),
            })?;

        if qty > *available {
            return Err(InventoryError::InsufficientStock {
                name: name.to_string(),
                requested: qty,
                available: *available,
            });
        }

        *available -= qty;
        Ok(*available)
    }
}

// --- Application side: process orders ---

fn process_order(
    inventory: &mut Inventory,
    items: &[(&str, u32)],
) -> Result<(), Box<dyn std::error::Error>> {
    for &(name, qty) in items {
        let remaining = inventory.reserve(name, qty)?;
        println!("reserved {qty} {name}(s), {remaining} remaining");
    }
    Ok(())
}

fn main() {
    let mut inventory = Inventory::new();

    // Successful order
    println!("--- order 1 ---");
    match process_order(&mut inventory, &[("widget", 10), ("gadget", 2)]) {
        Ok(()) => println!("order complete\n"),
        Err(e) => println!("order failed: {e}\n"),
    }

    // Failing order: not enough stock
    println!("--- order 2 ---");
    match process_order(&mut inventory, &[("gadget", 10)]) {
        Ok(()) => println!("order complete\n"),
        Err(e) => println!("order failed: {e}\n"),
    }

    // Failing order: item not found
    println!("--- order 3 ---");
    match process_order(&mut inventory, &[("sprocket", 1)]) {
        Ok(()) => println!("order complete"),
        Err(e) => println!("order failed: {e}"),
    }
}
```

Output:

```
--- order 1 ---
reserved 10 widget(s), 90 remaining
reserved 2 gadget(s), 3 remaining
order complete

--- order 2 ---
order failed: insufficient stock for 'gadget': requested 10, available 3

--- order 3 ---
order failed: item 'sprocket' not found
```

This example demonstrates the key patterns:

- **Typed library errors** with thiserror: `InventoryError` has variants that
  callers can match on — `NotFound` vs `InsufficientStock` carry different data
  and suggest different responses.
- **The `?` operator with `From`**: The `reserve` method uses `?` with
  `ok_or_else` to convert `Option` into `Result`, keeping the code linear.
- **Application-level error handling**: `process_order` returns
  `Box<dyn std::error::Error>`, accepting any error type. In a real
  application, this would be `anyhow::Result` — the effect is the same.
- **Error messages are lowercase, without trailing punctuation** — following
  the Rust convention so they compose well when wrapped in context.

## Why This Matters

Error handling is not about preventing crashes — it is about _communicating
what went wrong_ clearly enough that someone can fix it. A message like
"parse error" wastes the reader's time. A message like "invalid port number
'abc' in config.toml at line 12" tells them exactly where to look.

The two-crate pattern — thiserror for libraries, anyhow for applications — has
emerged as the Rust community's standard approach because it respects both
audiences:

- Library consumers get typed errors they can inspect programmatically.
- Application users get layered context chains that tell the full story.

Key points from this chapter:

- **The `Error` trait** requires `Debug` and `Display`. Its `source()` method
  links errors into a causal chain.
- **thiserror** generates `Display`, `Error`, and `From` implementations from
  a derive macro. Use `#[error("...")]` for messages, `#[from]` for automatic
  conversion, and `#[source]` when a variant carries extra context alongside
  the underlying error.
- **anyhow** provides a single error type that holds any `std::error::Error`.
  Use `.context()` and `.with_context()` to wrap errors with human-readable
  descriptions. Use `bail!` for early returns and `ensure!` for precondition
  checks.
- **Libraries use thiserror; applications use anyhow.** The `?` operator
  bridges the two — thiserror errors implement `std::error::Error`, which
  anyhow accepts.
- **`?` calls `From::from()`** on the error, which is how different error types
  compose in a single function.
- **`inspect_err` observes without modifying.** Use it to log or debug errors
  in `?` chains without breaking the linear flow. Its counterpart `inspect`
  does the same for `Ok` values.
- **`fn main() -> Result<(), E>`** lets you use `?` directly in `main`. Rust
  prints the error and exits with code 1 on failure. Use `ExitCode` when you
  need control over the specific exit code.
- **Panic for logic bugs, not for user errors.** Use `expect` with a message
  that explains the invariant. Reserve `unwrap` for tests.
