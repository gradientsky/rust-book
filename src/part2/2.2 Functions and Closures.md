# Functions and Closures

In the previous chapter, you saw that Rust is an expression-oriented language
— blocks produce values, `if` is an expression, and the distinction between
expressions and statements shapes how you write every line of code. Functions
take this idea to its natural conclusion: a function body is a block, and a
block evaluates to its last expression. This means Rust functions are
_concise by design_ — you rarely need a `return` keyword, and the flow from
input to output reads clearly from top to bottom.

But functions are more than a way to avoid repeating code. They are the primary
unit of _abstraction_ in Rust. A well-defined function signature tells you
exactly what goes in, what comes out, and — because Rust is statically typed —
the compiler guarantees those contracts at every call site. When you read a
function signature, you know what it does before reading its body.

This chapter covers functions and closures — named blocks of logic and anonymous
ones — and how to pass behavior around as data.

## Defining Functions

A function in Rust starts with `fn`, followed by a name, parameters in
parentheses, and a body in braces:

```rust
fn greet(name: &str) {
    println!("Hello, {name}!");
}

fn main() {
    greet("Rustacean");
}
```

Output: `Hello, Rustacean!`

Every parameter must have an explicit type annotation. Unlike `let` bindings
where the compiler infers types, function signatures are always explicit — they
serve as documentation and as a contract between the function and its callers.

The parameter `name: &str` means the function borrows a string slice. You do not
need to understand borrowing in depth yet — for now, think of `&str` as "a
read-only view of some text." This is the idiomatic type for functions that need
to read text without modifying or owning it.

### Return Values

A function that produces a value declares its return type after an arrow (`->`):

```rust
fn add(a: i32, b: i32) -> i32 {
    a + b
}

fn main() {
    let sum = add(3, 4);
    println!("sum = {sum}");
}
```

Output: `sum = 7`

Notice that `a + b` has no semicolon. This makes it an _expression_, and its
value becomes the return value of the function. This is the same rule you
learned for blocks in the previous chapter — the last expression in a block,
without a semicolon, is the block's value.

If you accidentally add a semicolon, the function returns `()` — the unit type
— and the compiler tells you exactly what went wrong:

```rust,does_not_compile
fn add(a: i32, b: i32) -> i32 {
    a + b; // oops — semicolon turns this into a statement
}
```

```
error[E0308]: mismatched types
 --> src/main.rs:1:27
  |
1 | fn add(a: i32, b: i32) -> i32 {
  |    ---                    ^^^ expected `i32`, found `()`
  |    |
  |    implicitly returns `()` as its body has no tail or `return` expression
2 |     a + b;
  |          - help: remove this semicolon to return this value
```

The error is clear and the fix is a single character. This is one of the
benefits of expression-oriented design: the common case — returning a value — is
the shortest to write.

### Early Returns

Sometimes you need to exit a function before reaching the end. The `return`
keyword does this explicitly:

```rust
fn first_positive(numbers: &[i32]) -> Option<i32> {
    for &n in numbers {
        if n > 0 {
            return Some(n);
        }
    }
    None
}

fn main() {
    let nums = [-3, -1, 0, 4, 7];
    println!("{:?}", first_positive(&nums));
}
```

Output: `Some(4)`

Use `return` for early exits. Use the implicit last-expression return for the
normal path. This convention makes functions easy to scan: if you see a `return`
keyword, something exceptional is happening — the happy path flows naturally to
the end.

The `Option` type you see here is Rust's replacement for null — a value is
either `Some(value)` or `None`. You will learn it in depth in Part 3, but its
meaning is intuitive enough to use now: "this function might not find a positive
number."

### Functions That Return Nothing

When a function performs an action without producing a meaningful value, you
omit the return type. The function implicitly returns `()`:

```rust
fn log_value(label: &str, value: i32) {
    println!("[LOG] {label}: {value}");
}

fn main() {
    log_value("score", 42);
}
```

Output: `[LOG] score: 42`

Writing `-> ()` explicitly is valid but unnecessary — omitting the return type
means the same thing and is the idiomatic style.

## Expression-Oriented Function Bodies

Because `if`, `match`, and blocks are all expressions, function bodies in Rust
tend to be more concise than in languages where these constructs are statements.
Compare these approaches to classifying a number:

```rust
fn classify(n: i32) -> &'static str {
    if n > 0 {
        "positive"
    } else if n < 0 {
        "negative"
    } else {
        "zero"
    }
}

fn main() {
    println!("{}", classify(42));
    println!("{}", classify(-7));
    println!("{}", classify(0));
}
```

Output:

```
positive
negative
zero
```

There is no mutable variable, no `return` keyword, and no risk of forgetting to
assign in one of the branches. The `if` expression directly produces the return
value. This pattern scales naturally — any expression that produces the right
type can be the tail of a function body.

The `'static` in `&'static str` means the string slice lives for the entire
duration of the program. String literals like `"positive"` are always
`&'static str` because they are baked into the compiled binary. You do not need
to understand lifetimes fully yet — this is simply the type that string literals
have.

## Closures

A _closure_ is an anonymous function that can capture variables from its
surrounding scope. Where a regular function stands alone, a closure can "close
over" the environment where it is defined — that is where the name comes from.

Here is a closure that adds a fixed offset to a number:

```rust
fn main() {
    let offset = 10;
    let add_offset = |x: i32| -> i32 { x + offset };

    println!("{}", add_offset(5));
    println!("{}", add_offset(20));
}
```

Output:

```
15
30
```

The syntax `|x: i32| -> i32 { x + offset }` defines a closure. The pipes (`|`)
delimit the parameter list, just as parentheses do for functions. The closure
uses `offset`, a variable defined outside of it — it _captures_ `offset` from
the enclosing scope.

### Concise Closure Syntax

Closures are designed to be lightweight. Rust can infer the parameter types and
return type from context, and if the body is a single expression, you can omit
the braces:

```rust
fn main() {
    let double = |x| x * 2;
    let is_even = |x: i32| x % 2 == 0;

    println!("double 5 = {}", double(5));
    println!("4 is even: {}", is_even(4));
}
```

Output:

```
double 5 = 10
4 is even: true
```

The `double` closure has no type annotations at all — the compiler infers both
the parameter type and the return type from how `double` is used. This makes
closures ideal for short, inline operations where full function signatures would
be noise.

Once the compiler infers concrete types for a closure, those types are fixed.
You cannot call `double` with a floating-point number after the compiler has
already inferred `i32` from the first use.

### How Closures Capture Variables

Closures can use variables from their surrounding scope, and the compiler
automatically decides _how_ to capture each variable based on what the closure
does with it:

```rust
fn main() {
    let greeting = "hello";
    let print_greeting = || println!("{greeting}");

    print_greeting();
    print_greeting();
    println!("still accessible: {greeting}");
}
```

Output:

```
hello
hello
still accessible: hello
```

The closure only reads `greeting`, so the compiler captures it by _reference_ —
a temporary borrow. The original variable remains usable after the closure is
defined and called.

When a closure modifies a captured variable, the compiler captures it by
_mutable reference_:

```rust
fn main() {
    let mut count = 0;

    let mut increment = || {
        count += 1;
        println!("count = {count}");
    };

    increment();
    increment();
}
```

Output:

```
count = 1
count = 2
```

Two things to notice: the closure itself must be declared `mut` because calling
it changes internal state, and the captured variable `count` must also be `mut`
because the closure modifies it.

The key insight is that _you do not need to specify how a closure captures its
variables_. The compiler figures out the least restrictive capture mode
automatically. If the closure only reads a value, it borrows. If it modifies a
value, it borrows mutably. You write the logic you need, and the compiler picks
the right mechanism.

### The Three Closure Traits

Every closure in Rust automatically implements one or more traits from a
hierarchy of three:

- `Fn` — the closure can be called repeatedly and does not modify its captured
  state. Think of it as a pure function with some extra context.
- `FnMut` — the closure can be called repeatedly but _may_ modify captured
  state. It needs exclusive access when called.
- `FnOnce` — the closure can be called at least once. It may consume (take
  ownership of) its captured state, which means it might not be callable a
  second time.

These traits form a hierarchy: every `Fn` closure is also `FnMut`, and every
`FnMut` closure is also `FnOnce`. A closure that only reads its captures
implements all three. A closure that modifies captures implements `FnMut` and
`FnOnce`. A closure that consumes a capture implements only `FnOnce`.

You do not need to memorize this hierarchy right now. The compiler picks the
right traits automatically based on what your closure does. The names become
important when you write functions that _accept_ closures as parameters — which
is what comes next.

## Higher-Order Functions

A _higher-order function_ is a function that takes another function — or a
closure — as a parameter. This is one of the most powerful patterns in Rust,
and you will encounter it constantly when working with iterators.

Here is a function that applies a transformation to each element of a slice
and shows the result:

```rust
fn apply_to_each(values: &[i32], f: impl Fn(i32) -> i32) {
    for &v in values {
        println!("  {v} -> {}", f(v));
    }
}

fn main() {
    let numbers = [1, 2, 3];

    println!("Doubled:");
    apply_to_each(&numbers, |x| x * 2);

    println!("Plus ten:");
    apply_to_each(&numbers, |x| x + 10);
}
```

Output:

```
Doubled:
  1 -> 2
  2 -> 4
  3 -> 6
Plus ten:
  1 -> 11
  2 -> 12
  3 -> 13
```

The parameter `f: impl Fn(i32) -> i32` means "any callable thing that takes an
`i32` and returns an `i32`." This accepts closures, regular functions, and
anything else that implements the `Fn` trait. The `impl` keyword here is
shorthand for a generic type parameter — you will learn the full generic syntax
in Part 4, but `impl Trait` reads naturally: "a parameter that implements this
trait."

### Passing Named Functions

You are not limited to passing closures. Named functions work too, as long as
their signature matches:

```rust
fn square(x: i32) -> i32 {
    x * x
}

fn apply(value: i32, f: impl Fn(i32) -> i32) -> i32 {
    f(value)
}

fn main() {
    println!("square of 5: {}", apply(5, square));
    println!("plus one:    {}", apply(5, |x| x + 1));
}
```

Output:

```
square of 5: 25
plus one:    6
```

A named function like `square` and an anonymous closure like `|x| x + 1` are
interchangeable when passed to a function that accepts `impl Fn`. This
uniformity is by design — it lets you choose whichever form is clearest for each
call site.

### Choosing the Right Trait Bound

When writing a function that accepts a closure, match the trait bound to how
your function uses it:

- Use `FnOnce` if you call the closure exactly once. This is the least
  restrictive — every closure can be called at least once.
- Use `FnMut` if you call the closure multiple times and it _may_ mutate state.
- Use `Fn` if you call the closure multiple times and it should not mutate
  state. This is the most restrictive.

`FnOnce` accepts the widest range of closures, so use it when your function
only needs a single call:

```rust
fn run_once(f: impl FnOnce()) {
    f();
}

fn run_repeatedly(mut f: impl FnMut()) {
    f();
    f();
    f();
}

fn main() {
    let greeting = String::from("hello");

    // `move` transfers ownership of `greeting` into the closure.
    // FnOnce is the right bound when you call the closure exactly once.
    run_once(move || println!("{greeting}"));

    // This closure mutates captured state — it needs FnMut.
    let mut count = 0;
    run_repeatedly(|| {
        count += 1;
        println!("count = {count}");
    });
}
```

Output:

```
hello
count = 1
count = 2
count = 3
```

The `move` keyword before a closure forces it to take ownership of captured
variables rather than borrowing them. Without `move`, this closure would borrow
`greeting`, and the original variable would remain accessible afterward. With
`move`, the `String` value transfers into the closure — `greeting` is consumed,
and the closure becomes fully self-contained, carrying everything it needs.

This matters whenever a closure must outlive the scope where it was created. If
the closure only borrows a variable, and that variable goes out of scope before
the closure runs, the borrow would become invalid. By moving ownership into the
closure, Rust guarantees the data lives as long as the closure does — no
dangling references, no runtime checks. The next section shows exactly this: a
function that returns a closure, where `move` is essential because the
function's local scope disappears after the return.

Notice that `run_repeatedly` declares its parameter as `mut f` — this is
required because each call to an `FnMut` closure may modify its internal state.

If `FnOnce` accepts the widest range of closures, why not always use it?
Because `FnOnce` only lets you call the closure _once_. If your function needs
multiple calls — like `run_repeatedly` does — you need `FnMut` or `Fn`.

## Returning Closures

Functions can also _return_ closures. Since every closure has a unique type
known only to the compiler, you use `impl Fn` in the return position to say
"I am returning something callable":

```rust
fn make_greeter(greeting: &'static str) -> impl Fn(&str) {
    move |name| println!("{greeting}, {name}!")
}

fn main() {
    let hello = make_greeter("Hello");
    let hi = make_greeter("Hi");

    hello("Alice");
    hi("Bob");
}
```

Output:

```
Hello, Alice!
Hi, Bob!
```

The `move` keyword is essential here — without it, the closure would try to
borrow `greeting` from the function's scope, but that scope no longer exists
after the function returns. With `move`, the closure takes ownership of
`greeting` and carries it along wherever it goes.

This pattern — a function that returns a configured closure — is called a
_function factory_. It is a clean way to create specialized behavior without
defining a new struct or trait.

## Putting It Together

Here is a small program that combines several ideas from this chapter —
functions, closures, and higher-order functions — to build a simple temperature
converter:

```rust
fn convert_temps(
    temps: &[f64],
    converter: impl Fn(f64) -> f64,
    label: &str,
) {
    for &t in temps {
        println!("{t:.1} -> {:.1} {label}", converter(t));
    }
}

fn main() {
    let celsius_temps = [0.0, 20.0, 37.0, 100.0];

    println!("Celsius to Fahrenheit:");
    convert_temps(&celsius_temps, |c| c * 9.0 / 5.0 + 32.0, "°F");

    println!();

    println!("Celsius to Kelvin:");
    convert_temps(&celsius_temps, |c| c + 273.15, "K");
}
```

Output:

```
Celsius to Fahrenheit:
0.0 -> 32.0 °F
20.0 -> 68.0 °F
37.0 -> 98.6 °F
100.0 -> 212.0 °F

Celsius to Kelvin:
0.0 -> 273.1 K
20.0 -> 293.1 K
37.0 -> 310.1 K
100.0 -> 373.1 K
```

The `convert_temps` function does not know or care what conversion formula it
applies — it just calls whatever closure is passed in. The logic is separated
from the data transformation, making both easy to test and reuse.

---

Functions and closures are the building blocks of every Rust program. Functions
give your code structure and clear contracts through explicit signatures.
Closures add flexibility by capturing context and letting you pass behavior as
data. Together with the expression-oriented design from the previous chapter,
they let you write code that is both concise and easy to reason about.

In the next chapter, you will learn about _ownership_ — the single most
important idea in Rust and the foundation of its memory safety guarantees. The
functions and closures you have just learned will be central to that story,
because ownership determines how values flow into, out of, and between
functions.
