# Functions and Closures

In the previous chapter, you saw that Rust is an expression-oriented language
— blocks produce values, `if` is an expression, and the distinction between
expressions and statements shapes how you write every line of code. Functions
take this idea to its natural conclusion: a function body is a block, and a
block evaluates to its last expression. This means Rust functions are
_concise by design_ — you rarely need a `return` keyword, and the flow from
input to output reads clearly from top to bottom.

But functions are more than a way to avoid repeating code. They are the primary
unit of _abstraction_ in Rust. A well-defined function signature tells you
exactly what goes in, what comes out, and — because Rust is statically typed —
the compiler guarantees those contracts at every call site. When you read a
function signature, you know what it does before reading its body.

This chapter covers functions and closures — named blocks of logic and anonymous
ones — and how to pass behavior around as data.

## Defining Functions

A function in Rust starts with `fn`, followed by a name, parameters in
parentheses, and a body in braces:

```rust
fn greet(name: &str) {
    println!("Hello, {name}!");
}

fn main() {
    greet("Rustacean");
}
```

Output: `Hello, Rustacean!`

Every parameter must have an explicit type annotation. Unlike `let` bindings
where the compiler infers types, function signatures are always explicit — they
serve as documentation and as a contract between the function and its callers.

Rust does not have default parameter values — you cannot write
`fn greet(name: &str = "world")`. If you want optional arguments, the standard
approach is to accept `Option<T>` and provide a default inside the function
with `.unwrap_or()`.

The parameter `name: &str` means the function borrows a string slice. You do not
need to understand borrowing in depth yet — for now, think of `&str` as "a
read-only view of some text." This is the idiomatic type for functions that need
to read text without modifying or owning it.

### Return Values

A function that produces a value declares its return type after an arrow (`->`):

```rust
fn add(a: i32, b: i32) -> i32 {
    a + b
}

fn main() {
    let sum = add(3, 4);
    println!("sum = {sum}");
}
```

Output: `sum = 7`

Notice that `a + b` has no semicolon. This makes it an _expression_, and its
value becomes the return value of the function. This is the same rule you
learned for blocks in the previous chapter — the last expression in a block,
without a semicolon, is the block's value.

If you accidentally add a semicolon, the function returns `()` — the unit type
— and the compiler tells you exactly what went wrong:

```rust,does_not_compile
fn add(a: i32, b: i32) -> i32 {
    a + b; // oops — semicolon turns this into a statement
}
```

```
error[E0308]: mismatched types
 --> src/main.rs:1:27
  |
1 | fn add(a: i32, b: i32) -> i32 {
  |    ---                    ^^^ expected `i32`, found `()`
  |    |
  |    implicitly returns `()` as its body has no tail or `return` expression
2 |     a + b;
  |          - help: remove this semicolon to return this value
```

The error is clear and the fix is a single character. This is one of the
benefits of expression-oriented design: the common case — returning a value — is
the shortest to write.

### Early Returns

Sometimes you need to exit a function before reaching the end. The `return`
keyword does this explicitly:

```rust
fn first_positive(numbers: &[i32]) -> Option<i32> {
    for &n in numbers {
        if n > 0 {
            return Some(n);
        }
    }
    None
}

fn main() {
    let nums = [-3, -1, 0, 4, 7];
    println!("{:?}", first_positive(&nums));
}
```

Output: `Some(4)`

Use `return` for early exits. Use the implicit last-expression return for the
normal path. This convention makes functions easy to scan: if you see a `return`
keyword, something exceptional is happening — the happy path flows naturally to
the end.

The `Option` type you see here is Rust's replacement for null — a value is
either `Some(value)` or `None`, and its meaning is intuitive: "this function
might not find a positive number."

When you have an `Option` and want to use the value inside it, `if let`
destructures the `Some` variant and binds the inner value:

```rust
fn main() {
    let nums = [-3, -1, 0, 4, 7];
    if let Some(n) = first_positive(&nums) {
        println!("found {n}");
    } else {
        println!("no positive numbers");
    }
}
```

Output: `found 4`

Inside the `if let` block, `n` is the unwrapped `i32` — not an `Option`.
You will see this pattern constantly in Rust. A full `match` works too, but
`if let` is more concise when you only care about one variant.

### Functions That Return Nothing

When a function performs an action without producing a meaningful value, you
omit the return type. The function implicitly returns `()`:

```rust
fn log_value(label: &str, value: i32) {
    println!("[LOG] {label}: {value}");
}

fn main() {
    log_value("score", 42);
}
```

Output: `[LOG] score: 42`

Writing `-> ()` explicitly is valid but unnecessary — omitting the return type
means the same thing and is the idiomatic style.

## Sketching Before You Build

When designing a program, you often know the shape of your code — the function
signatures, the types, the overall structure — before you know every
implementation detail. Rust lets you sketch that structure, get the code
compiling, and fill in the logic incrementally. This top-down workflow is
natural in a language with strong types: write the signatures first, make the
compiler happy, then implement one function at a time.

The `todo!()` macro is your primary tool for this:

```rust
fn shipping_cost(weight_kg: f64) -> f64 {
    if weight_kg <= 1.0 {
        5.99
    } else if weight_kg <= 5.0 {
        12.99
    } else {
        todo!("get rates for heavy packages")
    }
}

fn main() {
    println!("Light: ${:.2}", shipping_cost(0.5));
    println!("Medium: ${:.2}", shipping_cost(3.0));
}
```

Output:

```
Light: $5.99
Medium: $12.99
```

The function compiles and works for the cases you have implemented. The `else`
branch is a placeholder — `todo!()` satisfies the type checker because it never
actually returns a value. It always panics, so the compiler accepts it in place
of any return type. Your code compiles, and you can work on one piece at a
time.

If you called `shipping_cost(10.0)`, the program would panic with a message
showing your description, the file, and the line number:

```
not yet implemented: get rates for heavy packages
```

This tells you exactly which placeholder needs attention. You can also write
`todo!()` without a message — the panic will simply say "not yet implemented."

Rust provides two related macros for situations where the intent is different:

**`unimplemented!()`.** Signals that a code path exists but you have
intentionally chosen not to support it. Where `todo!()` says "I will get to
this," `unimplemented!()` says "this is deliberately absent."

**`unreachable!()`.** Asserts that a code path should never execute. Use it
when your logic guarantees a branch cannot be taken, but the compiler cannot
prove it. If the branch _is_ reached, the program panics — making the violated
assumption visible immediately.

All three macros produce a panic. The difference is the _intent_ they
communicate to anyone reading your code, including your future self.

## Expression-Oriented Function Bodies

Because `if`, `match`, and blocks are all expressions, function bodies in Rust
tend to be more concise than in languages where these constructs are statements.
Compare these approaches to classifying a number:

```rust
fn classify(n: i32) -> &'static str {
    if n > 0 {
        "positive"
    } else if n < 0 {
        "negative"
    } else {
        "zero"
    }
}

fn main() {
    println!("{}", classify(42));
    println!("{}", classify(-7));
    println!("{}", classify(0));
}
```

Output:

```
positive
negative
zero
```

There is no mutable variable, no `return` keyword, and no risk of forgetting to
assign in one of the branches. The `if` expression directly produces the return
value. This pattern scales naturally — any expression that produces the right
type can be the tail of a function body.

You can write the same logic with `match` by adding _match guards_ — `if`
conditions after a pattern:

```rust
fn classify(n: i32) -> &'static str {
    match n {
        n if n > 0 => "positive",
        n if n < 0 => "negative",
        _ => "zero",
    }
}
```

The guard `n if n > 0` binds the value to `n`, then checks the condition. The
arm only matches if the guard is true. Guards are checked top-to-bottom, so
the `_` arm catches whatever remains. Use match guards when you need conditions
beyond simple value matching — they are especially useful with ranges and
computed comparisons.

The `'static` in `&'static str` means the borrow points at text that stays valid
for the entire duration of the program. String literals like `"positive"` are
stored in the compiled binary, so they naturally have that lifetime. The key
idea is that `&'static str` is not a different kind of text from `&str`; it is
the same borrowed string slice with a stronger promise about how long the bytes
behind it live. For now, read it as "borrowed text that is always available."

## Functions That Run at Compile Time

In the previous chapter, you saw that `const` declarations must be set to
values the compiler can evaluate before your program runs. This raises a
question: what if you need a constant whose value requires a calculation?

A _const function_ is a function the compiler can evaluate at compile time. You
write it like any other function, with the `const` keyword before `fn`:

```rust
const fn square(n: i32) -> i32 {
    n * n
}

const BOARD_AREA: i32 = square(8);

fn main() {
    println!("An 8×8 board has {} squares", BOARD_AREA);

    // const fn works at runtime too — it is a normal function
    // that *also* works at compile time
    let side = 5;
    println!("A {side}×{side} board has {} squares", square(side));
}
```

Output:

```
An 8×8 board has 64 squares
A 5×5 board has 25 squares
```

The `const fn square` can appear in a `const` declaration because the compiler
evaluates `square(8)` during compilation — the value `64` is baked into the
binary just like a literal. But `square` is also a perfectly normal function
you can call at runtime with a variable argument. The `const` modifier _adds_
a capability; it does not restrict one.

Const functions can use `if`, `match`, `while`, and `loop` — the same control
flow you already know:

```rust
const fn factorial(n: u64) -> u64 {
    let mut result = 1;
    let mut i = 1;
    while i <= n {
        result *= i;
        i += 1;
    }
    result
}

const ARRANGEMENTS: u64 = factorial(10);

fn main() {
    println!("10 items can be arranged in {} ways", ARRANGEMENTS);
}
```

Output: `10 items can be arranged in 3628800 ways`

The compiler runs this loop at compile time and embeds the result directly. No
function call happens at runtime for `ARRANGEMENTS`.

You have already encountered `const fn` without realizing it. In the previous
chapter, methods like `midpoint`, `cast_signed`, and `cast_unsigned` are all
`const fn` in the standard library — that is why you can use them in constant
expressions. When you see "`const fn`" in documentation, it means you can use
that function in `const` and `static` declarations, not just at runtime.

## Closures

A _closure_ is an anonymous function that can capture variables from its
surrounding scope. Where a regular function stands alone, a closure can "close
over" the environment where it is defined — that is where the name comes from.

Here is a closure that adds a fixed offset to a number:

```rust
fn main() {
    let offset = 10;
    let add_offset = |x: i32| -> i32 { x + offset };

    println!("{}", add_offset(5));
    println!("{}", add_offset(20));
}
```

Output:

```
15
30
```

The syntax `|x: i32| -> i32 { x + offset }` defines a closure. The pipes (`|`)
delimit the parameter list, just as parentheses do for functions. The closure
uses `offset`, a variable defined outside of it — it _captures_ `offset` from
the enclosing scope.

### Concise Closure Syntax

Closures are designed to be lightweight. Rust can infer the parameter types and
return type from context, and if the body is a single expression, you can omit
the braces:

```rust
fn main() {
    let double = |x| x * 2;
    let is_even = |x: i32| x % 2 == 0;

    println!("double 5 = {}", double(5));
    println!("4 is even: {}", is_even(4));
}
```

Output:

```
double 5 = 10
4 is even: true
```

The `double` closure has no type annotations at all — the compiler infers both
the parameter type and the return type from how `double` is used. This makes
closures ideal for short, inline operations where full function signatures would
be noise.

Once the compiler infers concrete types for a closure, those types are fixed.
You cannot call `double` with a floating-point number after the compiler has
already inferred `i32` from the first use.

### How Closures Capture Variables

Closures can use variables from their surrounding scope, and the compiler
automatically decides _how_ to capture each variable based on what the closure
does with it:

```rust
fn main() {
    let greeting = "hello";
    let print_greeting = || println!("{greeting}");

    print_greeting();
    print_greeting();
    println!("still accessible: {greeting}");
}
```

Output:

```
hello
hello
still accessible: hello
```

The closure only reads `greeting`, so the compiler captures it by _reference_ —
a temporary borrow. The original variable remains usable after the closure is
defined and called.

When a closure modifies a captured variable, the compiler captures it by
_mutable reference_:

```rust
fn main() {
    let mut count = 0;

    let mut increment = || {
        count += 1;
        println!("count = {count}");
    };

    increment();
    increment();
}
```

Output:

```
count = 1
count = 2
```

Two things to notice: the closure itself must be declared `mut` because calling
it changes internal state, and the captured variable `count` must also be `mut`
because the closure modifies it.

The key insight is that _you do not need to specify how a closure captures its
variables_. The compiler figures out the least restrictive capture mode
automatically. If the closure only reads a value, it borrows. If it modifies a
value, it borrows mutably. You write the logic you need, and the compiler picks
the right mechanism.

### The Three Closure Traits

Every closure in Rust automatically implements one or more traits from a
hierarchy of three:

- `Fn` — the closure can be called repeatedly and does not modify its captured
  state. Think of it as a pure function with some extra context.
- `FnMut` — the closure can be called repeatedly but _may_ modify captured
  state. It needs exclusive access when called.
- `FnOnce` — the closure can be called at least once. It may consume (take
  ownership of) its captured state, which means it might not be callable a
  second time.

These traits form a hierarchy: every `Fn` closure is also `FnMut`, and every
`FnMut` closure is also `FnOnce`. A closure that only reads its captures
implements all three. A closure that modifies captures implements `FnMut` and
`FnOnce`. A closure that consumes a capture implements only `FnOnce`.

You do not need to memorize this hierarchy right now. The compiler picks the
right traits automatically based on what your closure does. The names become
important when you write functions that _accept_ closures as parameters — which
is what comes next.

## Higher-Order Functions

A _higher-order function_ is a function that takes another function — or a
closure — as a parameter. This is one of the most powerful patterns in Rust,
and you will encounter it constantly when working with iterators.

Here is a function that applies a transformation to each element of a slice
and shows the result:

```rust
fn apply_to_each(values: &[i32], f: impl Fn(i32) -> i32) {
    for &v in values {
        println!("  {v} -> {}", f(v));
    }
}

fn main() {
    let numbers = [1, 2, 3];

    println!("Doubled:");
    apply_to_each(&numbers, |x| x * 2);

    println!("Plus ten:");
    apply_to_each(&numbers, |x| x + 10);
}
```

Output:

```
Doubled:
  1 -> 2
  2 -> 4
  3 -> 6
Plus ten:
  1 -> 11
  2 -> 12
  3 -> 13
```

The parameter `f: impl Fn(i32) -> i32` means "any callable thing that takes an
`i32` and returns an `i32`." This accepts closures, regular functions, and
anything else that implements the `Fn` trait. The `impl` keyword here is
shorthand for a generic type parameter — the compiler generates a specialized
version of the function for each concrete type you pass, so there is no runtime
cost. Read `impl Trait` naturally: "a parameter that implements this trait."

### Passing Named Functions

You are not limited to passing closures. Named functions work too, as long as
their signature matches:

```rust
fn square(x: i32) -> i32 {
    x * x
}

fn apply(value: i32, f: impl Fn(i32) -> i32) -> i32 {
    f(value)
}

fn main() {
    println!("square of 5: {}", apply(5, square));
    println!("plus one:    {}", apply(5, |x| x + 1));
}
```

Output:

```
square of 5: 25
plus one:    6
```

A named function like `square` and an anonymous closure like `|x| x + 1` are
interchangeable when passed to a function that accepts `impl Fn`. This
uniformity is by design — it lets you choose whichever form is clearest for each
call site.

### Choosing the Right Trait Bound

When writing a function that accepts a closure, match the trait bound to how
your function uses it:

- Use `FnOnce` if you call the closure exactly once. This is the least
  restrictive — every closure can be called at least once.
- Use `FnMut` if you call the closure multiple times and it _may_ mutate state.
- Use `Fn` if you call the closure multiple times and it should not mutate
  state. This is the most restrictive.

`FnOnce` accepts the widest range of closures, so use it when your function
only needs a single call:

```rust
fn run_once(f: impl FnOnce()) {
    f();
}

fn run_repeatedly(mut f: impl FnMut()) {
    f();
    f();
    f();
}

fn main() {
    let greeting = String::from("hello");

    // `move` transfers ownership of `greeting` into the closure.
    // FnOnce is the right bound when you call the closure exactly once.
    run_once(move || println!("{greeting}"));

    // This closure mutates captured state — it needs FnMut.
    let mut count = 0;
    run_repeatedly(|| {
        count += 1;
        println!("count = {count}");
    });
}
```

Output:

```
hello
count = 1
count = 2
count = 3
```

The `move` keyword before a closure forces it to take ownership of captured
variables rather than borrowing them. Without `move`, this closure would borrow
`greeting`, and the original variable would remain accessible afterward. With
`move`, the `String` value transfers into the closure — `greeting` is consumed,
and the closure becomes fully self-contained, carrying everything it needs.

This matters whenever a closure must outlive the scope where it was created. If
the closure only borrows a variable, and that variable goes out of scope before
the closure runs, the borrow would become invalid. By moving ownership into the
closure, Rust guarantees the data lives as long as the closure does — no
dangling references, no runtime checks. The next section shows exactly this: a
function that returns a closure, where `move` is essential because the
function's local scope disappears after the return.

Notice that `run_repeatedly` declares its parameter as `mut f` — this is
required because each call to an `FnMut` closure may modify its internal state.

If `FnOnce` accepts the widest range of closures, why not always use it?
Because `FnOnce` only lets you call the closure _once_. If your function needs
multiple calls — like `run_repeatedly` does — you need `FnMut` or `Fn`.

## Returning Closures

Functions can also _return_ closures. Since every closure has a unique type
known only to the compiler, you use `impl Fn` in the return position to say
"I am returning something callable":

```rust
fn make_greeter(greeting: &'static str) -> impl Fn(&str) {
    move |name| println!("{greeting}, {name}!")
}

fn main() {
    let hello = make_greeter("Hello");
    let hi = make_greeter("Hi");

    hello("Alice");
    hi("Bob");
}
```

Output:

```
Hello, Alice!
Hi, Bob!
```

The `move` keyword is essential here — without it, the closure would try to
borrow `greeting` from the function's scope, but that scope no longer exists
after the function returns. With `move`, the closure takes ownership of
`greeting` and carries it along wherever it goes.

The `greeting` parameter is `&'static str` for the same reason as in
`classify`: callers pass string literals like `"Hello"` and `"Hi"`. The closure
owns that small reference, not a heap-allocated `String`, and the literal bytes
remain valid for the entire run of the program.

This pattern — a function that returns a configured closure — is called a
_function factory_. It is a clean way to create specialized behavior without
defining a new struct or trait.

## Putting It Together

Here is a small program that combines several ideas from this chapter —
functions, closures, and higher-order functions — to build a simple temperature
converter:

```rust
fn convert_temps(
    temps: &[f64],
    converter: impl Fn(f64) -> f64,
    label: &str,
) {
    for &t in temps {
        println!("{t:.1} -> {:.1} {label}", converter(t));
    }
}

fn main() {
    let celsius_temps = [0.0, 20.0, 37.0, 100.0];

    println!("Celsius to Fahrenheit:");
    convert_temps(&celsius_temps, |c| c * 9.0 / 5.0 + 32.0, "°F");

    println!();

    println!("Celsius to Kelvin:");
    convert_temps(&celsius_temps, |c| c + 273.15, "K");
}
```

Output:

```
Celsius to Fahrenheit:
0.0 -> 32.0 °F
20.0 -> 68.0 °F
37.0 -> 98.6 °F
100.0 -> 212.0 °F

Celsius to Kelvin:
0.0 -> 273.1 K
20.0 -> 293.1 K
37.0 -> 310.1 K
100.0 -> 373.1 K
```

The `convert_temps` function does not know or care what conversion formula it
applies — it just calls whatever closure is passed in. The logic is separated
from the data transformation, making both easy to test and reuse.

## Why This Matters

Functions and closures are the primary abstraction tools in Rust. Everything
you build — from a two-line helper to a multi-crate application — is composed
from them.

Key points from this chapter:

- **Function signatures are contracts.** Every parameter type and return type is
  explicit. The compiler checks every call site against these contracts, so
  reading a signature tells you what a function does before you read its body.
- **The last expression is the return value.** No `return` keyword needed on the
  normal path. Reserve `return` for early exits, so it signals "something
  exceptional is happening" to anyone reading the code.
- **`todo!()`, `unimplemented!()`, and `unreachable!()`** let you sketch code
  top-down. The structure compiles before every detail is implemented.
- **`const fn`** extends compile-time evaluation to your own functions. A
  `const fn` works at both compile time and runtime — the modifier adds a
  capability without restricting one.
- **Closures capture their environment automatically.** The compiler picks the
  least restrictive capture mode — borrow, mutable borrow, or move — based on
  what the closure does with each variable.
- **`Fn`, `FnMut`, `FnOnce`** form a hierarchy. Choose the least restrictive
  bound that satisfies your function's needs: `FnOnce` for single calls,
  `FnMut` for repeated calls that may mutate, `Fn` for pure repeated calls.
- **`impl Fn` in parameter position** accepts closures and named functions
  uniformly. In return position, it lets a function return a closure whose
  concrete type the caller does not need to know.
- **`move`** transfers ownership into a closure. This is essential when a
  closure must outlive the scope where it was created.

### Exercise 2-4: The Tip Calculator

**Goal.** Write a small billing program that uses regular functions, `const fn`,
early returns with `Option`, and formatted output.

**Your task:**

1. Write a `const fn default_tip_percent` that returns `18.0` as `f64`. Use it
   in a `const DEFAULT_TIP: f64` declaration.
2. Write `tax_rate` — takes a state abbreviation (`&str`) and returns the tax
   percentage as `f64` using `match` (`"OR"` → 0.0, `"CA"` → 7.25, `"TX"` →
   6.25, anything else → 5.0).
3. Write `tip_amount` — takes a subtotal and tip percentage, returns the tip.
4. Write `calculate_total` — takes a subtotal, state, and tip percentage.
   Return `None` if the subtotal is zero or negative (early return). Otherwise,
   compute tax + tip and return `Some(total)`.
5. Write `print_bill` — takes a label, subtotal, state, and tip percentage.
   Prints a formatted line using `calculate_total`.
6. In `main`, print the default tip, then call `print_bill` for several meals.

> **Key insight.** Each function does one job. `calculate_total` composes
> `tax_rate` and `tip_amount` without knowing their internals. The `const fn`
> demonstrates compile-time evaluation.

**Expected output:**

```
Default tip: 18%

Dinner in CA: $85.50 + 7.2% tax + 20% tip = $108.80
Lunch in OR: $32.00 + 0.0% tax + 15% tip = $36.80
Coffee in TX: $4.75 + 6.2% tax + 18% tip = $5.90
Bad input: invalid subtotal $-10.00
```

```rust
const fn default_tip_percent() -> f64 {
    18.0
}

const DEFAULT_TIP: f64 = default_tip_percent();

fn tax_rate(state: &str) -> f64 {
    match state {
        "OR" => 0.0,
        "CA" => 7.25,
        "TX" => 6.25,
        _ => 5.0,
    }
}

fn tip_amount(subtotal: f64, tip_percent: f64) -> f64 {
    subtotal * tip_percent / 100.0
}

fn calculate_total(subtotal: f64, state: &str, tip_percent: f64) -> Option<f64> {
    if subtotal <= 0.0 {
        return None;
    }
    let tax = subtotal * tax_rate(state) / 100.0;
    let tip = tip_amount(subtotal, tip_percent);
    Some(subtotal + tax + tip)
}

fn print_bill(label: &str, subtotal: f64, state: &str, tip_percent: f64) {
    print!("{label}: ");
    match calculate_total(subtotal, state, tip_percent) {
        Some(total) => println!(
            "${subtotal:.2} + {:.1}% tax + {tip_percent}% tip = ${total:.2}",
            tax_rate(state)
        ),
        None => println!("invalid subtotal ${subtotal:.2}"),
    }
}

fn main() {
    println!("Default tip: {DEFAULT_TIP}%");
    println!();

    print_bill("Dinner in CA", 85.50, "CA", 20.0);
    print_bill("Lunch in OR", 32.00, "OR", 15.0);
    print_bill("Coffee in TX", 4.75, "TX", DEFAULT_TIP);
    print_bill("Bad input", -10.0, "CA", 20.0);
}
```

Output:

```
Default tip: 18%

Dinner in CA: $85.50 + 7.2% tax + 20% tip = $108.80
Lunch in OR: $32.00 + 0.0% tax + 15% tip = $36.80
Coffee in TX: $4.75 + 6.2% tax + 18% tip = $5.90
Bad input: invalid subtotal $-10.00
```

Each function has a single responsibility: `tax_rate` maps states to rates,
`tip_amount` computes one multiplication, and `calculate_total` composes them
together. The `const fn` is trivial here, but demonstrates the pattern — the
value `18.0` is baked into the binary at compile time, just like a literal.

</details>

### Exercise 2-5: The Word Counter

**Goal.** Write a higher-order function that counts words matching any
predicate, then use it with different closures — including one that captures
a variable from its environment.

**Your task:**

1. Write `count_matching` — takes a slice of `&str` words and a predicate
   `impl Fn(&str) -> bool`, and returns a `usize` count of words where the
   predicate returns `true`.
2. In `main`, create an array of words. Use `count_matching` with several
   closures:
   - Count words longer than 3 characters.
   - Count words with 3 or fewer characters.
   - Count words starting with `'s'` or `'S'`.
   - Define a `min_length` variable, then pass a closure that _captures_ it
     to count words with at least that many characters.

> **Key insight.** The same `count_matching` function works with any criteria.
> The closures define _what_ to check; the function defines _how_ to iterate.
> The last closure demonstrates capture — it uses `min_length` from the
> surrounding scope without you having to pass it as a parameter.

**Expected output:**

```
Words: ["Rust", "is", "fast", "safe", "and", "Systems"]
Total: 6
Long (>3 chars): 4
Short (<=3 chars): 2
Starts with 's' or 'S': 2
At least 4 chars: 4
```

<details>
<summary>Solution</summary>

```rust
fn count_matching(words: &[&str], predicate: impl Fn(&str) -> bool) -> usize {
    words.iter().filter(|w| predicate(w)).count()
}

fn main() {
    let words = ["Rust", "is", "fast", "safe", "and", "Systems"];

    let total = words.len();
    let long_words = count_matching(&words, |w| w.len() > 3);
    let short_words = count_matching(&words, |w| w.len() <= 3);
    let starts_with_s = count_matching(&words, |w| {
        w.starts_with('s') || w.starts_with('S')
    });

    println!("Words: {:?}", words);
    println!("Total: {total}");
    println!("Long (>3 chars): {long_words}");
    println!("Short (<=3 chars): {short_words}");
    println!("Starts with 's' or 'S': {starts_with_s}");

    // This closure captures min_length from the surrounding scope
    let min_length = 4;
    let at_least_min = count_matching(&words, |w| w.len() >= min_length);
    println!("At least {min_length} chars: {at_least_min}");
}
```

Output:

```
Words: ["Rust", "is", "fast", "safe", "and", "Systems"]
Total: 6
Long (>3 chars): 4
Short (<=3 chars): 2
Starts with 's' or 'S': 2
At least 4 chars: 4
```

The `count_matching` function uses `.iter().filter().count()` — a chain of
iterator methods that does exactly what a manual loop would do. The closure
passed to `filter` is the same predicate passed to `count_matching`, showing
closures being used with real APIs, not just hand-written functions. A manual
loop with `let mut count` works equally well, but the iterator version is more
idiomatic. The final closure captures `min_length` by reference automatically
— the compiler picks the least restrictive capture mode.

</details>

### Exercise 2-6: The Formatter Factory

**Goal.** Write a function factory that returns closures, demonstrating
`impl Fn` in return position and the `move` keyword.

**Your task:**

1. Write `make_formatter` — takes a `&'static str` prefix and a `usize`
   precision, and returns an `impl Fn(f64) -> String` closure that formats a
   number with that prefix and precision.
   _Hint:_ Use `format!("{prefix}{value:.prec$}", prec = precision)` and the
   `move` keyword. The closure returns `String` (owned, heap-allocated text),
   not `str` (an unsized type you cannot return directly). `format!` always
   produces a `String`.
2. In `main`, create three formatters: USD (`"$"`, 2 decimal places), EUR
   (`"€"`, 2), and precise (`""`, 6).
3. Format a single amount with all three formatters.
4. Show that each formatter is reusable by formatting a list of prices.

> **Key insight.** Each call to `make_formatter` produces a different closure
> that carries its own `prefix` and `precision`. The `move` keyword is
> essential — without it, the closure would try to borrow `prefix` and
> `precision` from the function's scope, which no longer exists after the
> function returns.

**Expected output:**

```
Formatting 1234.5:
  USD: $1234.50
  EUR: €1234.50
  Precise: 1234.500000

Menu prices:
  $9.99
  $24.50
  $3.00
```

<details>
<summary>Solution</summary>

```rust
fn make_formatter(prefix: &'static str, precision: usize) -> impl Fn(f64) -> String {
    move |value| format!("{prefix}{value:.prec$}", prec = precision)
}

fn main() {
    let usd = make_formatter("$", 2);
    let eur = make_formatter("€", 2);
    let precise = make_formatter("", 6);

    let amount = 1234.5;
    println!("Formatting {amount}:");
    println!("  USD: {}", usd(amount));
    println!("  EUR: {}", eur(amount));
    println!("  Precise: {}", precise(amount));

    // Each formatter is reusable — call it with different values
    println!();
    println!("Menu prices:");
    let prices = [9.99, 24.50, 3.0];
    for price in prices {
        println!("  {}", usd(price));
    }
}
```

Output:

```
Formatting 1234.5:
  USD: $1234.50
  EUR: €1234.50
  Precise: 1234.500000

Menu prices:
  $9.99
  $24.50
  $3.00
```

The `make_formatter` function is a _function factory_ — each call creates a
new closure with its own captured data. The `move` keyword transfers ownership
of `prefix` and `precision` into the closure, making it fully self-contained.
Because `prefix` is `&'static str` (a reference to a string literal), the
closure owns that small reference, not a heap-allocated `String`. The closure
implements `Fn` (not just `FnOnce`) because it only reads its captured data,
so it can be called repeatedly.

The solution uses `for price in prices` to iterate over the array. This works
because `for` calls `.into_iter()` automatically, which yields owned `f64`
values. If you used `.iter()` instead, each element would be `&f64` (a
reference), and you would need to dereference it with `*price` before passing
it to the formatter.

</details>

Functions and closures are the building blocks of every Rust program. Functions
give your code structure and clear contracts through explicit signatures.
Closures add flexibility by capturing context and letting you pass behavior as
data. Together with the expression-oriented design from the previous chapter,
they let you write code that is both concise and easy to reason about.

In the next chapter, you will learn about _ownership_ — the single most
important idea in Rust and the foundation of its memory safety guarantees. The
functions and closures you have just learned will be central to that story,
because ownership determines how values flow into, out of, and between
functions.
